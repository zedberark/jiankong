# 基于 NetPulse 的网络监控与运维平台设计与实现

---

## 摘要

随着企业信息化与网络基础设施规模的持续扩张，数据中心与园区网络中的服务器、交换机、路由器及防火墙等设备数量不断增多，运维人员需要在统一视图中掌握设备在线状态、性能指标与告警信息，并能在故障或变更场景下快速登录设备执行处置。传统依赖分散登录与手工记录的方式难以保证效率与可追溯性，因此有必要构建一套集中式的网络监控与运维平台。

本文设计并实现了一套名为 NetPulse 的网络监控与运维平台。系统采用浏览器与服务器（B/S）架构及前后端分离模式：服务端基于 Spring Boot 3 与 Java 21，提供 RESTful 接口与 WebSocket 长连接；客户端基于 Vue 3 与 Vite 实现单页应用。数据存储方面，业务数据与权限、审计、巡检报告等持久化于 MySQL；Linux 主机通过 Telegraf 采集的 CPU、内存等时序数据存储于 InfluxDB；网络设备 SNMP 等采集结果以 Redis 为缓存供实时展示。平台实现了设备台账与健康探测、实时指标聚合、告警规则与告警历史、系统巡检与可选的大模型文字结论、浏览器内 Web SSH 与 Telnet、批量命令执行、配置备份与恢复、系统与第三方 API 配置、基于角色与菜单编码的访问控制、操作审计与多种数据导出，以及 AI 运维助手等能力。数据库结构遵循规范化原则，并通过 Flyway 进行版本化迁移管理。

在开发与联调基础上，对登录与权限、设备管理、指标与告警、巡检、远程终端与批量命令、备份与配置、审计与导出等模块进行了功能验证。结果表明，各模块行为与预期一致，接口响应与错误提示清晰，能够满足实验环境及中小规模网络场景下集中监控与运维辅助的毕业设计目标。

**关键词**：网络监控；运维平台；Spring Boot；Vue；MySQL；InfluxDB；Redis；系统巡检；Web SSH

---

## Abstract

With the expansion of enterprise networks, administrators need a unified view of device availability, performance metrics, and alerts, as well as efficient remote access to managed devices. This thesis presents NetPulse, a browser-based monitoring and operations platform built with Spring Boot 3 and Vue 3 in a separated front-end/back-end architecture. MySQL stores business and configuration data; InfluxDB stores time-series metrics from Telegraf; Redis caches real-time data for network devices. The system provides device management, health probing, real-time dashboards, alerting, scheduled and manual inspection reports with optional LLM-generated summaries, Web SSH/Telnet via WebSocket, batch command execution, configuration backup and restore, role-based menu access, audit logging and exports, and an AI assistant. Functional verification shows that core modules behave as designed and meet the objectives of the graduation project.

**Keywords**: Network monitoring; Operations platform; Spring Boot; Vue; MySQL; InfluxDB; Redis

## 1 绪论

### 1.1 研究背景与意义

随着企业信息化与网络规模的不断扩大，数据中心与园区网络中服务器、交换机、路由器、防火墙等设备数量众多，运维人员需要实时掌握设备在线状态、性能指标与告警信息，并在故障发生时快速定位与处置。传统依赖人工逐台登录设备执行命令、查看脚本输出的方式效率较低，操作过程难以统一记录与审计，且在设备数量增长时难以保持策略与数据的一致性。因此，构建一套集中式的网络监控与运维平台，实现设备台账统一、状态采集、指标可视化、告警管理与远程运维的规范化与自动化，具有明确的工程应用价值，亦符合本毕业设计选题的实践导向。

### 1.2 系统概述

本文所述 NetPulse 系统是一套基于 B/S 架构的网络监控与运维平台，面向虚拟机 Linux 环境与网络设备（路由器、交换机、防火墙等），在功能上涵盖设备统一管理、健康状态监控、实时指标展示、告警规则与历史记录、Web SSH 远程终端、批量命令执行、配置备份与恢复、用户与角色权限、操作审计以及 AI 运维助手等。系统在技术实现上采用前后端分离：后端基于 Spring Boot 3，使用 MySQL 存储业务数据、InfluxDB 存储时序指标、Redis 缓存网络设备实时数据；前端基于 Vue 3 与 Vite，通过 HTTP 调用 REST 接口、通过 WebSocket 支撑终端类业务。平台适用于中小规模网络环境的集中监控与日常运维，并可与现有采集体系（如 Telegraf、InfluxDB）及 Redis 缓存策略对接，便于后续扩展。

### 1.3 国内外相关研究与应用现状

在网络监控与可观测性领域，国际上已有 Zabbix、Prometheus、Grafana、Nagios 等成熟开源或商业方案；国内许多单位在开源组件基础上进行二次开发或建设自研监控平台。在远程运维方向，Web 终端、堡垒机与自动化运维工具已广泛落地。与上述通用平台相比，本设计以毕业设计可实现的规模为约束，借鉴“设备—告警规则—告警历史”等通用数据建模思路，并针对 Linux 主机（Telegraf 与 InfluxDB）与网络设备（Redis 按设备维度缓存）两类数据源进行统一展示与权限集成，突出可实现的工程完整性与文档化程度。

## 2 系统关键技术

### 2.1 后端技术选型

#### 2.1.1 Spring Boot 框架

后端采用 Spring Boot 3 构建。该框架能够较好地覆盖本课题对 REST 接口、定时任务（设备健康检查、告警评估）、WebSocket（Web 终端代理）以及多数据源访问（MySQL、Redis、InfluxDB）的需求；内嵌 Servlet 容器，打包为可执行 JAR 后便于部署；依赖注入与分层架构有利于控制层、业务层与持久层职责分离，便于在毕业设计周期内完成开发与文档撰写。

#### 2.1.2 数据存储与时序库

后端通过 Spring Data JPA 访问 MySQL、Spring Data Redis 访问 Redis，并以自定义客户端访问 InfluxDB。三类存储的技术特点与在本系统中的职责划分见 2.2 节。

#### 2.1.3 远程连接与采集组件

系统使用 JSch 实现 SSH 连接，用于 Web 终端、批量命令及告警自动修复等场景；使用 Apache Commons Net 实现 Telnet（如端口 23）连接；使用 Snmp4j 完成网络设备 SNMP v2c/v3 采集并将结果写入 Redis；使用 Spring WebSocket 实现浏览器与后端之间的长连接转发；Spring Mail 可用于告警邮件扩展；Spring Scheduling 承担设备探测、告警评估、SNMP 定时采集及巡检定时任务等。

### 2.2 数据存储与缓存

针对系统不同类型数据的存储特性，采用 MySQL、Redis、InfluxDB 协同的存储方案，分别承担结构化数据持久化、实时数据缓存与时序指标存储，在保障存取效率的同时支撑系统稳定运行。

#### 2.2.1 MySQL、Redis 与 InfluxDB 简介

**MySQL** 为关系型数据库，支持事务与复杂关联查询，适用于用户、设备、告警规则、审计日志等结构化业务数据的持久化；与 Spring Data JPA 配合可简化数据访问层开发与表结构演进管理。

**Redis** 为内存键值存储，支持多种数据结构与可选持久化策略，读写延迟低，适合作为缓存层承接高频读取场景，降低对磁盘型数据库的访问压力。

**InfluxDB** 为时序数据库，针对时间戳索引与高写入吞吐优化，适用于 CPU、内存、磁盘等随时间变化的监控指标存储，并支持时间范围查询与聚合分析。

#### 2.2.2 存储特性与系统应用

以下说明三类存储在本系统中的职责分工与数据路径，避免与 2.2.1 重复罗列技术定义，仅对应“存什么、从哪来”。

**MySQL** 作为核心持久化库，存储用户权限、设备信息、告警规则与告警历史、巡检与审计、系统与第三方配置、AI 会话等结构化业务数据，保障业务操作的一致性与可追溯性，是权限校验与业务流程的权威数据源。

**Redis** 作为实时数据缓存层，承接网络设备侧由 SNMP 及 SSH/Telnet 补充采集得到的运行指标，按设备或键维度组织后供监控大盘与实时指标接口毫秒级查询，减少重复计算与对 MySQL 的频繁访问。

**InfluxDB** 作为时序数据引擎：Linux 服务器由部署于被管主机的 **Telegraf** 周期性采集 CPU、内存、磁盘等指标并写入；后端在设备可达性探测时写入 **`device_status`** 测量（如往返时延、在线状态），供趋势分析与历史曲线。网络设备实时性能指标由 SNMP/CLI 采集后经 **Redis** 提供查询，与 InfluxDB 的上述两条写入路径区分，避免将“网络设备周期性能”误归入时序库主路径。

### 2.3 前端技术选型

前端采用 Vue 3 与 Vue Router 实现组件化开发与路由管理，菜单与路由根据角色权限进行控制。构建工具选用 Vite，以获得较快的开发体验。HTTP 访问使用 Axios 与后端 REST 接口交互。Web 终端界面采用 xterm.js 渲染。仪表盘与实时指标等可视化采用 ECharts 绘制图表。

### 2.4 外部依赖与运行环境

在 Linux 服务器上可部署 Telegraf 采集主机指标并写入 InfluxDB。生产环境可使用 Nginx 托管前端静态资源并反向代理后端接口与 WebSocket。后端构建采用 Maven，运行依赖 JDK 21；前端依赖 Node.js 与 npm 完成依赖安装与打包。

## 3 需求分析

本章从业务目标出发，对平台需满足的业务需求、功能需求与非功能需求进行分析，为系统设计与实现提供依据。

### 3.1 业务需求分析

结合企业日常运维工作流程，平台的业务目标可归纳为“统一纳管、实时感知、快速处置、过程留痕、持续优化”。围绕该目标，系统需满足以下核心业务场景。

第一，设备统一纳管场景。系统需将服务器与网络设备纳入统一台账，实现设备信息集中维护与分类管理，减少资产信息分散导致的管理盲区。

第二，状态持续监测场景。系统需持续呈现设备连通状态与关键运行指标，帮助运维人员在异常早期快速发现问题并定位影响范围。

第三，告警闭环处置场景。系统需支持告警从触发、通知、处理到恢复的全过程管理，提升故障响应效率并降低人工巡检压力。

第四，远程运维协同场景。系统需在浏览器环境中提供设备远程操作能力，并支持多设备协同执行，满足批量运维与应急处置需求。

第五，配置安全治理场景。系统需支持配置备份与恢复，并记录关键操作行为，确保运维活动可追溯、可审计。

第六，智能辅助运维场景。系统需提供面向运维语境的智能问答与巡检分析能力，辅助运维人员形成更高效的排障与决策方式。

### 3.2 功能需求分析

基于上述业务场景，系统功能需求可分解为以下八个模块：

（1）认证与权限管理：支持用户登录、身份识别与角色授权，确保不同角色在可见菜单与可执行操作上的权限隔离。  
（2）设备管理：支持设备信息维护、条件检索、分组管理与健康概览，满足运维资产管理与定位需求。  
（3）监控与指标展示：支持设备状态与核心性能指标统一展示，为运行态分析与异常识别提供依据。  
（4）告警管理：支持规则配置、事件记录、处理状态维护与通知联动，形成可跟踪的告警处理闭环。  
（5）系统巡检：支持按需巡检与周期巡检，输出可查看、可归档的巡检结果与报告。  
（6）远程运维：支持浏览器端远程终端与批量命令执行，提升跨设备操作效率。  
（7）AI 运维助手：支持会话式问答与上下文辅助，提升故障排查与运维决策效率。  
（8）系统管理：支持系统参数维护、配置备份恢复、操作审计与数据导出等治理能力。

功能模块之间需满足“相互协作但边界清晰”的要求：认证与权限为全局基础，设备与监控告警构成业务主链，巡检与 AI 能力形成分析增强，系统管理提供治理支撑。

### 3.3 非功能需求分析

除功能能力外，系统还需满足以下非功能目标：

（1）性能与可扩展性：在既定设备规模下保持可接受响应时间，并支持随规模增长进行容量扩展与参数调优。  
（2）可用性与稳定性：关键功能应具备稳定运行能力，异常场景下可提供明确提示并支持故障恢复。  
（3）安全性：系统应满足身份认证、权限控制与敏感信息保护要求，降低越权访问与数据泄露风险。  
（4）可维护性：系统应具有清晰模块边界与一致命名规范，便于后续功能迭代、问题排查与版本演进。  
（5）可部署性与兼容性：系统应支持主流运行环境部署，部署步骤可复现，满足教学与中小规模应用场景。  
（6）可审计性：关键操作过程应可追踪，保证运维活动可核查、可复盘。

### 3.4 本章小结

本章从业务目标出发，明确了系统在设备纳管、状态感知、告警处置、远程运维、配置治理与智能辅助等方面的需求边界，并对功能需求与非功能需求进行了分解。第 4 章将在此基础上进一步说明系统架构、模块设计、流程设计、数据库设计与核心类结构，实现“先需求、后设计”的章节衔接。

## 4 系统设计

第 4 章围绕系统设计展开，重点说明系统总体架构、功能模块划分、关键业务逻辑流程、数据库设计与核心类结构，为第 5 章的实现细节提供结构化依据。相关图示（架构图、功能结构图、流程图、类图与 E-R 图）统一维护在项目 `docs` 目录。

### 4.1 系统总体架构

系统采用 B/S 架构与前后端分离模式。前端基于 Vue 3 + Vite 构建单页应用，主要通过 REST 接口与后端交互，统一以 `/api` 作为接口前缀；远程终端等长连接场景通过 WebSocket 实现双向通信。

后端基于 Spring Boot 3，按“控制层—业务层—持久层”组织：控制层负责参数校验与接口编排，业务层封装巡检、告警、采集、备份与 AI 对话等领域逻辑，持久层以 Spring Data JPA 与数据库交互。系统中的定时任务（状态探测、告警评估、巡检调度等）由 Spring Scheduling 统一调度。

数据层采用分工式设计。**MySQL** 作为核心持久化库，存储用户权限、设备信息、告警规则、巡检与审计、系统配置等结构化业务数据，保障事务一致性与业务主数据完整性，是系统核心功能的基础数据支撑。**Redis** 作为实时数据缓存层，承接网络设备侧通过 SNMP 采集（以及 SSH/Telnet 补充采集）得到的运行指标与状态，为前端监控大盘提供毫秒级查询响应，并减轻对 MySQL 的频繁访问压力。**InfluxDB** 作为时序数据引擎，主要承接 Linux 服务器侧由 Telegraf 采集并写入的周期性监控指标（如 CPU、内存、磁盘等），支撑监控趋势分析、历史数据回溯与可视化图表展示；后端可达性探测等产生的时序数据亦可写入该库，便于统一按时间维度查询。三者协同兼顾事务一致性、实时读取性能与时序查询效率。

外部依赖包括被管设备（SSH/Telnet/SNMP）、SMTP 邮件服务（可选）和大模型 HTTP API。总体上形成“前端可视化 + 后端编排 + 多存储协同 + 外部设备接入”的架构闭环。

### 4.2 功能模块划分

根据系统功能模块图，监控运维系统由 8 个一级模块构成，各模块及其二级功能如下。

（1）登录认证模块：账号密码登录、菜单权限配置、权限验证、设备鉴权与校验。  
（2）首页仪表盘模块：设备健康汇总、在线/离线统计、告警摘要、设备类型分布。  
（3）设备管理模块：设备列表与增删改查、设备拓扑可视化、设备分布分析、Ping 连通性检测。  
（4）监控与告警模块：实时指标展示、告警规则管理、告警历史查询、邮件通知联动。  
（5）远程运维模块：Linux 运维控制台、网络设备运维控制台、Web SSH、远程会话管理。  
（6）系统巡检模块：定时巡检、巡检任务执行报告、巡检报告 PDF 导出、巡检报告生成。  
（7）AI 运维助手模块：智能会话、设备与监控上下文、故障诊断分析、大模型问答。  
（8）系统管理模块：用户与角色权限、系统设置、配置备份与恢复、操作审计。

上述划分与功能模块图保持一致，形成“认证入口—运行监控—处置执行—治理支撑”的模块化体系。各模块在边界内独立实现，通过统一接口与权限体系进行协作，满足系统可扩展与可维护要求。

### 4.3 系统逻辑流程

系统整体流程以“登录认证—功能分流—统一退出”为主线，具体过程如下。

（1）用户进入登录页面并输入用户名、密码后提交登录请求，系统对账号与密码进行校验。  
（2）当校验失败时，系统提示登录失败并返回输入环节，用户可重新输入后再次提交。  
（3）当校验成功时，系统生成会话信息，随后加载用户角色与菜单权限，完成首页初始化并进入首页仪表盘。  
（4）进入首页后，系统进入功能分流节点。用户可按需进入设备管理、监控与告警、系统巡检、远程运维、AI 运维助手、系统管理、用户管理等功能模块。  
（5）各功能模块处理完成后返回主界面，用户可继续选择其他模块，形成“登录后多模块循环操作”的业务模式。  
（6）当用户结束操作时执行退出登录，系统返回登录页面并结束本次业务流程。

该流程与功能分流图保持一致，完整体现了从认证入口到业务操作再到会话退出的闭环逻辑。

### 4.4 数据库设计

#### 4.4.1 设计原则

数据库设计以业务完整性与工程可维护性为导向，主要遵循以下原则：  
（1）**规范化与适度冗余结合**：核心业务表满足第三范式要求，减少数据冗余；对巡检明细、告警历史等高频查询场景保留必要业务冗余字段以提升检索效率。  
（2）**可扩展性**：通过配置表、中间表与状态字段支持功能演进，新增设备类型、告警规则或权限映射时尽量避免大规模表结构变更。  
（3）**可审计性**：关键业务操作保留时间戳与审计记录，保证操作可追踪、问题可复盘。  
（4）**一致性与完整性**：通过主键、唯一约束、外键或业务约束保证数据一致，确保实体关系表达清晰。  
（5）**可迁移性**：结构变更采用 Flyway 版本迁移管理，支持多环境统一部署与回溯。

#### 4.4.2 概念结构设计

### 4.5 系统部署与数据流

本节从工程落地角度说明 NetPulse 的典型部署形态，以及登录、业务接口、远程终端、定时采集与多存储之间的数据流向，便于与第 4.1 节架构描述及第 5 章实现相互对照。

#### 4.5.1 部署拓扑与运行组件

系统采用浏览器/服务器（B/S）模式，典型部署包含以下逻辑组件：

（1）**前端应用**：基于 Vue 3 与 Vite 构建的单页应用。开发阶段由 Vite 开发服务器提供页面，并将以 `/api` 为前缀的请求代理至后端；生产阶段可将打包后的静态资源置于 Nginx 等 Web 服务器上，由反向代理统一转发 REST 与 WebSocket。  
（2）**后端服务**：基于 Spring Boot 3.2、JDK 21 的可执行应用，对外提供 RESTful 接口（统一前缀 `/api`）及 WebSocket 端点（如 Web 终端）。业务逻辑、定时任务、告警与巡检调度均运行在该进程中。  
（3）**MySQL**：持久化用户与角色、设备台账、告警规则与历史、巡检报告与明细、配置备份、操作审计、系统配置及 AI 会话等业务数据。  
（4）**InfluxDB**：存储时序类数据。一方面由 **Telegraf** 在被管 Linux 主机上采集 CPU、内存等指标并写入；另一方面后端在设备可达性探测时可将 `device_status` 等测量数据写入，供仪表盘与时间线查询。  
（5）**Redis**（可选）：在开启相关配置时，缓存网络设备 SNMP 或 SSH/Telnet 定时采集的实时指标，供实时指标接口快速聚合返回。  
（6）**被管设备**：Linux 服务器（SSH、Telegraf）、网络设备（SNMP v2c/v3，必要时 SSH/Telnet 端口 22 或 23）作为数据产生方与命令执行目标。

上述组件在物理上可同机部署（毕业设计/实验环境），也可按角色拆分到多台主机（生产环境），逻辑关系不变。

#### 4.5.2 部署步骤要点（可复现）

（1）准备 MySQL、Redis（若启用缓存）、InfluxDB（若启用时序与 Telegraf 联动），并创建库、用户与访问策略。  
（2）配置后端 `application-*.yml` 中的数据源地址、InfluxDB 组织与存储桶、Redis 连接及跨域等参数；执行 Flyway 迁移完成表结构初始化。  
（3）启动后端进程，确认健康检查或登录接口可用。  
（4）在需监控的 Linux 主机上部署 Telegraf，将输出指向与本系统一致的 InfluxDB，使主机名或标签能与设备台账中的标识对应。  
（5）启动前端开发服务或部署静态资源；确保浏览器访问路径下 `/api` 与 WebSocket 能正确代理到后端。  
（6）在平台中录入设备、配置 SNMP/SSH 等采集参数后，观察定时任务驱动的状态与指标是否按预期刷新。

#### 4.5.3 同步业务数据流（REST）

用户通过浏览器访问前端后，典型同步数据流为：

（1）**登录与权限**：前端提交账号密码 → 后端 `AuthController` 校验 → 返回用户信息、角色及可见菜单编码等 → 前端写入本地会话并用于路由与菜单守卫。  
（2）**CRUD 与查询**：前端发起 `/api/...` 请求并携带令牌 → `AuthTokenInterceptor` 等完成鉴权 → 对应 `Controller` 接收参数 → `Service` 编排业务 → `Repository` 访问 **MySQL** → 结果序列化为 JSON 返回前端展示。

该类请求以“请求—响应”为主，数据落点以 MySQL 为核心，强调事务一致性与权限边界。

#### 4.5.4 终端与批量命令数据流

（1）**Web SSH / Telnet**：浏览器与后端建立 **WebSocket**（如 `/api/ws/ssh` 并携带设备标识）→ 后端 `SshWebSocketHandler` 与 `WebSshService` 建立到目标设备的 SSH 或 Telnet 会话 → 双向转发终端字节流。该链路不经过传统 REST 请求体，但同样受登录态与设备配置约束。  
（2）**批量命令**：前端提交设备 ID 列表与命令文本 → `BatchCommandController` 调用 `BatchCommandService` → 对每台设备依次通过 SSH/Telnet 执行命令 → 将每台成功/失败及输出汇总返回前端。结果可辅以审计日志记录，便于追溯。

#### 4.5.5 异步采集与时序数据流

平台中“持续运行”的监控数据主要由 **Spring 定时任务** 与 **主机侧采集器（Telegraf）** 产生，与单次页面点击的 REST 请求相互独立。为便于与实现对照，下面对各条采集链路的**触发条件、默认周期、数据落点与失败策略**分别说明（具体参数以工程 `application.yml` 及环境变量覆盖为准）。

**（1）采集任务总览**

| 采集任务 | 实现类（后端） | 调度方式 | 默认周期（工程默认值） | 主要数据落点 |
|----------|----------------|----------|------------------------|--------------|
| 设备可达性与在线状态 | `MonitorService` | `@Scheduled`，`monitor.collect-interval` | 60 秒 | MySQL（`device` 状态、最近轮询时间）；InfluxDB 测量 `device_status`（可选） |
| Linux 等指标汇聚与 SSH 补采 | `DeviceStatsService` | `@Scheduled`，`monitor.stats-interval` / 首次延迟 `stats-initial-delay` | 首次约 30 秒后，之后约每 60 秒 | 读 InfluxDB（Telegraf）；必要时 SSH 执行命令补全缓存 |
| 按设备的 SNMP 采集（网络设备） | `DeviceSnmpCollectService` | `@Scheduled`，`snmp.collect.interval-ms` | 300000 ms（5 分钟） | Redis（`snmp.use-redis=true` 时）；失败可回退 SSH/Telnet |
| 固定单目标的 SNMP 样例采集（可选） | `SnmpCollectService` | 同 SNMP 周期配置 | 与上类似 | Redis（早期“单台防火墙”场景；默认 `snmp.firewall.enabled=false`） |
| 网络设备 SSH/Telnet 定时采集 | `DeviceSshCollectService` | `@Scheduled`，`ssh-collect.interval-ms` | 300000 ms（5 分钟），首次延迟约 15 s | Redis（与 SNMP 缓存体系配合展示） |
| 巡检中的批量可达性探测 | `InspectionService` 等 | 手动触发或 `InspectionScheduleService` 定时 | 由巡检配置决定 | MySQL（巡检报告与明细） |

**（2）设备可达性采集（MonitorService）**

对每台未逻辑删除的设备，定时执行可达性判断，流程要点如下：

（1）**首选 ICMP Ping**：向设备管理 IP 发送 Ping，超时时间由 `monitor.ping-timeout`（默认 3000 ms）约束，得到 RTT（毫秒）或判定不可达。  
（2）**类型相关的探测策略**：对 **Linux 服务器（`server`）** 一般仅依赖 Ping 结果；对 **路由器/交换机/防火墙等非服务器类型**，若 Ping 失败，则依次尝试对 **SNMP 端口（默认 161）** 与 **SSH 端口（设备表中配置，常见 22）** 做 TCP 连通性探测，以降低“禁 Ping 但服务端口可达”时的误判。  
（3）**状态写回 MySQL**：根据 RTT 与可达性将设备状态更新为在线正常、在线高延迟（warning）或离线（offline），并刷新 `last_poll_time`。  
（4）**写入 InfluxDB（若客户端可用）**：向测量 **`device_status`** 写入点位，携带 `device_id`、`device_name`、`type` 等标签，以及 `rtt_ms`、`status` 等字段，供首页趋势与历史曲线类查询。  
（5）**告警联动**：当状态相对上一轮发生变化时，调用 `AlertService` 相关逻辑，使“设备离线/恢复”等事件进入告警闭环。

**（3）Linux 主机性能采集（Telegraf + DeviceStatsService）**

（1）**主路径**：在各 Linux 服务器部署 **Telegraf**，按运维规范配置输入插件与输出到 **InfluxDB**（与后端 `influxdb.org`、`influxdb.bucket` 等配置一致）。Telegraf 持续将 CPU、内存等时序数据写入 InfluxDB，**不经过 NetPulse 后端进程**。  
（2）**后端读路径**：`DeviceStatsService` 在定时任务中调用 `collectFromInfluxDB`，按设备与主机标识（如 hostname 等）从 InfluxDB 查询最近一段时间内的 **last()** 类聚合结果，填充内存中的设备统计缓存，供指标接口与页面展示。  
（3）**SSH 补采**：若某台设备在缓存中仍缺少 CPU/内存/磁盘等关键字段，且设备台账中已配置 SSH/Telnet 用户名与密码、且设备非离线，则尝试通过 **JSch** 执行轻量命令补采，以提升展示完整性；补采失败一般仅记录调试日志，不阻塞其他设备。

**（4）网络设备 SNMP 采集（DeviceSnmpCollectService）**

（1）**启用条件**：由配置项 `snmp.collect.enabled` 控制；未启用时不执行该定时任务。  
（2）**设备范围**：从数据库查询 **已配置 SNMP 且未删除** 的设备列表；**跳过类型为 `server` 的 Linux 服务器**（与“服务器走 Telegraf/SSH”策略一致）；**跳过已标记离线的设备**，避免无效 SNMP 请求。  
（3）**采集内容与 OID**：网络设备 CPU、内存等通过 **Snmp4j** 按设备上配置的 SNMP 版本（v2c/v3 等）拉取；工程中对华为类设备在配置文件中给出了 **OID 示例**（如 `snmp.oid.cpu-usage`、`snmp.oid.mem-usage`），实际以设备厂商与 MIB 为准，可在运维上按机型调整。  
（4）**写入 Redis**：当 `snmp.use-redis=true` 时，将每台设备最新采集结果序列化后写入 Redis，键前缀由 `snmp.redis.key-prefix`（如 `netpulse:snmp:device`）与设备维度组合，供 `SnmpStatsService`、指标控制层等 **O(1) 或批量读取**，降低页面刷新时的延迟。  
（5）**失败与回退**：单次 SNMP 异常时记录告警日志，并可按实现逻辑 **回退到 `DeviceSshCollectService`**，在设备已配置 SSH/Telnet 账号且策略允许时，用命令行方式尝试采集 CPU/内存等并同样写入 Redis，形成“SNMP 优先、命令行兜底”的韧性路径。  
（6）**无 SNMP 设备时的行为**：若当前没有任何已配 SNMP 的设备，定时任务中可转而尝试仅依赖 SSH/Telnet 的采集路径（与实现中“无 SNMP 设备时触发 SSH 采集”的分支一致）。

**（5）可选固定目标 SNMP（SnmpCollectService）**

工程保留对 **单一固定主机**（如早期防火墙地址）的 SNMP 轮询能力，由 `snmp.firewall.enabled` 等配置控制，**默认关闭**。启用后，定时将结果写入 Redis，与“设备管理中的按设备采集”并存时，应注意避免论文表述中与 `DeviceSnmpCollectService` 混淆——正文可说明：**生产环境推荐全部在设备台账中配置 SNMP，由按设备服务统一采集。**

**（6）网络设备 SSH/Telnet 定时采集（DeviceSshCollectService）**

（1）**启用条件**：`ssh-collect.enabled`（默认可为 true，以工程配置为准）。  
（2）**设备过滤**：对 **非 server**、**非离线**、已配置用户名密码、具备有效 IP 的设备执行；对曾判定为 **Web SSH 不可达** 的设备会暂时跳过，并由定时 **重检任务**（配置项 `webssh-recheck-*`）周期性探测是否恢复，恢复后再参与采集。  
（3）**厂商与命令适配**：实现中对部分厂商（如思科、华为、锐捷、华三等）通过 **SSH 执行显示类命令** 解析 CPU/内存等文本输出；不支持的厂商可在实现中跳过，以免错误登录或无效命令。  
（4）**周期与资源**：默认 **5 分钟** 一轮，与 SNMP 周期对齐，减少频繁建连、断开对网络设备的压力；端口 **23** 时走 **Telnet** 路径，端口 **22** 等走 **SSH**。  
（5）**落点**：结果写入 **Redis**，与 SNMP 缓存一并供实时指标接口聚合。

**（7）采集数据的消费与告警**

（1）**前端展示**：用户打开实时指标、仪表盘等页面时，前端调用后端 **Metrics / 设备指标相关接口**，后端从 **Redis（网络设备快照）**、**InfluxDB（时序曲线与 Telegraf）**、**MySQL（设备状态与台账）** 组合读取并返回统一视图。  
（2）**告警评估**：`AlertService` 等根据 **设备状态变化**、**规则配置** 与 **指标阈值** 判断是否产生或恢复告警，并写入告警历史；邮件通知等可作为扩展能力依赖 Spring Mail。  
（3）**与巡检的关系**：巡检任务中的 **纯探测** 可调用 `MonitorService` 的仅探测接口，批量生成报告明细，**不重复实现** Ping/TCP 逻辑，保证与在线状态判定一致。

可将上述流程概括为：**Telegraf 与多条 `@Scheduled` 任务产生时序与缓存 → Service 聚合与规则评估 → REST 输出给前端**；**台账与告警记录仍以 MySQL 为权威**。用户操作与配置变更则主要走 **REST + MySQL**。

**采集流程（步骤化表述，便于正文与答辩口述）**

**一、Linux 服务器（`type = server`）采集流程**

步骤 1：在目标 Linux 主机部署 **Telegraf**，配置输出到与本系统一致的 **InfluxDB**（组织、存储桶等与后端配置一致），使 CPU、内存、磁盘等指标持续写入时序库。  
步骤 2：平台侧 **`MonitorService`** 按周期对该设备管理 IP 做 **Ping** 探测，将在线/离线及高延迟等状态写入 **MySQL** 设备表，必要时向 InfluxDB 写入 **`device_status`** 测量；状态变化可触发告警。  
步骤 3：平台侧 **`DeviceStatsService`** 按周期从 **InfluxDB** 查询各 Linux 设备对应的 Telegraf 时序数据，将最新 CPU、内存、磁盘等写入内存缓存，供指标接口读取。  
步骤 4：若某台 Linux 设备在 InfluxDB 中仍缺少部分指标，且该设备在台账中已配置 **SSH/Telnet 账号**、且当前 **非离线**，则通过 **SSH** 执行系统命令进行 **补采**，结果合并入同一缓存。  
步骤 5：用户在前端打开实时指标或仪表盘时，通过 **REST** 拉取后端已聚合的指标与 MySQL 中的设备状态，完成展示。

**二、网络设备（交换机/路由器/防火墙等非 server）采集流程**

步骤 1：与 Linux 相同，**`MonitorService`** 先按周期做 **可达性探测**：优先 Ping；若 Ping 失败则对 **SNMP 端口（如 161）** 与 **SSH 端口** 等尝试 **TCP 连通**，更新 **MySQL** 在线状态并可选写 InfluxDB **`device_status`**；状态变化可触发告警。  
步骤 2：在 **`snmp.collect.enabled=true`** 且设备已在台账中配置 **SNMP 参数** 的前提下，**`DeviceSnmpCollectService`** 按周期对该设备发起 **SNMP** 采集（跳过已离线设备），将 CPU、内存等结果在 **`snmp.use-redis=true`** 时写入 **Redis** 设备维度缓存。  
步骤 3：若 SNMP 采集失败或未配置 SNMP，且设备已配置 **SSH/Telnet** 并满足实现中的过滤条件，则由 **`DeviceSshCollectService`** 按周期通过 **SSH 或 Telnet（如端口 23）** 执行厂商 **显示类命令**，解析文本后写入 **Redis**，与 SNMP 缓存并列供查询。  
步骤 4：用户在前端打开实时指标时，后端从 **Redis** 读取网络设备最新快照，并与 **MySQL** 台账、**InfluxDB** 中与本设备相关的时序数据（若有）一并聚合后通过 **REST** 返回前端。

**三、两类设备的共性与差异小结**

共性：**在线/离线**均以 **`MonitorService`** 的周期性探测为主链，结果进 **MySQL**（及可选 InfluxDB 状态测量），并参与告警。  
差异：**Linux 性能指标**以 **Telegraf → InfluxDB** 为主、后端 **读 Influx + SSH 补缺**；**网络设备性能指标**以 **SNMP → Redis** 为主、**SSH/Telnet 命令采集 → Redis** 为辅。

#### 4.5.6 多存储数据分工小结

| 数据类型 | 主要落点 | 典型用途 |
|----------|----------|----------|

### 4.6 核心类设计

后端采用 Spring Boot 典型分层结构，将类职责划分为控制层、业务层、持久层与领域实体，并通过 WebSocket 组件承载远程终端会话。该划分有利于接口统一、业务复用与单元测试，也与第 5 章按模块实现的叙述方式一致。

#### 4.6.1 分层职责说明

| 层次 | 主要职责 | 典型类 |
|------|----------|--------|
| 控制层（Controller） | 接收 HTTP 请求、参数校验、调用业务服务并返回统一响应 | `AuthController`、`DeviceController`、`AlertController`、`InspectionController`、`AiChatController` 等 |
| 业务层（Service） | 封装领域逻辑：设备管理、监控探测、告警评估、巡检调度、AI 对话、配置备份、审计写入等 | `DeviceService`、`AlertService`、`InspectionService`、`AiChatService`、`ConfigBackupService`、`WebSshService` 等 |
| 持久层（Repository） | 基于 Spring Data JPA 完成实体持久化与查询 | `DeviceRepository`、`AlertRuleRepository`、`InspectionReportRepository`、`AiChatSessionRepository` 等 |
| 领域层（Entity） | 映射数据库表结构，表达用户、设备、告警、巡检、AI 会话等核心业务对象 | `SysUser`、`Device`、`AlertRule`、`AlertHistory`、`InspectionReport`、`AiChatSession` 等 |
| WebSocket 终端 | 维护浏览器与设备 SSH/Telnet 之间的会话转发 | `SshWebSocketHandler` 依赖 `WebSshService` 建立与释放终端连接 |

控制层不直接操作数据库表，而是通过 Service 聚合业务规则；持久层仅负责数据访问，避免在 Repository 中编写复杂业务流程，从而保持“高内聚、低耦合”。

#### 4.6.2 核心类协作关系

系统主链路的类协作可概括如下：

（1）**认证与权限**：`AuthController` 调用 `UserService` 完成登录校验；菜单与角色由 `RoleMenuService` 与 `RoleController` 协同维护，实体侧通过 `SysUser`、`Role`、`UserRole`、`RoleMenu` 表达多对多关系。  
（2）**设备与监控**：`DeviceController` 调用 `DeviceService`、`MonitorService` 等维护设备主数据与探测状态；指标展示由 `MetricsController`、`MetricController` 与 `DeviceStatsService`、`InfluxDbMetricService` 等协作完成。  
（3）**告警闭环**：`AlertController` 调用 `AlertService` 管理规则与历史；`AlertService` 与 `AlertRule`、`AlertHistory` 实体及对应 Repository 绑定，并可联动邮件与自动修复相关服务。  
（4）**系统巡检**：`InspectionController` 调用 `InspectionService` 生成报告与明细；定时任务由 `InspectionScheduleService` 触发同一套巡检逻辑。  
（5）**远程运维**：`BatchCommandController` 调用 `BatchCommandService` 执行批量命令；浏览器终端由 `SshWebSocketHandler` 与 `WebSshService` 建立长连接并转发字节流。  
（6）**AI 运维助手**：`AiChatController` 调用 `AiChatService` 完成会话管理与 LLM 调用；会话与消息由 `AiChatSession`、`AiChatMessage` 持久化。  
（7）**配置与审计**：`ConfigBackupController`、`SystemConfigController` 分别调用 `ConfigBackupService`、`SystemConfigService`；关键操作通过 `AuditService` 写入 `AuditLog`。

## 5 系统实现与效果

第 5 章介绍系统实现与效果，按业务模块说明系统关键功能的实现方式，并给出对应的页面效果展示建议。为便于论文排版与答辩展示，每个小节均可配 1~2 张模块界面截图。

### 5.1 登录认证模块实现

登录认证模块负责系统入口鉴权与权限初始化。用户输入账号密码后，后端完成身份校验并返回登录态信息，前端据此保存会话并进入首页。系统同时加载角色与菜单权限，确保不同角色看到的功能入口与可调用操作保持一致。  
**截图建议**：登录页面、登录成功后首页首屏。

### 5.2 首页仪表盘模块实现

首页仪表盘模块用于集中展示系统运行总览，包括设备健康统计、在线离线数量、告警摘要与关键趋势信息。该模块通过聚合后端接口数据，帮助运维人员在进入系统后快速掌握当前运行态势。  
**截图建议**：首页仪表盘总览页面（含统计卡片与图表区域）。

### 5.3 设备管理模块实现

设备管理模块实现设备信息的新增、修改、删除、查询与分组管理，支持按关键条件筛选设备列表。模块同时提供连通性检测与拓扑展示入口，为后续监控、告警与巡检提供基础设备数据。  
**截图建议**：设备列表页、设备拓扑页或设备编辑弹窗。

### 5.4 监控与告警模块实现

该模块由实时监控与告警管理两部分组成。监控部分通过定时探测与指标汇聚展示设备状态变化；告警部分基于规则进行阈值判断，触发后生成历史记录并支持状态跟踪与通知联动，形成“发现异常—触发告警—跟踪处理”的闭环。  
**截图建议**：实时指标页面、告警规则页面、告警历史页面。

### 5.5 远程运维模块实现

远程运维模块支持浏览器内终端访问与批量命令执行。用户选择目标设备后可建立远程会话进行操作；在多设备场景下，系统可并行下发命令并汇总结果，提升重复性运维任务的执行效率。  
**截图建议**：Web SSH 终端页面、批量命令执行结果页面。

### 5.6 系统巡检模块实现

系统巡检模块支持手动巡检与定时巡检。任务执行后生成巡检报告与设备级明细，展示正常、异常、离线等统计结果，并支持报告查看与导出。通过巡检机制可实现对设备健康状态的周期性评估。  
**截图建议**：巡检任务页面、巡检报告列表与报告详情页面。

### 5.7 AI 运维助手与系统管理模块实现

AI 运维助手支持多轮问答与会话管理，可结合设备与监控上下文生成运维建议。系统管理模块负责参数配置、配置备份恢复、用户与角色维护、操作审计及数据导出等治理能力，保障系统长期稳定运行与可追溯管理。  
**截图建议**：AI 助手会话页面、系统设置页面、操作审计页面。

### 5.8 本章小结

本章按模块说明了系统核心功能的实现方式，并结合页面展示给出了可视化效果说明。总体来看，系统已形成“登录认证—监控展示—告警处置—远程运维—巡检分析—系统治理”的完整能力链，能够满足中小规模运维场景下对可观测性、可操作性与可审计性的综合需求。

## 6 系统测试

### 6.1 黑盒测试目标与策略

为了确保 NetPulse 平台满足毕业设计验收要求，本章仅采用**黑盒功能测试**。测试从用户视角出发，不关注源码内部实现细节，重点验证“输入—处理—输出”是否符合预期、业务流程是否闭环、关键页面与接口是否可用。

本章测试目标包括：  
（1）验证登录鉴权、设备管理、监控告警、系统巡检、**Web SSH 远程终端**等核心功能正确性；

## 7 总结与展望

### 7.1 总结

针对小型企业运维监控场景中**分布式设备纳管困难**、**人工巡检效率偏低**以及**告警触达与处置不够及时**等问题，本文设计并实现了一套基于 **Spring Boot** 与 **Vue 3** 前后端分离架构的 NetPulse 运维监控平台。全文工作总结可归纳为**数据采集、告警、远程运维、运维智能支持**四条主线，具体如下。

（1）**数据采集**。为适应 Linux 服务器与网络设备在接口和协议上的差异，系统采用多源采集路径：**Telegraf** 采集主机 CPU、内存等性能指标；**SNMP（Snmp4j）** 获取网络设备侧运行数据；在 SNMP 不可用或需补充时，通过 **SSH/Telnet（JSch、Commons Net）** 登录设备执行命令并解析输出。平台定时执行**可达性探测**并将设备状态等时序数据写入 **InfluxDB**（如 `device_status` 等测量），主机指标亦由 Telegraf 汇入 InfluxDB，支撑趋势与历史曲线；网络设备实时指标写入 **Redis**，供监控大盘与实时接口快速读取。设备台账、采集与告警相关的业务配置及结果明细等结构化数据落在 **MySQL**，表结构由 **Flyway** 版本化管理，保证采集链路可追溯、可演进。

（2）**告警**。系统在**设备上下线、指标越限**等条件下形成可配置的告警闭环：支持**告警规则**维护与**告警历史**查询、处理状态记录；结合定时任务对当前指标与规则进行评估，使异常能够**及时发现并落库展示**；通知侧可对接邮件等渠道，便于运维人员接收与后续处置。该模块与数据采集、设备状态探测衔接，使“采得到、看得见、能报警”在同一平台内完成。

（3）**远程运维**。为减少逐台登录与工具分散带来的成本，平台提供 **Web SSH/Telnet**（浏览器经 **WebSocket** 与后端会话代理连接设备）、**批量命令**在多设备上统一执行并汇总结果，以及**配置备份与恢复**等能力；配合**基于角色与菜单的访问控制**与**操作审计、数据导出**，在远程操作的同时保留权限边界与操作留痕，支撑日常变更与应急处置场景。

（4）**运维智能支持**。平台**集成大模型 API**，提供 **AI 运维助手**，可结合设备与监控上下文进行故障排查类问答；在**系统巡检**流程中，可对巡检报告**生成文字总结结论**，辅助从长报告中提炼要点。该能力以可配置的第三方接口接入，与核心业务解耦，便于按环境启用或替换模型服务，在传统监控与自动化操作之外补充**智能化辅助决策与文档化输出**。

经第 6 章**黑盒功能测试**及联调过程中的**稳定性观察**，上述主线涉及的核心链路行为与预期一致，**功能完整**，达到了本毕业设计在实现与工程可交付性方面的**预期目标**。

### 7.2 系统不足与展望

NetPulse 在毕业设计周期内已打通主要业务链路并通过第 6 章黑盒测试，但面向长期运行与生产化落地仍存在差距。以下**分两部分撰写**：**（一）系统不足**归纳当前局限；**（二）系统展望**给出对应改进方向；**（三）分步实施建议**说明落地顺序，便于后续工作计划对照。

**（一）系统不足**

综合前文实现与测试范围，当前不足主要体现在以下三方面。

其一，**告警能力仍偏基础**。告警目前仍以**电子邮件**为主，实时性与移动端协同有限；对维护期**静默**、短时重复告警**聚合**、久未处理**升级**等治理策略缺乏系统化支持，告警密集时易造成信息疲劳。

其二，**平台自可观测性不足**。系统对**被管设备**在线与指标呈现相对完整，但对**监控系统自身**是否健康缺少成套观测手段；遇采集堆积、接口变慢、**WebSocket** 不稳、大模型等**外部依赖失败**等问题时，仍多依赖**单机日志人工排查**，定位效率不高。

其三，**安全、工程质量与扩展性有待加强**。**HTTPS** 全覆盖、口令与令牌策略、密钥与敏感配置的安全管理、**自动化测试**与接口回归覆盖仍需加强；设备与任务规模增大后，单实例承载探测与采集的压力、**高可用与容量规划**亦需进一步设计与验证。

**（二）系统展望**

针对上述不足，后续工作可从以下方面推进。

其一，**完善告警触达与治理**。在保留邮件等渠道基础上，扩展**企业微信、钉钉、飞书**等即时通知；逐步引入**告警静默**、**告警聚合**、**告警升级**，在保障关键事件不漏报的前提下降低无效打扰。

其二，**增强平台自监控并与观测栈联动**。建设**健康检查**、**链路追踪**（如 OpenTelemetry）、**关键指标暴露**（Micrometer 等，覆盖采集与告警调度耗时、接口与 **WebSocket**、外部 API 成功率与延迟等）。在**后期持续优化阶段**、规范稳定后，**可择机将指标接入 Prometheus**，用 **Grafana** 做大盘与平台侧告警；追踪与结构化日志**关联输出至统一日志平台**（或 Tempo/Jaeger 等），形成**指标—链路—日志**互证，实现**监控系统可监控自身**。

其三，**夯实安全、测试与扩展能力**。推广 **HTTPS**，强化密码与令牌、密钥安全注入；补充**单元测试与接口回归**；规模上升时推进采集与探测**拆分调度、多实例**及**压力测试**驱动的容量评估，提升长期可靠性。

**（三）分步实施建议**

为使展望可执行，建议按优先级分阶段推进：**第一步**先做告警渠道与 1～2 项治理策略试点；**第二步**落地健康检查、核心指标与链路追踪，在本系统内形成排障闭环；**第三步**在基础稳定后**按需**对接 Prometheus/Grafana 与统一日志/追踪后端；**第四步**并行推进 HTTPS、密钥治理、测试门禁及规模化架构优化。各步可部分重叠，但不宜在指标与日志规范未稳时过早引入过多外部组件，以免增加排障复杂度。
