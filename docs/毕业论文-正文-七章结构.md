## 1 前言（系统概述）

### 1.1 研究背景与意义

随着企业信息化与网络规模的不断扩大，数据中心与园区网络中服务器、交换机、路由器、防火墙等设备数量众多，运维人员需要实时掌握设备在线状态、性能指标与告警信息，并在故障发生时快速定位与处置。传统依赖人工登录设备执行命令、查看脚本输出的方式效率低、可追溯性差，且难以在设备数量增长时保持一致性。因此，构建一套集中式的网络监控与运维平台，实现设备发现、状态采集、指标可视化、告警管理与远程运维的自动化，具有重要的实际应用价值。

### 1.2 系统概述

本系统 NetPulse 是一套基于 B/S 架构的网络监控与运维平台，面向虚拟机 Linux 与网络设备（路由器、交换机、防火墙等），提供**设备统一管理、健康状态监控、实时指标展示、告警规则与历史、Web SSH 远程终端、批量命令执行、配置备份与恢复、用户权限与操作审计、以及 AI 运维助手**等核心功能。系统采用前后端分离技术栈：后端基于 Spring Boot 3，使用 MySQL 存储业务数据、InfluxDB 存储时序指标、Redis 缓存网络设备实时数据；前端基于 Vue 3 与 Vite，通过 RESTful 接口与 WebSocket 与后端通信。平台可用于中小规模网络环境的集中监控与日常运维，便于二次开发与集成现有监控数据源（如 Telegraf、外部脚本写入 Redis），对运维自动化具有较好的参考意义。

### 1.3 国内外相关现状

在网络监控与可观测性领域，国外已有 Zabbix、Prometheus、Grafana、Nagios 等成熟方案；国内不少企业基于上述系统进行二次开发或自研监控平台。在远程运维方面，Web SSH、堡垒机等产品已广泛应用。本系统在设计上借鉴“设备—告警规则—历史记录”的数据模型，并针对 Linux 主机（Telegraf + InfluxDB）与网络设备（Redis 按 IP 缓存）双数据源做了统一展示与权限集成。

## 2 系统关键技术

### 2.1 后端技术选型

#### 2.1.1 Spring Boot

本系统后端基于 **Spring Boot 3** 构建。选型理由：（1）与项目需求高度匹配——需大量 RESTful 接口、定时任务（设备健康检查、告警评估）、长连接（Web SSH 的 WebSocket 代理）及多数据源访问（MySQL、Redis、InfluxDB），Spring Boot 在上述方面均有成熟方案；（2）内嵌 Tomcat，打包为 jar 后可直接运行，部署简单；（3）依赖注入与分层架构便于控制器、服务、仓储分离，可维护性好；（4）文档与社区完善，与 MySQL、Redis、InfluxDB 等集成示例丰富。

#### 2.1.2 数据存储与缓存

- **MySQL**：业务主库，存储用户、角色、设备、告警规则与历史、配置备份、审计、Web SSH 会话、AI 会话与消息等，满足事务与复杂查询。
- **InfluxDB**：时序数据库，存储 Telegraf 写入的 CPU/内存等时序数据，适合指标随时间变化的场景。
- **Redis**：缓存网络设备按 IP 的 CPU/内存等实时数据，支持 Pipeline 批量读取，适合高并发访问。

#### 2.1.3 远程与采集相关

- **JSch**：实现 SSH 连接，用于 Web 终端与批量命令、告警自动修复。
- **Apache Commons Net**：实现 Telnet 连接（端口 23），用于不支持 SSH 的设备。
- **Snmp4j**：网络设备 SNMP（v2c/v3）采集，结果写 Redis 供实时指标展示。
- **Spring WebSocket**：Web SSH 的 WebSocket 代理，实现浏览器与设备终端双向转发。
- **Spring Mail**：告警邮件通知（可扩展）。
- **Spring Scheduling**：定时任务，如设备 Ping、告警规则评估、SNMP 采集。

### 2.2 前端技术选型

- **Vue 3 + Vue Router**：组件化开发，路由与菜单按角色控制。
- **Vite**：构建与开发服务器，热更新快，与 Vue 3 配合良好。
- **Axios**：HTTP 请求，对接后端 RESTful API。
- **xterm.js**：浏览器内终端，实现 Web SSH 的输入输出展示。
- **ECharts**：仪表盘、设备类型分布、实时指标饼图、时间线等图表。

### 2.3 外部依赖与部署

- **Telegraf**：部署在 Linux 服务器上，采集 CPU、内存等并写入 InfluxDB。
- **Nginx**（可选）：托管前端静态资源或反向代理后端。
- 构建与运行：后端 Maven + JDK 21；前端 Node.js / npm。

## 3 系统分析

### 3.1 业务需求

系统需支撑以下核心业务场景：（1）**设备统一纳管**：对服务器、路由器、交换机、防火墙等设备进行登记，记录管理 IP、类型、厂商型号、SSH 凭证与分组；（2）**状态与指标监控**：定期或按需对设备进行可达性检查（如 Ping），更新在线/离线/告警状态，并展示 CPU、内存等实时指标；（3）**告警管理**：支持按设备或设备类型配置告警规则，记录告警历史，并可扩展通知渠道与自动修复；（4）**远程运维**：通过 Web SSH 登录设备执行命令，支持批量命令下发与操作记录；（5）**配置与审计**：支持配置备份与恢复、系统参数与 API 配置、用户与角色权限、操作审计日志；（6）**智能辅助**：可选集成大模型 API，提供 AI 运维助手。

### 3.2 功能需求

- **认证与权限**：用户名密码登录，角色（管理员/运维/只读）控制菜单与接口访问，前后端权限一致。
- **设备管理**：设备增删改查、批量导入、分组筛选、健康汇总、Ping 与连通性检测、设备拓扑图。
- **数据采集与实时指标**：定时/手动 Ping 更新状态；Linux 设备指标来自 Telegraf/InfluxDB，网络设备来自 Redis；前端表格与饼图展示，按数据来源区分。
- **告警**：告警规则配置（设备/类型、指标、条件、严重程度）、告警历史、可选自动修复与通知。
- **Web SSH 与批量命令**：浏览器内 SSH/Telnet 终端；多设备批量执行命令并汇总结果。
- **配置备份与系统配置**：设备/规则/系统配置 JSON 备份与恢复；系统名称、时区、API 密钥等键值配置。
- **审计与导出**：关键操作写审计日志；设备健康、告警历史、审计日志等 CSV 导出。
- **AI 助手**：多会话、多轮对话，结合设备与监控上下文调用大模型 API。

### 3.3 非功能需求

- **可用性**：核心接口具备异常处理与超时控制；中间件配置与高可用在部署文档中说明。
- **可扩展性**：设备类型、监控指标、告警渠道等通过表结构与配置扩展，避免硬编码。
- **数据持久化**：业务数据落 MySQL；时序落 InfluxDB；网络设备实时数据经 Redis 缓存。

## 4 系统设计

### 4.1 系统架构

系统采用 B/S 架构与前后端分离。用户通过浏览器访问前端（Vue 3 + Vite），前端通过 HTTP/WebSocket 与后端（Spring Boot 3）通信；后端对接 MySQL（nebula_ops）、InfluxDB、Redis。数据分工：MySQL 存储业务数据；InfluxDB 存储 Telegraf 时序；Redis 存储网络设备按 IP 的实时数据。整体架构便于在不同环境中部署与扩展。（可配合 `docs/系统功能架构图.md` 中的总体架构图。）

### 4.2 功能模块划分

- **认证与权限模块**：登录、会话、角色与菜单控制。
- **设备管理模块**：设备 CRUD、批量导入、分组筛选、健康汇总、Ping、设备拓扑。
- **数据采集与指标模块**：状态更新、InfluxDB/Redis 查询、实时指标聚合与来源标识。
- **告警模块**：告警规则 CRUD、告警评估、告警历史、自动修复、通知记录查询。
- **Web SSH 与批量命令模块**：WebSocket 代理、SSH 连接、批量命令执行与结果汇总。
- **配置备份与系统配置模块**：备份创建与恢复、系统配置键值管理。
- **AI 助手模块**：会话与消息持久化、大模型 API 调用。
- **审计与导出模块**：审计日志写入与查询、CSV 导出。

### 4.3 系统逻辑流程（供绘制流程图）

- **用户登录与权限**：用户输入账号密码 → 后端校验 sys_user、user_role、role → 返回用户信息与角色 → 前端保存并控制菜单。
- **设备管理与健康**：设备增删改查/批量导入 → 写入 device 表；定时/手动健康检查 → Ping 设备 IP → 更新 status、last_poll_time → 前端展示列表与健康汇总。
- **实时指标**：前端请求指标接口 → 后端从 InfluxDB（Linux）、Redis（网络设备）获取数据 → 与 device 表匹配后合并返回 → 前端展示表格与饼图。
- **告警**：配置规则写入 alert_rule；评估时根据规则与当前指标判断 → 触发则写 alert_history、可选自动修复；恢复后更新 alert_history 状态与结束时间。
- **Web SSH**：用户选择设备 → 前端建立 WebSocket → 后端用 device 的 SSH 凭证建立 SSH → 双向转发；可选记录 webssh_session。
- **配置备份**：创建备份 → 汇总设备/规则/系统配置 → 序列化为 JSON 写入 config_backup；恢复时解析 JSON 写回相应表。
- **AI 助手**：用户发消息 → 加载会话历史 → 调用大模型 API → 写入 ai_chat_message 并返回。

### 4.4 数据库设计

#### 4.4.1 设计原则

与业务实体一致、适度规范化、便于扩展、关键表保留审计与时间戳。

#### 4.4.2 E-R 与实体关系（供绘制 E-R 图）

核心实体：sys_user、role、device、alert_rule、alert_history、config_backup、system_config、audit_log、webssh_session、ai_chat_session、ai_chat_message、alert_template、user_role、role_permission 等。关系：用户与角色多对多（user_role）；角色与权限多对多（role_permission）；设备与告警规则/告警历史/Web SSH 会话为一对多；告警历史与通知记录一对多；AI 会话与消息一对多；配置备份、审计日志与用户多对一。绘图时实体用矩形、联系用菱形或连线并标注 1、n。

#### 4.4.3 主要表结构说明

- **device**：id、name、ip、type、vendor、model、ssh_port、ssh_user、ssh_password、remark、group_name、status、last_poll_time、create_time、update_time、deleted 等。
- **sys_user**：id、username、password、real_name、email、enabled、create_time、update_time。
- **role**、**user_role**：角色及用户-角色关联。
- **alert_rule**：id、name、device_id、device_types、metric_key、rule_condition、severity、enabled、auto_fix_*、create_time、update_time。
- **alert_history**：id、rule_id、device_id、metric_key、trigger_value、start_time、end_time、status、severity、message 等。
- **config_backup**：id、name、backup_type、summary、content（LONGTEXT）、user_id、create_time。
- **audit_log**：id、user_id、username、action、target_type、target_id、detail、ip、create_time。
- **webssh_session**：id、user_id、device_id、session_id、start_time、end_time、input_log 等。
- **ai_chat_session**、**ai_chat_message**：会话与消息。
- **system_config**：id、config_key（唯一）、config_value、remark 等。

详细字段与索引参见 `docs/后端与数据库表结构对照.md` 及现有建表脚本。

## 5 系统实现与效果

### 5.1 认证与权限实现

采用表单登录：前端提交用户名与密码至 POST /auth/login，后端在 sys_user 中校验并查询 user_role、role，返回用户信息与角色列表；前端将用户信息存入 localStorage，后续请求携带用户标识（如 X-User-Name）。前端根据角色控制菜单（如仅管理员可见用户管理、系统设置）；角色与权限通过 RBAC 模型（role、sys_permission、role_permission）实现，便于扩展细粒度权限。

### 5.2 设备管理实现

设备列表增删改查与批量导入；列表支持按分组筛选、排序；健康汇总接口按 device 表 status 分组统计 normal、warning、critical、offline 数量；单设备 Ping 与连通性检测由专门接口实现。设备拓扑图页面展示设备节点、按分组筛选，点击设备可进入 Web SSH。

### 5.3 数据采集与实时指标实现

设备状态通过定时或手动 Ping 更新，结果写 device 表并可写 InfluxDB。实时指标：Linux 设备从 InfluxDB 按 host 查询 Telegraf 数据；网络设备从 Redis 按 IP 或 key 模式读取（Pipeline 批量）；与 device 表匹配后合并 stats 与 statsSource 返回前端。前端每 10 秒刷新，展示表格与数据来源饼图。

### 5.4 告警实现

告警规则在 alert_rule 表配置；评估由定时任务或状态变更触发，满足条件则写 alert_history（status=firing），可选执行自动修复（SSH/本地脚本）；恢复时更新 status=resolved、end_time。告警历史支持分页查询、人工标记与 CSV 导出。

### 5.5 Web SSH 与批量命令实现

Web SSH：前端建立 WebSocket（/api/ws/ssh?deviceId=xxx），后端从 device 表取 SSH 凭证与设备建立 SSH，终端输入输出在 WebSocket 与 SSH 间双向转发；可选记录 webssh_session。批量命令：用户选择多台设备并输入命令，后端逐台 SSH 执行并汇总结果返回。

### 5.6 配置备份与系统配置实现

备份时将设备、告警规则、系统配置等序列化为 JSON 写入 config_backup；恢复时解析 JSON 写回相应表。系统配置存 system_config 键值对，供前端系统设置页与后端 AI 服务使用。

### 5.7 AI 助手与审计导出实现

AI 助手：多会话（ai_chat_session）、多轮消息（ai_chat_message）；发送消息时加载历史并调用大模型 API，将回复写入库并返回。审计：关键操作由 AuditService 写入 audit_log；导出接口提供设备健康、告警历史、审计日志等 CSV 导出。

## 7 总结与展望

### 7.1 总结

本文设计并实现了基于 Spring Boot 与 Vue 的网络监控与运维平台 NetPulse，完成了设备管理、健康状态监控、实时指标展示（Linux + 网络设备双数据源）、告警规则与历史、Web SSH、批量命令、配置备份、系统配置、用户权限、操作审计与 AI 助手等核心功能。系统采用前后端分离架构，基于 MySQL、InfluxDB、Redis 进行数据持久化与缓存，数据库设计规范、实体关系清晰，并提供了系统逻辑流程与 E-R 关系的描述，便于绘制答辩用流程图与 E-R 图。平台可用于中小规模网络环境的集中监控与日常运维，具有较好的实用性与可扩展性。

### 7.2 不足与展望

当前可改进之处：（1）用户密码建议改为 BCrypt 等不可逆摘要并加强会话管理；（2）告警通知渠道（邮件、钉钉、企业微信等）需进一步对接与配置；（3）InfluxDB 与 Redis 的高可用、备份策略需结合生产环境规划；（4）实时指标与告警评估的并发与限流可根据设备规模优化。后续可在上述方面增强，并扩展更多采集方式、可视化大屏与报表，以更好地支撑企业级运维场景。
