# 基于 NetPulse 的网络监控与运维平台设计与实现

---

## 摘要

随着企业信息化与网络基础设施规模的持续扩张，数据中心与园区网络中的服务器、交换机、路由器及防火墙等设备数量不断增多，运维人员需要在统一视图中掌握设备在线状态、性能指标与告警信息，并能在故障或变更场景下快速登录设备执行处置。传统依赖分散登录与手工记录的方式难以保证效率与可追溯性，因此有必要构建一套集中式的网络监控与运维平台。

本文设计并实现了一套名为 NetPulse 的网络监控与运维平台。系统采用浏览器与服务器（B/S）架构及前后端分离模式：服务端基于 Spring Boot 3 与 Java 21，提供 RESTful 接口与 WebSocket 长连接；客户端基于 Vue 3 与 Vite 实现单页应用。数据存储方面，业务数据与权限、审计、巡检报告等持久化于 MySQL；Linux 主机通过 Telegraf 采集的 CPU、内存等时序数据存储于 InfluxDB；网络设备 SNMP 等采集结果以 Redis 为缓存供实时展示。平台实现了设备台账与健康探测、实时指标聚合、告警规则与告警历史、系统巡检与可选的大模型文字结论、浏览器内 Web SSH 与 Telnet、批量命令执行、配置备份与恢复、系统与第三方 API 配置、基于角色与菜单编码的访问控制、操作审计与多种数据导出，以及 AI 运维助手等能力。数据库结构遵循规范化原则，并通过 Flyway 进行版本化迁移管理。

在开发与联调基础上，对登录与权限、设备管理、指标与告警、巡检、远程终端与批量命令、备份与配置、审计与导出等模块进行了功能验证。结果表明，各模块行为与预期一致，接口响应与错误提示清晰，能够满足实验环境及中小规模网络场景下集中监控与运维辅助的毕业设计目标。

**关键词**：网络监控；运维平台；Spring Boot；Vue；MySQL；InfluxDB；Redis；系统巡检；Web SSH

---

## Abstract

With the expansion of enterprise networks, administrators need a unified view of device availability, performance metrics, and alerts, as well as efficient remote access to managed devices. This thesis presents NetPulse, a browser-based monitoring and operations platform built with Spring Boot 3 and Vue 3 in a separated front-end/back-end architecture. MySQL stores business and configuration data; InfluxDB stores time-series metrics from Telegraf; Redis caches real-time data for network devices. The system provides device management, health probing, real-time dashboards, alerting, scheduled and manual inspection reports with optional LLM-generated summaries, Web SSH/Telnet via WebSocket, batch command execution, configuration backup and restore, role-based menu access, audit logging and exports, and an AI assistant. Functional verification shows that core modules behave as designed and meet the objectives of the graduation project.

**Keywords**: Network monitoring; Operations platform; Spring Boot; Vue; MySQL; InfluxDB; Redis

## 1 绪论

### 1.1 研究背景与意义

随着企业信息化与网络规模的不断扩大，数据中心与园区网络中服务器、交换机、路由器、防火墙等设备数量众多，运维人员需要实时掌握设备在线状态、性能指标与告警信息，并在故障发生时快速定位与处置。传统依赖人工逐台登录设备执行命令、查看脚本输出的方式效率较低，操作过程难以统一记录与审计，且在设备数量增长时难以保持策略与数据的一致性。因此，构建一套集中式的网络监控与运维平台，实现设备台账统一、状态采集、指标可视化、告警管理与远程运维的规范化与自动化，具有明确的工程应用价值，亦符合本毕业设计选题的实践导向。

### 1.2 系统概述

本文所述 NetPulse 系统是一套基于 B/S 架构的网络监控与运维平台，面向虚拟机 Linux 环境与网络设备（路由器、交换机、防火墙等），在功能上涵盖设备统一管理、健康状态监控、实时指标展示、告警规则与历史记录、Web SSH 远程终端、批量命令执行、配置备份与恢复、用户与角色权限、操作审计以及 AI 运维助手等。系统在技术实现上采用前后端分离：后端基于 Spring Boot 3，使用 MySQL 存储业务数据、InfluxDB 存储时序指标、Redis 缓存网络设备实时数据；前端基于 Vue 3 与 Vite，通过 HTTP 调用 REST 接口、通过 WebSocket 支撑终端类业务。平台适用于中小规模网络环境的集中监控与日常运维，并可与现有采集体系（如 Telegraf、InfluxDB）及 Redis 缓存策略对接，便于后续扩展。

### 1.3 国内外相关研究与应用现状

在网络监控与可观测性领域，国际上已有 Zabbix、Prometheus、Grafana、Nagios 等成熟开源或商业方案；国内许多单位在开源组件基础上进行二次开发或建设自研监控平台。在远程运维方向，Web 终端、堡垒机与自动化运维工具已广泛落地。与上述通用平台相比，本设计以毕业设计可实现的规模为约束，借鉴“设备—告警规则—告警历史”等通用数据建模思路，并针对 Linux 主机（Telegraf 与 InfluxDB）与网络设备（Redis 按设备维度缓存）两类数据源进行统一展示与权限集成，突出可实现的工程完整性与文档化程度。

## 2 系统关键技术

### 2.1 后端技术选型

#### 2.1.1 Spring Boot 框架

后端采用 Spring Boot 3 构建。该框架能够较好地覆盖本课题对 REST 接口、定时任务（设备健康检查、告警评估）、WebSocket（Web 终端代理）以及多数据源访问（MySQL、Redis、InfluxDB）的需求；内嵌 Servlet 容器，打包为可执行 JAR 后便于部署；依赖注入与分层架构有利于控制层、业务层与持久层职责分离，便于在毕业设计周期内完成开发与文档撰写。

#### 2.1.2 数据存储与时序库

MySQL 作为业务主库，存储用户、角色与菜单权限、设备、告警规则与历史、配置备份、审计日志、巡检报告与明细、AI 会话与消息等，满足事务与复杂查询需求。InfluxDB 用于存储 Telegraf 写入的 CPU、内存等时序数据，适用于指标随时间变化的查询场景。Redis 用于缓存网络设备按 IP 或 key 组织的实时指标数据，并支持批量读取以降低接口延迟。

#### 2.1.3 远程连接与采集组件

系统使用 JSch 实现 SSH 连接，用于 Web 终端、批量命令及告警自动修复等场景；使用 Apache Commons Net 实现 Telnet（如端口 23）连接；使用 Snmp4j 完成网络设备 SNMP v2c/v3 采集并将结果写入 Redis；使用 Spring WebSocket 实现浏览器与后端之间的长连接转发；Spring Mail 可用于告警邮件扩展；Spring Scheduling 承担设备探测、告警评估、SNMP 定时采集及巡检定时任务等。

### 2.2 前端技术选型

前端采用 Vue 3 与 Vue Router 实现组件化开发与路由管理，菜单与路由根据角色权限进行控制。构建工具选用 Vite，以获得较快的开发体验。HTTP 访问使用 Axios 与后端 REST 接口交互。Web 终端界面采用 xterm.js 渲染。仪表盘与实时指标等可视化采用 ECharts 绘制图表。

### 2.3 外部依赖与运行环境

在 Linux 服务器上可部署 Telegraf 采集主机指标并写入 InfluxDB。生产环境可使用 Nginx 托管前端静态资源并反向代理后端接口与 WebSocket。后端构建采用 Maven，运行依赖 JDK 21；前端依赖 Node.js 与 npm 完成依赖安装与打包。

## 3 需求分析

本章从业务目标出发，对平台需满足的业务需求、功能需求与非功能需求进行分析，为系统设计与实现提供依据。

### 3.1 业务需求分析

结合企业日常运维工作流程，平台的业务目标可归纳为“统一纳管、实时感知、快速处置、过程留痕、持续优化”。围绕该目标，系统需满足以下核心业务场景。

第一，设备统一纳管场景。系统需将服务器与网络设备纳入统一台账，实现设备信息集中维护与分类管理，减少资产信息分散导致的管理盲区。

第二，状态持续监测场景。系统需持续呈现设备连通状态与关键运行指标，帮助运维人员在异常早期快速发现问题并定位影响范围。

第三，告警闭环处置场景。系统需支持告警从触发、通知、处理到恢复的全过程管理，提升故障响应效率并降低人工巡检压力。

第四，远程运维协同场景。系统需在浏览器环境中提供设备远程操作能力，并支持多设备协同执行，满足批量运维与应急处置需求。

第五，配置安全治理场景。系统需支持配置备份与恢复，并记录关键操作行为，确保运维活动可追溯、可审计。

第六，智能辅助运维场景。系统需提供面向运维语境的智能问答与巡检分析能力，辅助运维人员形成更高效的排障与决策方式。

### 3.2 功能需求分析

基于上述业务场景，系统功能需求可分解为以下八个模块：

（1）认证与权限管理：支持用户登录、身份识别与角色授权，确保不同角色在可见菜单与可执行操作上的权限隔离。  
（2）设备管理：支持设备信息维护、条件检索、分组管理与健康概览，满足运维资产管理与定位需求。  
（3）监控与指标展示：支持设备状态与核心性能指标统一展示，为运行态分析与异常识别提供依据。  
（4）告警管理：支持规则配置、事件记录、处理状态维护与通知联动，形成可跟踪的告警处理闭环。  
（5）系统巡检：支持按需巡检与周期巡检，输出可查看、可归档的巡检结果与报告。  
（6）远程运维：支持浏览器端远程终端与批量命令执行，提升跨设备操作效率。  
（7）AI 运维助手：支持会话式问答与上下文辅助，提升故障排查与运维决策效率。  
（8）系统管理：支持系统参数维护、配置备份恢复、操作审计与数据导出等治理能力。

功能模块之间需满足“相互协作但边界清晰”的要求：认证与权限为全局基础，设备与监控告警构成业务主链，巡检与 AI 能力形成分析增强，系统管理提供治理支撑。

### 3.3 非功能需求分析

除功能能力外，系统还需满足以下非功能目标：

（1）性能与可扩展性：在既定设备规模下保持可接受响应时间，并支持随规模增长进行容量扩展与参数调优。  
（2）可用性与稳定性：关键功能应具备稳定运行能力，异常场景下可提供明确提示并支持故障恢复。  
（3）安全性：系统应满足身份认证、权限控制与敏感信息保护要求，降低越权访问与数据泄露风险。  
（4）可维护性：系统应具有清晰模块边界与一致命名规范，便于后续功能迭代、问题排查与版本演进。  
（5）可部署性与兼容性：系统应支持主流运行环境部署，部署步骤可复现，满足教学与中小规模应用场景。  
（6）可审计性：关键操作过程应可追踪，保证运维活动可核查、可复盘。

### 3.4 本章小结

本章从业务目标出发，明确了系统在设备纳管、状态感知、告警处置、远程运维、配置治理与智能辅助等方面的需求边界，并对功能需求与非功能需求进行了分解。第 4 章将在此基础上进一步说明系统架构、模块设计、流程设计与数据库设计，实现“先需求、后设计”的章节衔接。

## 4 系统设计

第 4 章围绕系统设计展开，重点说明系统总体架构、功能模块划分、关键业务逻辑流程与数据库设计，为第 5 章的实现细节提供结构化依据。相关图示（架构图、功能结构图、流程图、类图与 E-R 图）统一维护在项目 `docs` 目录。

### 4.1 系统总体架构

系统采用 B/S 架构与前后端分离模式。前端基于 Vue 3 + Vite 构建单页应用，主要通过 REST 接口与后端交互，统一以 `/api` 作为接口前缀；远程终端等长连接场景通过 WebSocket 实现双向通信。

后端基于 Spring Boot 3，按“控制层—业务层—持久层”组织：控制层负责参数校验与接口编排，业务层封装巡检、告警、采集、备份与 AI 对话等领域逻辑，持久层以 Spring Data JPA 与数据库交互。系统中的定时任务（状态探测、告警评估、巡检调度等）由 Spring Scheduling 统一调度。

数据层采用分工式设计：MySQL 存储业务主数据（用户权限、设备、告警、巡检、审计、配置等）；Redis 提供网络设备实时指标与 SNMP 结果缓存；InfluxDB 存储 Telegraf 写入的时序指标。该组合兼顾事务一致性、实时读取性能与时序查询效率。

外部依赖包括被管设备（SSH/Telnet/SNMP）、SMTP 邮件服务（可选）和大模型 HTTP API。总体上形成“前端可视化 + 后端编排 + 多存储协同 + 外部设备接入”的架构闭环。

### 4.2 功能模块划分

根据系统功能模块图，监控运维系统由 8 个一级模块构成，各模块及其二级功能如下。

（1）登录认证模块：账号密码登录、菜单权限配置、权限验证、设备鉴权与校验。  
（2）首页仪表盘模块：设备健康汇总、在线/离线统计、告警摘要、设备类型分布。  
（3）设备管理模块：设备列表与增删改查、设备拓扑可视化、设备分布分析、Ping 连通性检测。  
（4）监控与告警模块：实时指标展示、告警规则管理、告警历史查询、邮件通知联动。  
（5）远程运维模块：Linux 运维控制台、网络设备运维控制台、Web SSH、远程会话管理。  
（6）系统巡检模块：定时巡检、巡检任务执行报告、巡检报告 PDF 导出、巡检报告生成。  
（7）AI 运维助手模块：智能会话、设备与监控上下文、故障诊断分析、大模型问答。  
（8）系统管理模块：用户与角色权限、系统设置、配置备份与恢复、操作审计。

上述划分与功能模块图保持一致，形成“认证入口—运行监控—处置执行—治理支撑”的模块化体系。各模块在边界内独立实现，通过统一接口与权限体系进行协作，满足系统可扩展与可维护要求。

### 4.3 系统逻辑流程

系统整体流程以“登录认证—功能分流—统一退出”为主线，具体过程如下。

（1）用户进入登录页面并输入用户名、密码后提交登录请求，系统对账号与密码进行校验。  
（2）当校验失败时，系统提示登录失败并返回输入环节，用户可重新输入后再次提交。  
（3）当校验成功时，系统生成会话信息，随后加载用户角色与菜单权限，完成首页初始化并进入首页仪表盘。  
（4）进入首页后，系统进入功能分流节点。用户可按需进入设备管理、监控与告警、系统巡检、远程运维、AI 运维助手、系统管理、用户管理等功能模块。  
（5）各功能模块处理完成后返回主界面，用户可继续选择其他模块，形成“登录后多模块循环操作”的业务模式。  
（6）当用户结束操作时执行退出登录，系统返回登录页面并结束本次业务流程。

该流程与功能分流图保持一致，完整体现了从认证入口到业务操作再到会话退出的闭环逻辑。

### 4.4 数据库设计

#### 4.4.1 设计原则

数据库设计以业务完整性与工程可维护性为导向，主要遵循以下原则：  
（1）**规范化与适度冗余结合**：核心业务表满足第三范式要求，减少数据冗余；对巡检明细、告警历史等高频查询场景保留必要业务冗余字段以提升检索效率。  
（2）**可扩展性**：通过配置表、中间表与状态字段支持功能演进，新增设备类型、告警规则或权限映射时尽量避免大规模表结构变更。  
（3）**可审计性**：关键业务操作保留时间戳与审计记录，保证操作可追踪、问题可复盘。  
（4）**一致性与完整性**：通过主键、唯一约束、外键或业务约束保证数据一致，确保实体关系表达清晰。  
（5）**可迁移性**：结构变更采用 Flyway 版本迁移管理，支持多环境统一部署与回溯。

#### 4.4.2 概念结构设计

系统概念模型围绕“用户权限、设备监控、告警闭环、巡检分析、配置治理”五类业务对象展开，核心实体包括：`sys_user`、`role`、`user_role`、`role_menu`、`device`、`alert_rule`、`alert_history`、`inspection_report`、`inspection_item`、`config_backup` 等。

实体间主要关系如下：  
（1）用户与角色为多对多关系，通过 `user_role` 建立关联；  
（2）角色与菜单权限为一对多关系，通过 `role_menu` 映射可访问菜单；  
（3）设备与告警规则为一对多关系，告警规则与告警历史为一对多关系；  
（4）巡检报告与巡检明细为一对多关系，明细记录设备级检测结果；  
（5）用户与配置备份为一对多关系，用于记录备份操作来源。

## 5 系统实现与效果

第 5 章介绍系统实现与效果，按业务模块说明系统关键功能的实现方式，并给出对应的页面效果展示建议。为便于论文排版与答辩展示，每个小节均可配 1~2 张模块界面截图。

### 5.1 登录认证模块实现

登录认证模块负责系统入口鉴权与权限初始化。用户输入账号密码后，后端完成身份校验并返回登录态信息，前端据此保存会话并进入首页。系统同时加载角色与菜单权限，确保不同角色看到的功能入口与可调用操作保持一致。  
**截图建议**：登录页面、登录成功后首页首屏。

### 5.2 首页仪表盘模块实现

首页仪表盘模块用于集中展示系统运行总览，包括设备健康统计、在线离线数量、告警摘要与关键趋势信息。该模块通过聚合后端接口数据，帮助运维人员在进入系统后快速掌握当前运行态势。  
**截图建议**：首页仪表盘总览页面（含统计卡片与图表区域）。

### 5.3 设备管理模块实现

设备管理模块实现设备信息的新增、修改、删除、查询与分组管理，支持按关键条件筛选设备列表。模块同时提供连通性检测与拓扑展示入口，为后续监控、告警与巡检提供基础设备数据。  
**截图建议**：设备列表页、设备拓扑页或设备编辑弹窗。

### 5.4 监控与告警模块实现

该模块由实时监控与告警管理两部分组成。监控部分通过定时探测与指标汇聚展示设备状态变化；告警部分基于规则进行阈值判断，触发后生成历史记录并支持状态跟踪与通知联动，形成“发现异常—触发告警—跟踪处理”的闭环。  
**截图建议**：实时指标页面、告警规则页面、告警历史页面。

### 5.5 远程运维模块实现

远程运维模块支持浏览器内终端访问与批量命令执行。用户选择目标设备后可建立远程会话进行操作；在多设备场景下，系统可并行下发命令并汇总结果，提升重复性运维任务的执行效率。  
**截图建议**：Web SSH 终端页面、批量命令执行结果页面。

### 5.6 系统巡检模块实现

系统巡检模块支持手动巡检与定时巡检。任务执行后生成巡检报告与设备级明细，展示正常、异常、离线等统计结果，并支持报告查看与导出。通过巡检机制可实现对设备健康状态的周期性评估。  
**截图建议**：巡检任务页面、巡检报告列表与报告详情页面。

### 5.7 AI 运维助手与系统管理模块实现

AI 运维助手支持多轮问答与会话管理，可结合设备与监控上下文生成运维建议。系统管理模块负责参数配置、配置备份恢复、用户与角色维护、操作审计及数据导出等治理能力，保障系统长期稳定运行与可追溯管理。  
**截图建议**：AI 助手会话页面、系统设置页面、操作审计页面。

### 5.8 本章小结

本章按模块说明了系统核心功能的实现方式，并结合页面展示给出了可视化效果说明。总体来看，系统已形成“登录认证—监控展示—告警处置—远程运维—巡检分析—系统治理”的完整能力链，能够满足中小规模运维场景下对可观测性、可操作性与可审计性的综合需求。

## 6 系统测试

### 6.1 测试环境

为验证系统在目标场景下的可用性与稳定性，测试环境按“前端 + 后端 + 多数据源”方式搭建，具体如下：

| 类别 | 配置说明 |
|------|----------|
| 操作系统 | Windows 10/11（开发联调）或 Linux（部署验证） |
| 后端运行环境 | JDK 21、Spring Boot 3 |
| 前端运行环境 | Node.js + npm，Vite 开发服务 |
| 关系型数据库 | MySQL 8.x（业务主数据） |
| 缓存数据库 | Redis 6.x（实时缓存数据） |
| 时序数据库 | InfluxDB 2.x（监控时序指标） |
| 接口与通信 | REST API（`/api`）+ WebSocket（终端会话） |

测试前完成以下准备：  
（1）执行数据库迁移脚本并初始化基础账号；  
（2）确认 MySQL、Redis、InfluxDB 网络连通；  
（3）完成前后端配置文件中的数据库地址、令牌、端口等参数设置；  
（4）准备至少 3~5 台测试设备（可包含 Linux 与网络设备）用于联调验证。

### 6.2 测试方法与评价指标

本系统采用“功能测试 + 接口测试 + 运行稳定性测试”的组合方式。

（1）功能测试：以模块为单位验证业务流程是否完整可用，重点关注登录鉴权、设备管理、监控告警、远程运维、巡检、AI 助手与系统管理等核心功能。  
（2）接口测试：通过浏览器开发者工具或 API 工具检查状态码、响应结构与异常返回信息。  
（3）稳定性测试：对定时任务、实时指标刷新、WebSocket 会话等长周期功能进行连续运行观察。  

评价指标主要包括：  
（1）功能正确性：功能是否按预期执行；  
（2）交互完整性：前后端数据是否一致；  
（3）异常可处理性：错误提示是否清晰；  
（4）运行稳定性：长时间运行是否出现明显故障或资源异常。

## 7 总结与展望

### 7.1 总结

本文围绕网络监控与运维应用场景，完成了 NetPulse 平台的设计与实现。系统采用 Spring Boot 与 Vue 3 前后端分离架构，整合 MySQL、Redis 与 InfluxDB 完成业务数据、缓存与时序存储，实现了设备管理、状态探测、实时指标、告警、系统巡检与大模型结论、Web SSH 与批量命令、配置备份与系统配置、AI 运维助手以及操作审计与导出等功能。数据库采用规范化设计与 Flyway 版本迁移，主要业务流程与实体关系已在第 4 章结合图示文档说明。经测试，核心模块运行稳定，能够达到本毕业设计在功能完整性与工程可交付性方面的预期目标。

### 7.2 不足与展望

后续可在以下方面继续改进：在安全性方面，全面使用 HTTPS，强化密码策略与令牌生命周期管理，对敏感配置采用密钥管理或环境注入；在告警方面，扩展企业微信、钉钉等通知渠道，并支持告警静默与聚合；在性能与高可用方面，对大规模设备场景进行探测分片与集群部署；在可观测性方面，为平台自身增加健康检查与链路追踪；在功能方面，可扩展报表大屏、资产配置自动发现及与工单系统对接等。
