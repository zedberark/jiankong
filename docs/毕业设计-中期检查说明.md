# 毕业设计中期检查说明

**项目名称**：基于 NetPulse 的网络监控与运维平台设计与实现  
**检查类型**：中期自查（完成进度、存在问题、解决方案）  
**说明**：本文档用于毕业设计中期检查，重点说明**完成进度情况**、**存在的问题**以及**拟采取的解决方案**。中期检查不要求毕业设计已完全完成，而是对当前阶段进展与后续计划进行说明。

---

## 重要说明：毕业设计的重点

**毕业设计的重点在于毕业设计文档（Word/doc 文件），包括其内容与格式，而非程序代码。**  
中期检查及最终答辩时，应突出论文/设计说明书的结构完整度、章节内容与格式规范；系统代码作为支撑材料，用于说明“设计与实现”的可行性及效果，但不替代文档本身。后续工作将优先保证论文各章节内容充实、格式符合学校要求，并在文档中正确引用架构图、ER 图、表结构、流程图等图示。

## 一、中期检查目的与当前完成情况总述

### 1.1 检查目的

中期检查旨在对毕业设计进行**阶段性**自查，重点反映：
- **完成进度情况**：当前阶段已完成的工作（文档与系统）及整体进度；

不要求系统功能全部完成，但需对“已完成—待完善—问题与对策”有清晰说明。

### 2.2 数据与采集

- **业务数据**：MySQL（nebula_ops）存储设备、用户、角色、告警、审计、配置备份等，表结构与 JPA 实体一致，参见 `docs/后端与数据库表结构对照.md`。
- **时序与缓存**：InfluxDB 存储 Telegraf 采集的 CPU/内存等时序；Redis 用于 SNMP 采集结果缓存及部分业务缓存。
- **采集能力**：MonitorService 定时 Ping 更新设备状态；SNMP 采集（DeviceSnmpCollectService/SnmpCollectService）写 Redis；Telegraf 需在 Linux 侧单独部署并配置写入 InfluxDB。
- **数据库迁移**：`db/migration/` 下存在 V2～V16 等 SQL 脚本，可用于库表版本化或手动执行；当前同时使用 JPA `ddl-auto: update` 便于开发。

### 2.3 文档与配置（支撑毕业设计文档的素材）

- **根目录**：`README.md`（运行步骤、环境说明、常见问题）；`README_DB_ADDON.md`（补充建表说明）。
- **docs/**：论文正文与提纲、万字版论文、项目结构说明、后端与表结构对照、系统功能架构图、ER 图、流程图、数据库表图、图示汇总等，为整理成正式毕业设计 doc 提供内容与图示来源。
- **配置**：后端 `application.yml` 完整；支持 `application-local.yml`、`application-secrets.yml` 等敏感信息隔离。
