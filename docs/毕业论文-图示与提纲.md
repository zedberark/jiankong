## 告警规则 — 系统功能流程图（文字描述，供画流程图用）

以下为**告警规则**从配置、评估、触发到恢复的完整功能流程，按步骤画出即可得到「告警规则系统功能流程图」（不是数据库 ER 图）。

### 流程一：告警规则的配置与管理

```
[ 用户 ] 
   |
   v
[ 前端：告警管理页 ]
   |
   +---> 新建规则：填写 规则名称、指标(metric_key)、条件(rule_condition)、严重程度、设备/设备类型、是否启用、是否自动修复 等
   |           |
   |           v
   |     POST /api/alerts/rules  --> [ 后端 AlertController ] --> 写入 alert_rule 表 --> 返回成功
   |
   +---> 或从模板创建：选择 alert_template --> POST /api/alerts/templates/{id}/apply --> 生成一条新 alert_rule
   |
   +---> 编辑规则：PUT /api/alerts/rules/{id} --> 更新 alert_rule
   |
   +---> 删除规则：DELETE /api/alerts/rules/{id} --> 删除 alert_rule
   |
   v
[ 规则列表 ] GET /api/alerts/rules
```

### 流程二：告警评估与触发（两条入口）

**入口 A：设备状态变化（上线/离线）**

```
[ 设备健康检查 / Ping 等 ] 发现设备状态变化（如 在线→离线 或 离线→在线）
   |
   v
[ AlertService.onDeviceStatusChange(device, 原状态, 新状态) ]
   |
   v
查询 metric_key = "device_status" 且 enabled=true 的告警规则
   |
   v
对每条规则：条件(rule_condition) 是否等于 "offline" 或 "online"？设备类型(device_types) 是否匹配？
   |
   +---> 不匹配 --> 跳过该规则
   +---> 匹配 --> 写入 alert_history（rule_id, device_id, metric_key=device_status, trigger_value=offline/online, status=firing, severity, message）
   |           |
   |           v
   |     若规则启用自动修复 --> [ RemediationService.runRemediationAsync ] --> 异步执行 SSH 命令或本地脚本
   |
   v
结束
```

**入口 B：指标阈值（CPU/内存）定时评估**

```
[ 定时任务 ] 每 N 分钟执行一次（如 5 分钟）
   |
   v
[ AlertService.evaluateMetricRules() ]
   |
   v
从 DeviceStatsService 获取当前所有设备的 CPU/内存 快照（来自 InfluxDB/Redis 等）
   |
   v
若快照为空 --> 记录日志并结束
   |
   v
对 metric_key = "cpu_usage" 和 "mem_usage" 的已启用规则分别处理：
   |
   v
对每条规则 + 每个设备：
   |
   +---> 规则若指定了 device_id：设备必须等于该 device_id，否则跳过
   +---> 规则若指定了 device_types：设备类型必须在列表中，否则跳过
   |
   v
取该设备当前指标值 value（CPU 或 内存百分比）
   |
   v
解析规则条件 rule_condition（如 ">80"、">=90"、"<10"）--> 判断 value 是否满足条件
   |
   +---> 不满足（未超阈值）：
   |        查找该 规则+设备 是否已有 status=firing 的告警历史
   |        若有 --> 将该条告警置为 resolved，写入 end_time，保存
   |        若没有 --> 无操作
   |
   +---> 满足（超阈值）：
   |        查找该 规则+设备 是否已有 status=firing 的告警历史
   |        若已有 --> 不重复建告警（避免刷屏）
   |        若没有 --> 新建 alert_history（rule_id, device_id, metric_key, trigger_value=当前值, start_time, status=firing, severity, message）
   |                  |
   |                  v
   |            若规则启用自动修复 --> RemediationService.runRemediationAsync（SSH 或本地脚本）
   |
   v
结束
```

### 流程三：告警历史与人工处理

```
[ 用户 ] 打开告警历史页
   |
   v
GET /api/alerts/history?page=0&size=15&status=firing&severity=critical 等（可选筛选）
   |
   v
[ 后端 ] 从 alert_history 表分页查询，按状态、严重程度、时间排序
   |
   v
[ 前端 ] 展示告警列表（规则名、设备、指标、触发值、开始时间、状态、严重程度、消息）
   |
   v
用户点击「标记已处理」
   |
   v
POST /api/alerts/history/{id}/resolve 或 POST /api/alerts/resolve/{id}
   |
   v
[ 后端 ] 将该条 alert_history 的 status 改为 resolved，end_time 设为当前时间，保存
   |
   v
[ 审计 ] 写入 audit_log（RESOLVE_ALERT）
```

### 流程四：告警汇总与通知（扩展）

```
GET /api/alerts/summary?group=可选分组
   |
   v
[ 后端 ] 统计：规则总数、当前 firing 状态下 critical/warning/info 数量、最近 10 条告警
   |
   v
返回给前端仪表盘或告警页

（通知记录表 notification_log 仅支持查询列表，实际邮件/钉钉等发送未实现）
```

### 画图时建议的图形

- **开始/结束**：椭圆
- **用户/系统/模块**：矩形
- **判断（是否匹配、是否满足条件）**：菱形
- **数据库/表（alert_rule、alert_history）**：圆柱或矩形注明“表”
- **箭头**：表示顺序与数据流，在箭头上可标“是/否”或“满足/不满足”

将上述四个流程（配置管理、状态触发、指标定时评估、历史与处理）在同一张或分张图中画出，即得到「告警规则系统功能流程图」。

## 一、系统逻辑流程图（文字描述，便于画图）

### 1. 总体架构流程（自上而下）

```
[ 用户/浏览器 ]
       |
       v
[ 前端 Vue3 (localhost:5173) ]  <--->  [ 后端 Spring Boot (8082/api) ]
       |                                      |
       |                                      +---> [ MySQL (nebula_ops) ]  设备/用户/告警/配置等
       |                                      +---> [ Redis ]               网络设备 CPU/内存缓存
       |                                      +---> [ InfluxDB ]            时序指标（Telegraf 写入）
       v
[ 实时指标 / 设备管理 / 告警 / Web SSH / 批量命令 / AI 助手 / 审计 ]
```

### 2. 用户登录与权限流程

```
开始 --> 用户输入 用户名/密码
       --> POST /auth/login
       --> 后端查询 sys_user 表，校验密码
       --> 成功：返回 user + roles，前端写入 localStorage
       --> 失败：返回 401，提示“用户名或密码错误”
       --> 前端根据 roles 显示菜单（ADMIN 可见用户管理、系统设置）
       --> 后续请求可带 Token 或 X-User-Name
       --> 结束
```

### 3. 设备管理与健康状态流程

```
设备管理 CRUD:
  添加/编辑设备 --> 填写 name, ip, type, ssh_user, ssh_password 等
                 --> POST/PUT /api/devices
                 --> 写入/更新 device 表
                 --> 返回设备列表

健康状态更新（定时或手动）:
  定时任务 / 手动刷新
       --> 对 device 表中未删除设备执行 Ping
       --> 根据结果更新 device.status (normal/warning/critical/offline)、last_poll_time
       --> 可选：写入 InfluxDB 设备状态时序
       --> 前端 GET /api/devices/health 获取 normal/offline/warning/critical 数量
```

### 4. 实时指标数据流（CPU/内存）

```
Linux 设备（服务器）:
  Telegraf 在主机上采集 CPU/内存
       --> 写入 InfluxDB (bucket: device_metrics, measurement: cpu/mem 等)
       --> 后端 GET /api/metrics/realtime
       --> DeviceStatsService 从 InfluxDB 按 host 查询最近数据
       --> 与 device 表按 名称/IP 匹配，得到 stats + statsSource=telegraf
       --> 返回前端展示

网络设备（路由器/交换机/防火墙）:
  外部/EVE 脚本 将采集结果写入 Redis
       --> Key: device:R1 等，Hash: ip, cpu, mem, collect_time
       --> 后端 GET /api/metrics/realtime
       --> 后端从 Redis 按 IP 读取 (device:* 或按 IP 的 key)
       --> 与 device 表按 管理 IP 匹配，得到 stats + 数据来源标识（Redis）
       --> 返回前端展示
```

### 5. 告警规则与告警历史流程

```
告警规则（alert_rule）:
  创建/编辑规则 --> 选择设备或设备类型、metric_key、条件(如 >80)、严重程度
                 --> 保存到 alert_rule 表

告警评估（可定时或事件触发）:
  读取已启用的 alert_rule
       --> 根据 device_id/device_types 确定目标设备
       --> 从 InfluxDB/Redis 或采集服务获取当前 metric 值
       --> 判断是否满足 rule_condition
       --> 满足：写入 alert_history（rule_id, device_id, severity, status=firing）
       --> 可选：执行自动修复（auto_fix_command）；通知记录表当前不写入
       --> 恢复后更新 alert_history.status=resolved, end_time
```

### 6. Web SSH 流程

```
用户选择设备 --> 前端请求建立 WebSocket 连接
       --> 后端 /api/ws/ssh?deviceId=xxx
       --> 根据 deviceId 查 device 表取 ssh_user、ssh_password、ip、ssh_port
       --> 后端与设备建立 SSH 连接，转发终端 I/O
       --> 可选：记录 webssh_session（user_id, device_id, start_time, input_log）
       --> 用户断开 --> 记录 end_time
```

### 7. 配置备份流程

```
创建备份 --> 读取当前设备列表、告警规则、系统配置等
          --> 序列化为 JSON 存入 config_backup 表（name, backup_type, content, user_id）
          --> 支持恢复：解析 content 写回 device/alert_rule/system_config 等
```

### 8. AI 运维助手流程

```
用户发起会话 --> POST /api/ai/chat (sessionId, message)
       --> 后端取 ai_chat_session、ai_chat_message 历史
       --> 调用千问/DeepSeek API（system_config 中配置 key）
       --> 将 user/assistant 消息写入 ai_chat_message
       --> 返回 AI 回复给前端
```

## 二、数据库 ER 图 / 关系图（文字描述，便于画图）

### 1. 实体列表与主键

| 表名 | 说明 | 主键 |
|------|------|------|
| sys_user | 系统用户 | id |
| role | 角色 | id |
| user_role | 用户-角色关联 | (user_id, role_id) |
| sys_permission | 权限 | id |
| role_permission | 角色-权限关联 | (role_id, permission_id) |
| device | 设备 | id |
| monitor_item | 监控项（按设备） | id |
| alert_rule | 告警规则 | id |
| alert_template | 告警模板 | id |
| alert_history | 告警历史 | id |
| notification_log | 通知记录（仅查询） | id |
| config_backup | 配置备份 | id |
| system_config | 系统配置键值 | id |
| audit_log | 操作审计日志 | id |
| webssh_session | Web SSH 会话 | id |
| ai_chat_session | AI 会话 | id |
| ai_chat_message | AI 消息 | id |

### 2. 关系（用于画连线）

- **sys_user —— user_role —— role**  
  - sys_user.id = user_role.user_id（一对多）  
  - role.id = user_role.role_id（一对多）  
  - 关系：用户与角色多对多

- **role —— role_permission —— sys_permission**  
  - role.id = role_permission.role_id  
  - sys_permission.id = role_permission.permission_id  
  - 关系：角色与权限多对多

- **device**（独立，无外键指向其他业务表）  
  - 被引用：alert_rule.device_id、monitor_item.device_id、alert_history.device_id、webssh_session.device_id

- **alert_rule**  
  - device_id -> device.id（多对一，可选）  
  - monitor_item_id -> monitor_item.id（多对一，可选）  
  - 无外键：device_types、metric_key、rule_condition、severity、enabled、auto_fix_*

- **monitor_item**  
  - device_id -> device.id（多对一）

- **alert_history**  
  - rule_id -> alert_rule.id（多对一）  
  - device_id -> device.id（多对一）  
  - monitor_item_id -> monitor_item.id（多对一，可选）

- **notification_log**（当前仅支持查询，不写入）  
  - alert_id -> alert_history.id（多对一）

- **config_backup**  
  - user_id -> sys_user.id（多对一，可选）

- **audit_log**  
  - user_id -> sys_user.id（多对一，可选），冗余 username、ip

- **webssh_session**  
  - user_id -> sys_user.id（多对一）  
  - device_id -> device.id（多对一）

- **ai_chat_session**  
  - 无外键，仅 username（字符串）

- **ai_chat_message**  
  - session_id -> ai_chat_session.id（多对一）

- **system_config**  
  - 独立，config_key 唯一

- **alert_template**  
  - 独立，用于快速创建告警规则的模板

### 3. 画图时可用的“方框内字段”（核心表）

**device**  
id, name, ip, type, vendor, model, ssh_port, ssh_user, ssh_password, remark, group_name, status, last_poll_time, create_time, update_time, deleted（表中另有 snmp 相关字段预留，当前未使用）

**sys_user**  
id, username, password, real_name, email, enabled, create_time, update_time

**role**  
id, code, name, description, create_time, update_time

**alert_rule**  
id, name, device_id, device_types, monitor_item_id, metric_key, rule_condition, severity, enabled, auto_fix_enabled, auto_fix_type, auto_fix_command, create_time, update_time

**alert_history**  
id, rule_id, device_id, monitor_item_id, metric_key, trigger_value, start_time, end_time, status, severity, message, create_time, update_time

**monitor_item**  
id, device_id, name, metric_key, oid, collector_type, data_type, unit, interval, enabled, create_time, update_time

**ai_chat_session**  
id, username, title, create_time

**ai_chat_message**  
id, session_id, role, content, create_time

**config_backup**  
id, name, backup_type, summary, content, user_id, create_time

**audit_log**  
id, user_id, username, action, target_type, target_id, detail, ip, create_time

**webssh_session**  
id, user_id, device_id, session_id, start_time, end_time, input_log, create_time, update_time

**system_config**  
id, config_key, config_value, remark, create_time, update_time

**notification_log**（当前仅查询，不写入）  
id, alert_id, channel, recipient, content, status, error_msg, create_time

**alert_template**  
id, name, metric_key, rule_condition, severity, device_types, create_time, update_time

**user_role**  
user_id, role_id（联合主键）

**role_permission**  
role_id, permission_id（联合主键）

### 4. 简图示意（可画成框图）

```
[ sys_user ]<----(user_role)---->[ role ]<----(role_permission)---->[ sys_permission ]
     |
     |(config_backup.user_id, audit_log.user_id, webssh_session.user_id)
     v
[ device ]<----(monitor_item.device_id)
     ^
     |(alert_rule.device_id, alert_history.device_id, webssh_session.device_id)
     |
[ alert_rule ]----(alert_history.rule_id)---->[ alert_history ]----(notification_log.alert_id)---->[ notification_log ]
     |
     |(alert_rule.monitor_item_id)
     v
[ monitor_item ]

[ ai_chat_session ]<----(ai_chat_message.session_id)----[ ai_chat_message ]

[ system_config ]   [ config_backup ]   [ alert_template ]  （独立或与上述表单线连接）
```

## 三、论文提纲（按功能与数据库写万字答辩用）

1. **绪论**  
   - 研究背景与意义（网络规模扩大、运维自动化、监控与可观测性）  
   - 国内外研究/应用现状（Zabbix、Prometheus、Grafana、商业运维平台）  
   - 论文主要工作与结构安排  

2. **需求分析**  
   - 业务需求：设备统一管理、状态监控、实时指标展示、告警与通知、远程运维（Web SSH）、配置备份、审计与权限  
   - 功能需求：设备管理、健康状态、实时指标（Linux + 网络设备）、告警规则与历史、Web SSH、批量命令、AI 助手、用户与角色、系统配置  
   - 非功能需求：可用性、可扩展性、数据持久化（MySQL + InfluxDB + Redis）  

3. **系统总体设计**  
   - 架构：B/S、前后端分离、后端 Spring Boot、前端 Vue3、MySQL + InfluxDB + Redis  
   - 功能模块划分：认证与权限、设备管理、数据采集与指标、告警、Web SSH 与批量命令、配置备份、AI 助手、审计与导出  
   - 系统逻辑流程（可引用上文“一、系统逻辑流程图”）  

4. **数据库设计**  
   - 设计原则：规范化、可扩展、与 JPA 实体一致  
   - E-R 设计（可引用上文“二、数据库 ER 图”）  
   - 主要表结构说明：device、sys_user、role、user_role、alert_rule、alert_history、monitor_item、config_backup、audit_log、webssh_session、ai_chat_session、ai_chat_message、system_config、notification_log、alert_template、sys_permission、role_permission  
   - 索引与关键字段说明  

5. **功能详细设计与实现**  
   - 认证与权限：登录、角色权限、前端路由与菜单  
   - 设备管理：CRUD、批量导入、健康汇总、Ping 与状态更新  
   - 实时指标：Telegraf + InfluxDB（Linux）、Redis（网络设备按 IP）、Pipeline 优化  
   - 告警：规则配置、条件评估、历史记录、通知与自动修复（可选）  
   - Web SSH：WebSocket 代理、设备凭证、会话记录  
   - 配置备份与恢复、系统配置、审计日志、AI 助手对接  

6. **系统测试与运行**  
   - 测试环境、功能测试要点、接口与前端联调  
   - 部署说明：MySQL、Redis、InfluxDB、后端、前端  

7. **总结与展望**  
   - 完成的工作总结  
   - 不足与后续改进（如告警通知渠道扩展、更多采集方式、高可用与性能优化）
