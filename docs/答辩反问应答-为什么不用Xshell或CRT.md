# 答辩反问应答：为什么不用 Xshell / CRT，而要开发本系统？

**老师可能问**：既然你的设备都可以走 Telnet 和 SSH，为什么还要开发这套系统？直接用现成的商用 Xshell、SecureCRT 不是更好吗？

---

## 一、一句话核心回答

**Xshell、CRT 是“终端软件”，只解决“连上一台设备、敲命令”；本系统是“监控运维平台”，解决的是“有多少设备、谁在线、指标多少、谁在什么时候动了什么、告警谁收、权限谁管、配置怎么备份”等一整套运维闭环。** 终端只是其中一环，不能替代整个平台。

---

## 二、分点对比（便于口头表述）

| 能力 | Xshell / SecureCRT | 本系统（NetPulse） |
|------|--------------------|---------------------|
| **定位** | 单机终端客户端，连上一台设备操作 | B/S 监控运维平台：设备管理 + 状态 + 指标 + 告警 + 终端 + 权限 + 审计 + 备份 |
| **设备一览** | 无统一列表，需自己记 IP/账号 | 设备表：分组、类型、状态、最后轮询时间，按分组筛选、分页 |
| **在线状态** | 不知道设备是否在线，要自己连了才知道 | 定时可达性检测（Ping + TCP 161/22/23），列表直接显示在线/离线/告警 |
| **CPU/内存等指标** | 要自己登录后敲 display cpu-usage 等命令再看 | SNMP/SSH 自动采集，写入 Redis/InfluxDB，实时指标页统一展示，告警规则可基于阈值触发 |
| **告警** | 无 | 规则引擎：设备上下线、CPU>80 等可触发告警历史、邮件通知、可选自动修复 |
| **多人与权限** | 每人本地装软件、各自保存会话，难以统一管控 | 用户/角色/菜单权限，谁能看到哪些菜单、谁能改设备密码，可审计 |
| **操作留痕** | 无 | 谁在什么时候对哪台设备/哪条规则做了增删改，写审计日志，可查 |
| **配置备份与恢复** | 无 | 设备列表、告警规则、系统配置一键备份/恢复 |
| **使用方式** | 需安装客户端，会话分散在各自电脑 | 浏览器打开即可，Web 终端 + 仪表盘 + 设备列表 + 告警同一套系统，跨地域、多角色协作方便 |
| **批量命令** | 需自己写脚本或买带批量功能的版本 | 对多台设备选命令模板，系统依次 SSH/Telnet 执行并汇总结果 |

所以：**不是“有 SSH 就不需要系统”，而是“有 SSH 只是接入手段之一”；系统把设备、状态、指标、告警、权限、审计、备份和终端整合在一起，这是 Xshell/CRT 做不到的。**

---

## 三、可说的几句话（答辩时直接用的表述）

1. **定位不同**  
   “Xshell、CRT 是终端软件，解决的是‘怎么连上一台设备、敲命令’；我们做的是监控运维平台，解决的是‘有多少设备、哪些在线、指标多少、谁在什么时候做了什么、告警谁处理、权限怎么管’。终端只是平台里的一个功能，不能代替整套平台。”

2. **没有统一视图和自动化**  
   “用 Xshell/CRT 的话，设备列表、在线状态、CPU 内存指标都要人工登录一台台看，没有统一仪表盘，也没有根据阈值自动告警、发邮件。我们系统是定时采集状态和指标，自动判断在线/离线、超阈值就告警，并可选发邮件或自动修复。”

3. **多人与合规**  
   “商用终端多是单人本地使用，会话和账号分散；我们系统是 B/S 的，多角色登录后按权限看不同菜单，谁对设备/规则做了操作都有审计日志，适合团队协作和事后追溯，这是单机终端做不到的。”

4. **总结一句**  
   “所以不是‘有 Telnet/SSH 就不用做系统’，而是‘有了 SSH/Telnet 作为接入方式，还需要一个平台把设备管理、监控、告警、权限、审计、备份和 Web 终端整合起来’；本课题做的就是这样一个一体化平台，终端只是其中一部分。”

---

## 四、若老师追问“那和 Zabbix、SolarWinds 比呢？”

可答：  
“Zabbix、SolarWinds 等偏通用监控，我们重点做的是**一体化**：在同一个系统里既有设备管理、状态与指标、告警，又有 Web 终端、配置备份、角色菜单权限和操作审计，并对华为、华三、锐捷等国内常见厂商做了命令与解析适配；同时基于 Spring Boot 和 Vue 实现，便于二次开发和集成到现有环境，成本和可控性上更适合作为课题和中小规模运维场景的参考实现。”
