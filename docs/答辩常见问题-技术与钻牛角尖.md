# 毕业答辩常见问题：技术问题与钻牛角尖类

本文档整理答辩时老师可能问到的**技术问题**和**钻牛角尖/细节追问**，并给出简要应答思路，便于提前准备。回答时结合本系统实际实现即可。

---

# 一、技术类问题

## 1. 架构与框架

**Q1：为什么用 Spring Boot 而不是纯 Spring 或别的框架？**  
- 答：Spring Boot 提供自动配置、内嵌 Tomcat、起步依赖，能快速集成 JPA、Redis、WebSocket、定时任务等，减少 XML 和样板代码；本系统用到了 @Scheduled 定时采集、@ConditionalOnProperty 按配置开关模块、WebSocket 与 REST 共用一个应用，这些在 Spring Boot 里开箱即用，开发效率高，也便于部署成单个 JAR。

**Q2：前后端为什么分离？有什么好处？**  
- 答：前端 Vue 和后端 Spring Boot 独立开发、独立部署；前端可以单独用 Nginx 托管静态资源，后端只提供 REST 和 WebSocket，职责清晰。多角色、多页面时前端按菜单和路由渲染，后端按接口做权限控制，便于协作和后续扩展（比如换前端框架或做移动端只改前端）。

**Q3：为什么用 Vue 3 而不是 React 或 Angular？**  
- 答：Vue 3 学习曲线平缓、文档友好，和 Vite 搭配构建快；本课题重点是监控运维业务与后端设计，前端选型以够用、易维护为主，Vue 3 能满足设备列表、仪表盘、Web 终端（xterm）、图表（ECharts）等需求。

---

## 2. 数据库与存储

**Q4：为什么同时用 MySQL、Redis、InfluxDB？不能只用一个数据库吗？**  
- 答：三者用途不同。MySQL 存业务关系数据（设备、用户、告警规则与历史、审计等），适合事务和复杂查询；Redis 存设备指标缓存和 SNMP 结果，读写频繁、要快，减轻 MySQL 压力；InfluxDB 是时序库，存设备状态、RTT、Telegraf 写入的 CPU/内存等，按时间范围聚合、做时间线合适。只用一个库要么关系库扛不住高写、要么时序查询不方便，所以按数据特性分工。

**Q5：为什么用 Flyway 做数据库迁移？**  
- 答：表结构变更（加列、删表、建 role_menu 等）用版本化的 SQL 脚本管理，可重复执行、可追溯；和 JPA 实体保持一致，多人协作或换环境时不会漏改库结构，也便于论文里写“数据库设计与版本管理”。

**Q6：逻辑删除和物理删除有什么区别？你们为什么用逻辑删除？**  
- 答：物理删除是直接从表里 DELETE，数据没了，外键引用会出问题；逻辑删除是给记录打标记（如 device.deleted=true），查询时过滤掉，数据还在、历史关联可保留。设备删除后，告警历史、审计里可能还引用 device_id，用逻辑删除避免外键和统计错误。

## 3. 采集与协议

**Q7：SNMP 和 SSH 采集分别用在什么场景？为什么还要“回退”？**  
- 答：网络设备（路由器、交换机、防火墙）一般支持 SNMP，系统优先用 SNMP 取 CPU、内存等 OID，效率高、不占登录会话；但有些设备没配 SNMP 或 SNMP 失败（防火墙、老旧设备），就回退到 SSH/Telnet 登录设备执行 display cpu-usage 等命令并解析输出。这样“能 SNMP 就用 SNMP，不行再用 SSH”，兼顾覆盖率和性能。

**Q8：设备状态为什么先 Ping 再试 TCP 端口？**  
- 答：很多环境禁 Ping 或 Windows 上 ICMP 不可靠，单靠 Ping 会把在线设备判成离线。所以先 Ping，失败后对非 server 设备再依次试 TCP 161（SNMP）、22（SSH）、23（Telnet），任一连通就判在线，这样列表和实际可达性一致。

**Q9：Telegraf 是什么？和你们系统什么关系？**  
- 答：Telegraf 是部署在 Linux 服务器上的采集 agent，把本机 CPU、内存等指标写到 InfluxDB；本系统不直接采服务器指标，而是从 InfluxDB 读 Telegraf 写入的数据，和网络设备（SNMP/SSH 采）的指标一起在“实时指标”页展示，分工是“服务器用 Telegraf + InfluxDB，网络设备用本系统 SNMP/SSH”。

## 4. 安全与权限

**Q10：用户密码怎么存的？有没有加密？**  
- 答：密码存在 sys_user 表；若当前实现是明文或简单存储，可如实说“目前是明文/可逆存储，后续可改为 BCrypt 等不可逆摘要”；若已用 BCrypt，就说“采用 BCrypt 加密存储，不存明文”。

**Q11：权限是怎么控制的？**  
- 答：用户和角色多对多（user_role），角色和可分配菜单多对多（role_menu）；登录后后端返回该用户可见的菜单编码列表，前端按菜单渲染侧栏和路由；设备密码等敏感字段只对管理员角色展示和可编辑；关键接口后端校验角色或菜单权限，和前端一致。

**Q12：Web 终端连设备，密码会不会在前端或网络里泄露？**  
- 答：密码存在后端数据库，建立 SSH/Telnet 会话时在后端用设备表里的凭据连接，前端只通过 WebSocket 传用户按键和接收设备输出，不直接拿到密码；传输层面可补充“生产环境应使用 HTTPS/WSS 加密”。

## 5. 性能与可靠性

**Q13：设备很多时，定时采集会不会把系统或网络打满？**  
- 答：采集间隔可配置（如 5 分钟），SNMP/SSH 采集是定时任务按设备逐个执行，不是同时并发上百个；单机采集失败会回退或跳过，不会因为一台卡住就堵死；后续可按设备数量做分片、限流或分布式采集扩展。

**Q14：Redis 挂了怎么办？**  
- 答：Redis 主要存设备指标缓存，挂了后实时指标页可能缺一部分网络设备数据，但设备列表、告警、Web 终端等不依赖 Redis；可说明“生产环境可做 Redis 高可用，或降级为只从数据库和 InfluxDB 读”。

**Q15：为什么用内存缓存还要 Redis？**  
- 答：内存缓存是进程内、刚采完的数据立刻可被当前请求读到，延迟最低；Redis 可以跨请求、跨实例共享，且能持久一段时间，接口重启或多实例部署时还能从 Redis 取到最近采到的指标，两者配合用。

---

## 6. Web 终端与网络

**Q16：WebSocket 和 HTTP 有什么区别？为什么终端用 WebSocket？**  
- 答：HTTP 是请求-响应，一次请求一次回应；WebSocket 是长连接，建立后双向实时推送。终端要持续把用户输入发给设备、把设备输出推给浏览器，用 WebSocket 更合适，避免轮询或频繁建连。

**Q17：SSH 的 ServerAliveInterval 是干什么的？**  
- 答：每隔多少秒向设备发一个保活包，告诉设备“会话还在用”；有些设备会在一段时间无操作后断开会话，设了保活可以减少“用户只是停了一会儿就被踢下线”的情况。

# 二、钻牛角尖 / 细节追问

**Q18：你说“SNMP 优先、SSH 回退”，那如果设备既支持 SNMP 又支持 SSH，为什么不全用 SNMP？**  
- 答：是的，能 SNMP 就只用 SNMP，不会再去走 SSH；回退只发生在 SNMP 采集失败或未配置时，这样既保证有数据，又减少对设备的 SSH 连接占用。

**Q19：告警规则里“设备类型”过滤是怎么做的？**  
- 答：规则表有 device_types 字段，存的是逗号分隔的类型（如 router,switch）；评估时取设备的 type 和规则里的 device_types 做匹配，只有类型在范围内的设备才参与这条规则的判断。

**Q20：设备删了（逻辑删除），告警历史里还有 device_id，会不会有问题？**  
- 答：逻辑删除后设备不再出现在列表和采集里，但历史记录保留，便于查“这台设备曾经触发过什么告警”；查询告警历史时通常按规则、时间筛选，如需可关联 device 表并过滤 deleted，不会因为设备已删就报错。

**Q21：审计日志是同步写还是异步写？为什么？**  
- 答：本系统是异步写（@Async）：先在请求线程里取出当前用户、IP 等，再交给异步方法落库。这样不阻塞接口响应，避免用户操作等审计写入；若同步写，每次操作都要等数据库插入完成再返回。

**Q22：为什么有的表没有建外键？**  
- 答：部分表（如 audit_log 的 username、target_type/target_id）是逻辑关联，不建外键便于历史数据保留（例如用户删了审计还在）、也减少锁和级联；核心关系如 alert_history 的 rule_id、device_id 可以建外键，具体以迁移脚本为准，设计上在“引用完整性”和“灵活性”之间做了权衡。

**Q23：批量命令是多线程并发执行还是串行？**  
- 答：一般是串行或限制并发（逐台执行），避免同时向大量设备建 SSH 连接把系统或网络打满；具体实现可看 BatchCommandService，答辩时如实说“当前是逐台执行并汇总结果，后续可加可控并发”。

**Q24：如果同一台设备既被 SNMP 采又被 SSH 采，会不会重复？**  
- 答：不会同时采；策略是“优先 SNMP，失败才回退 SSH”，同一时刻只会用一种方式采一台设备，结果写同一套缓存/Redis，按 IP 或设备 ID 覆盖，不会重复写入两套。

**Q25：InfluxDB 没配的话，系统还能用吗？**  
- 答：能用。InfluxDB 主要用于设备状态时序和 Telegraf 的 Linux 指标；不配的话，设备状态检测仍会更新 MySQL 里的 device 表，仪表盘时间线可能没有或改用其他数据源，核心的设备管理、告警、Web 终端、实时指标（从 Redis/缓存）都不依赖 InfluxDB。

**Q26：你们和 Zabbix 比，差在哪里？**  
- 答：Zabbix 是成熟的开源监控系统，功能多、社区大；本课题重点是“基于 Spring Boot 的一体化设计与实现”，把设备管理、状态、指标、告警、Web 终端、权限、审计、备份做在一个系统里，并对国内厂商做了命令解析适配，体量上不如 Zabbix 大，但架构清晰、便于二次开发和作为毕业设计展示；差距主要在监控深度、模板生态和运维成熟度。

**Q27：为什么用 JSch 而不用别的 SSH 库？**  
- 答：JSch 是 Java 里常用的纯 Java 实现，无需本地 openssh，跨平台方便；Spring 生态里也有 SSHD 等可选，本课题选用 JSch 实现 Web 终端和 SSH 采集，满足需求且资料多、易集成。

**Q28：前端怎么知道用户有哪些菜单权限？**  
- 答：登录接口返回用户信息时带上“可访问的菜单编码列表”（从 role_menu 按用户角色汇总）；前端根据这份列表渲染侧栏和路由，没有编码的菜单不显示、访问路由时也可做校验，保证前后端权限一致。

## 7. 接口与前后端协作

**Q29：REST 接口怎么设计的？有没有统一返回格式？**  
- 答：后端用 Spring MVC 提供 REST，路径一般 /api 下按资源划分（如 /api/devices、/api/alerts）；返回用 ResponseEntity 或统一 DTO，成功时 200 + 业务数据，失败时由 GlobalExceptionHandler 等做统一异常处理、返回错误码和消息，前端 Axios 在 request.js 里统一处理 401/403 等。

**Q30：跨域（CORS）怎么处理的？**  
- 答：前后端分离后浏览器会做同源检查；后端在 WebConfig 里配置 CORS，允许前端所在域名访问接口，或开发时允许 localhost；生产环境可限定具体前端域名。

**Q31：登录状态怎么保持？用 Cookie 还是 Token？**  
- 答：按当前实现如实说；若是 Session + Cookie，就说“登录后服务端建 Session，Cookie 带 sessionId”；若是 JWT 或自定义 Token，就说“登录后返回 Token，前端存 localStorage 或 Cookie，请求头里带 Token，后端校验”。两者都可补充“生产环境注意 Cookie 的 HttpOnly、Secure”。

## 8. 业务与设计细节

**Q32：为什么设备类型要区分 server、router、switch、firewall？**  
- 答：不同类型采集和展示策略不同：server 用 Telegraf/InfluxDB，不做 SNMP 采集；网络设备（router/switch/firewall）才做 SNMP 和 SSH 回退，状态检测时 server 只 Ping、不试 TCP 161/22/23；实时指标页也按“Linux 设备”和“网络设备”分开展示，所以类型要区分。

**Q33：OID 是什么？你们怎么知道用哪个 OID 取 CPU？**  
- 答：OID（对象标识符）是 SNMP 里某一项数据的唯一编号，不同厂商、不同指标 OID 不同；系统里在 SnmpUtils 或配置里写死了华为、华三等常用设备的 CPU/内存 OID（或从文档、厂商 MIB 查），采集时请求这些 OID 得到数值。

**Q34：配置备份恢复时，会不会把正在用的设备表覆盖掉？**  
- 答：恢复一般是“用备份内容写回表”，会覆盖当前数据，所以通常做恢复前要有确认或仅允许管理员操作；备份是快照，恢复是人为触发的，不会自动覆盖。

**Q35：告警“已恢复”是怎么判定的？**  
- 答：定时任务评估规则时，若某条规则下某设备当前指标不再满足条件（如 CPU 从 >80 降到 ≤80），且该规则该设备此前有一条“触发中”的告警历史，就把这条历史更新为“已恢复”，填上 end_time；设备上下线类同理，设备从离线变回在线可记一次“上线”或恢复。

# 二、钻牛角尖 / 细节追问（续）

**Q36：为什么不用 Prometheus + Grafana 做监控？**  
- 答：Prometheus 拉模型、Grafana 做看板，更适合指标采集和可视化；本课题要做的是“设备管理 + 状态 + 指标 + 告警 + Web 终端 + 权限 + 审计 + 备份”的一体化平台，需要业务库、用户角色、审计等，这些不是 Prometheus/Grafana 的主场；可以说明“指标部分可以对接 Prometheus 或把数据推到 Prometheus，但整体平台用 Spring Boot 自建更贴合需求”。

**Q37：多人同时用 Web 终端连同一台设备会怎样？**  
- 答：每个用户各建一条 SSH/Telnet 会话，相当于多窗口同时登录同一台设备，设备侧会看到多个连接；没有做“单设备单会话”限制，若要限制可以后续加“同一设备同一时刻只允许一个 Web 终端”等策略。

**Q38：设备密码在列表里是脱敏显示吗？**  
- 答：按实现说；若只对管理员显示明文、对其他人不显示或打星号，就说“按角色脱敏，非管理员不展示或脱敏”；若全部不展示就说“列表不展示密码，仅编辑时管理员可看可改”。

**Q39：定时任务采集间隔 5 分钟，那 5 分钟内指标是旧的，会不会影响告警及时性？**  
- 答：会有最多一个周期的延迟；告警评估也是定时跑（如 5 分钟），所以理论上最坏情况是“指标刚超阈值后要等下一次采集 + 下一次评估”才告警。可说明“间隔可配置，设备多时可适当拉长、设备少时可缩短，或在关键规则上配合实时采集触发”。

**Q40：Redis 里的指标数据有过期时间吗？会不会一直涨？**  
- 答：若设置了 key 的 TTL（过期时间），到期会自动删，不会无限涨；若没设，就靠新数据覆盖（按 IP 或设备 ID 写同一 key）。答辩时如实说当前实现，可补充“生产上建议设 TTL 或定期清理旧 key”。

**Q41：Telnet 明文传输，为什么还要支持？**  
- 答：部分老旧设备或封闭环境只开放 Telnet、不支持 SSH，不支持就纳不进来；支持 Telnet 是为了“能采到、能连上”，在安全要求高的场景可建议只在内网用或仅对部分设备开 Telnet。

**Q42：分页每页 10 条是谁定的？能改吗？**  
- 答：一般是前端或后端默认值（如 size=10），可做成前端传 page、size 参数，后端做 limit/offset；答辩时说“当前默认 10 条，分页参数可配置或前端传参”。

**Q43：如果告警规则很多、设备很多，定时评估会不会太慢？**  
- 答：当前是单机定时任务遍历规则和设备；规则和设备量很大时可以考虑“按规则分片、多线程评估”或“只评估有变化的设备/指标”，如实说“当前规模下可接受，更大规模可做优化”。

**Q44：AI 助手调用的是哪家模型？要钱吗？**  
- 答：按实际实现说（如 OpenAI 兼容接口、国内大模型 API 等）；要钱就说“调用第三方 API，按用量计费，可配置开关”；可补充“主要展示如何把设备/指标上下文注入对话，模型本身可替换”。

**Q45：用户能自己改自己的密码吗？**  
- 答：按实现说；若有“修改密码”接口且只允许改自己，就说“可以”；若没有就说“当前只有管理员能改用户信息，后续可加自助改密码”。

**Q46：为什么用 Java 不用 Go/Python？**  
- 答：本课题强调“基于 Spring Boot”，Java 生态里 Spring 在企业级应用、集成数据库、缓存与定时任务等方面成熟；Go/Python 也能做，但课题定位是 Spring Boot 实践与监控运维结合，选 Java 更贴合题目，也便于和 JPA、Flyway、大量现成 Starter 配合。

**Q47：后端单机部署，挂了是不是全挂？有高可用吗？**  
- 答：当前是单机部署，挂了服务确实不可用；高可用可以做多实例 + 负载均衡，共享同一套 MySQL/Redis，无状态接口可水平扩展；定时任务若多实例要避免重复执行（可用分布式锁或单独抽成 worker 节点）。可如实说“课题阶段是单机，生产可做多实例和共享存储”。

**Q48：为什么用 MySQL 不用 PostgreSQL？**  
- 答：MySQL 使用面广、和 Spring Boot 默认支持好，本系统用到的特性（事务、索引、JPA）两者都满足；选 MySQL 更多是习惯和环境现成，没有必须用 PostgreSQL 的强需求，换库也可以。

**Q49：设备表里 SSH 密码明文存，不怕被拖库吗？**  
- 答：若当前是明文，就如实说“目前为便于和设备对接是明文存库，确实有风险；生产建议加密存或放独立凭据服务，只在内网、控制 DB 权限”；若已加密就说采用的加密方式。

**Q50：实时指标页“刷新”是前端轮询还是后端推？**  
- 答：一般是前端定时请求接口（如每 10 秒调一次实时指标 API）或用户点刷新时请求，不是 WebSocket 推送；答辩时按实现说，可补充“若要实时可改为 WebSocket 推送或 SSE”。

# 三、总结与应变建议

- **技术问题**：紧扣“本系统是怎么做的”（Spring Boot、MySQL/Redis/InfluxDB 分工、SNMP 回退 SSH、逻辑删除、审计异步、WebSocket 保活等），用一两句话说清原理和原因即可。
- **钻牛角尖**：多数是“为什么这样设计”“有没有考虑某某情况”；如实回答“当前实现是这样”“这里确实可以再加强”（如密码加密、批量并发），不要夸大，必要时说“后续可改进”。
- **不会的**：可以说“这块在课题里没有深入，后续可以查资料再完善”，避免现场编造。
