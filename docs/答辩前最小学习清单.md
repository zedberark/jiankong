# 零基础到答辩：最小学习清单

从**完全没学过 Java / Spring Boot / Vue** 到能**稳住答辩**，只学下面这些就够了。按**优先级**和**时间**排好，先做 1、2，再做 3，最后有时间再看 4。

---

## 一、优先学这些（不写代码，只看文档 + 能说出来）

**目标**：答辩时 80% 的问题都能用“系统做什么、表是什么、流程怎么走”来答。  
**时间**：2～3 天（每天 1～2 小时）  
**材料**：`论文正文与ER及表关系说明.md`、`项目结构说明.md`

| 序号 | 学什么 | 怎么学 | 答辩时能说出的 |
|------|--------|--------|----------------|
| 1.1 | **系统是干什么的** | 看论文 1.1 研究背景与意义、1.3 主要工作 | 一句话：监控运维平台，管设备、状态、指标、告警、Web 终端、权限、审计、备份；和 Xshell 的区别是“整套平台 vs 只是终端” |
| 1.2 | **有哪些功能模块** | 看论文 3.2 功能需求、4.3 功能模块划分、项目结构说明里的“模块与目录对应表” | 能数出来：设备管理、状态监测、指标采集、告警、Web 终端、配置备份、审计、用户权限、系统配置、AI 助手、仪表盘、拓扑、批量命令；每个能说一句“干啥的” |
| 1.3 | **数据库有哪些表、各管什么** | 看论文 4.7 核心表概览、4.9 主要表字段说明、后端与数据库表结构对照.md | device、alert_rule、alert_history、sys_user、role、user_role、role_menu、audit_log、config_backup、system_config、ai_chat_session、ai_chat_message 各存什么、谁和谁有关系 |
| 1.4 | **两个核心流程** | 看论文 5.11 关键业务流程简述、第二部分/第三部分流程图文字 | ① 登录：用户登录 → 查角色和菜单 → 前端按菜单显示、后端按权限校验；② 采集与告警：定时采状态和指标 → 写库/缓存 → 规则评估 → 满足就记告警、可发邮件 |
| 1.5 | **ER / 表关系** | 看论文第二部分 ER、第四部分表关系 | 设备-告警、用户-角色-菜单、告警历史-规则-设备 谁引用谁；能指着论文里的图或表说“这是多对一、这是中间表” |

**检验**：合上文档，自己能口头说一遍“系统干啥、有哪些模块、主要表名和用途、登录流程、采集告警流程”。能说顺就过关。

## 二、技术“一句话”必背（被问到能接话）

**目标**：老师问“Java 是什么”“为什么用 Redis”时，能答出一句不卡壳。  
**时间**：半天（抄到本子上或手机里，背下来）  
**材料**：`答辩求生指南-技术不熟如何过.md` 里的万能话术表 + 下面补充

| 技术 | 背这一句（答辩时用） |
|------|----------------------|
| Java | 后端开发语言，跨平台、企业里用得多，我们用的 Java 21。 |
| Spring Boot | 基于 Spring 的框架，自动配置数据库和 Web，打成一个 JAR 就能跑，我们后端接口、定时任务、WebSocket 都是它做的。 |
| Vue | 前端框架，做页面的，数据变界面自动更新，我们设备列表、仪表盘、终端页都是 Vue 3。 |
| 前后端分离 | 前端 Vue 和后端 Spring Boot 分开开发、分开部署，前端调后端 REST 接口拿数据。 |
| MySQL | 存业务数据：设备、用户、告警规则与历史、审计等。 |
| Redis | 存设备指标缓存，读得快，减轻 MySQL 压力。 |
| InfluxDB | 存时序数据：设备状态、CPU 曲线，适合按时间查。 |
| SNMP | 网络设备用 SNMP 取 CPU/内存，效率高。 |
| SSH / Telnet | SNMP 不行就回退到 SSH 或 Telnet 登录设备执行命令取数。 |
| REST | 前后端通过 HTTP 接口传数据，比如 GET /api/devices 拿设备列表。 |
| WebSocket | 长连接，Web 终端用它在浏览器和后端之间实时传按键和终端输出。 |
| Flyway | 数据库表结构用版本化的 SQL 脚本管理，可重复执行、可追溯。 |
| 逻辑删除 | 不真删记录，只打标记（如 deleted=true），查询时过滤掉，历史数据还在。 |

**检验**：让人随便抽 3 个词（如 Redis、Vue、逻辑删除），你能不看书说出一句。

## 三、常见追问的“标准答法”（看一遍有个印象）

**目标**：老师问“为什么用三个库”“为什么先 Ping 再 TCP”时，能答出要点。  
**时间**：1 天（通读一遍，不必逐字背）  
**材料**：`答辩常见问题-技术与钻牛角尖.md`、`答辩反问应答-为什么不用Xshell或CRT.md`

重点看这些问题的**答法思路**（不用背原句）：

- 为什么用 MySQL + Redis + InfluxDB？（分工：业务 / 缓存 / 时序）
- 为什么先 Ping 再试 TCP 161/22/23？（禁 Ping 时还能判在线）
- SNMP 和 SSH 分别干啥、为啥要“回退”？（能 SNMP 就用 SNMP，不行再用 SSH）
- 为什么用逻辑删除？（保留历史、避免外键问题）
- 和 Xshell/CRT 比有什么不同？（他们是终端，我们是整套平台）
- 权限怎么控制？（用户-角色-菜单，登录返回菜单列表，前后端按权限展示和校验）
- 审计是同步还是异步？（异步写，不阻塞请求）

**检验**：看到问题能想到“从哪几个点答”，能说出一两点即可。

## 四、可选：能“对上号”的英文名（被问到不懵）

**目标**：老师提到类名、表名、技术名时，你知道是干啥的。  
**时间**：扫一眼即可，不用背  
**材料**：`项目结构说明.md` 里后端目录、核心表名

- **Controller**：接前端请求的（DeviceController = 设备相关接口）
- **Service**：干具体事的（MonitorService = 状态监测，AlertService = 告警，DeviceSnmpCollectService = SNMP 采集）
- **Repository**：查数据库的（DeviceRepository = 查 device 表）
- **entity**：和表一一对应的类（Device、AlertRule、SysUser…）
- **user_role / role_menu**：用户-角色中间表、角色-菜单表

被问到“MonitorService 是干什么的”能答“做状态监测、定时检测设备在不在线”就够。

## 五、不建议在答辩前死磕的

- **不用**系统学 Java 语法、面向对象、集合类。  
- **不用**学 Spring 源码、Bean 生命周期、AOP 原理。  
- **不用**学 Vue 组件化、响应式原理、Vue 源码。  
- **不用**手写或调试代码（除非你自己想跑一遍系统）。

答辩通过的关键是：**能说清“做了什么系统、有哪些模块和表、关键流程、为什么用这些技术”**，不是会写每一行代码。

## 六、按天排个计划（假设还有 5～7 天答辩）

| 天数 | 做啥 |
|------|------|
| 第 1 天 | 通读论文正文前 5 章 + 第二部分 ER/流程；列出“系统干啥、模块、主要表、两个流程”，自己能复述。 |
| 第 2 天 | 看项目结构说明、后端与数据库表结构对照；把表名和用途对上号；再过一遍 ER 和表关系。 |
| 第 3 天 | 背“二、技术一句话”表；看答辩求生指南里的万能话术和挡箭牌。 |
| 第 4 天 | 通读答辩常见问题、为什么不用 Xshell/CRT；每类问题知道“从哪答”。 |
| 第 5 天 | 自己模拟问自己：系统干啥？有哪些表？登录流程？为什么用 Redis？和 Xshell 有啥区别？答不顺的回去看文档。 |
| 第 6～7 天 | 再过一遍论文目录和 1.1 / 1.3 / 4.7 / 5.11；休息好，别熬夜。 |

---

**总结**：从零到答辩，你要学的就是——**① 系统做什么、模块、表、流程（看论文和文档）② 技术名词各一句话（背）③ 常见追问的答法（看一遍有印象）**。不要求会写代码，把“说清楚”练熟就够了。
