# 系统功能架构图说明

以下用**文字 + 分层/框图描述**给出系统功能架构，便于在 draw.io、Visio、PPT 或论文中绘制**系统架构图**与**功能模块图**。

---

## 一、总体架构图（分层）

系统采用**前后端分离的 B/S 架构**，自顶向下分为：**用户层 → 表现层（前端）→ 业务层（后端）→ 数据层 / 外部系统**。

```

│  用户层：浏览器（管理员 / 运维人员 / 只读用户）                              │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  表现层（前端）                                                           │

│  · 登录 / 仪表盘 / 设备列表 / 拓扑 / 实时指标 / Web 终端 / 批量命令        │
```

### 1.2 绘图建议（总体架构）

- **从上到下**画四层：用户层、表现层（前端）、业务层（后端）、数据/外部层。
- **表现层**框内写：Vue 3、主要页面（登录、仪表盘、设备列表、实时指标、Web 终端、告警、备份、审计、用户、系统设置、AI 助手）。
- **业务层**框内写：Spring Boot、Controller-Service-Repository、主要功能模块名；可单独画小框表示“定时任务”。
- **数据层**画三个框：MySQL、Redis、InfluxDB，并注明用途（业务数据 / 缓存 / 时序）。
- **外部系统**画框：网络设备（SNMP/SSH/Telnet）、邮件服务器、大模型 API、Telegraf（可选）。
- 用**箭头**表示：用户 → 前端、前端 → 后端（HTTP/WebSocket）、后端 → MySQL/Redis/InfluxDB、后端 → 设备/邮件/API。

### 2.2 功能模块清单（用于图例或标注）

| 模块         | 主要功能简述 |
|--------------|--------------|
| 设备管理     | 设备 CRUD、分组筛选、分页、Ping、状态回写、SSH/SNMP 配置 |
| 状态监测     | 定时可达性检测（Ping + TCP 161/22/23）、更新状态与最后轮询时间、状态变化触发告警 |
| 指标采集     | SNMP 采集网络设备 CPU/内存，失败回退 SSH/Telnet；Linux 指标来自 Telegraf/InfluxDB；结果写 Redis/缓存 |
| 告警管理     | 告警规则配置、阈值与上下线评估、告警历史、邮件通知、可选自动修复 |
| Web 终端     | 浏览器内 SSH/Telnet 连接设备、保活、输入输出双向转发、可选解析写指标缓存 |
| 配置备份     | 设备/告警规则/系统配置备份为 JSON，支持恢复 |
| 操作审计     | 关键操作记录（谁、何时、对何对象、何操作、IP），支持按条件查询 |
| 用户与权限   | 用户、角色、用户-角色、角色-菜单；登录返回菜单列表，前后端按权限展示与校验 |
| 系统配置     | 键值对存储系统名称、时区、邮件、API 密钥等 |
| AI 助手      | 会话与消息持久化，调用大模型时注入设备与指标上下文 |
| 仪表盘       | 设备健康汇总、在线/离线时间线、设备类型分布、告警摘要 |
| 设备拓扑     | 设备节点与关系图形化展示，按类型与状态渲染 |
| 批量命令     | 多设备按模板执行 SSH/Telnet 命令并汇总结果 |
| 网络 AI 命令 | 结合大模型与厂商命令模板推荐或生成命令，经 SSH/Telnet 执行 |

### 2.3 绘图建议（功能架构）

- **中心**：写“NetPulse 监控运维系统”或“系统功能架构”。
- **周围**：用矩形框画出上述各模块，用短线或箭头连到中心（或分层排列）。
- **每个框内**：写模块名 + 一行关键词（如“设备管理：CRUD、Ping、分组分页”）。
- 可区分**核心业务**（设备、监测、采集、告警、Web 终端）与**支撑功能**（备份、审计、用户权限、配置、AI、仪表盘、拓扑、批量命令）。

## 四、可直接绘图的要点小结

1. **总体架构图**：四层——用户 → 前端(Vue) → 后端(Spring Boot) → 数据(MySQL/Redis/InfluxDB) + 外部(设备/邮件/API)；标出 HTTP、WebSocket。
2. **功能架构图**：中心“系统”，周围 14 个功能模块框，每框模块名 + 一句话功能。
3. **数据流**：前端 ↔ 后端 ↔ MySQL/Redis/InfluxDB ↔ 设备/邮件/Telegraf，按需画箭头并标注“REST”“WebSocket”“写缓存”“读时序”等。
