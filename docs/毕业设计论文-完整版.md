# 基于 NetPulse 的网络监控与运维平台设计与实现

## 摘要

随着企业信息化与网络规模的不断扩大，数据中心与园区网络中服务器、交换机、路由器、防火墙等设备数量众多，运维人员需要实时掌握设备在线状态、性能指标与告警信息，并在故障发生时快速定位与处置。传统依赖人工登录设备执行命令的方式效率低、可追溯性差。本文设计并实现了基于 **Spring Boot 3** 与 **Vue 3** 的网络监控与运维平台 **NetPulse**，采用 B/S 架构与前后端分离，提供设备统一管理、健康状态监控、实时指标展示（Linux 与网络设备双数据源）、告警规则与历史、Web SSH 远程终端、批量命令执行、配置备份与恢复、用户权限与操作审计、以及 AI 运维助手等核心功能。系统使用 MySQL 存储业务数据、InfluxDB 存储时序指标、Redis 缓存网络设备实时数据，通过 RESTful 接口与 WebSocket 与前端通信，满足中小规模网络环境的集中监控与日常运维需求，具有较好的实用性与可扩展性。

**关键词**：网络监控；运维平台；Spring Boot；Vue；MySQL；InfluxDB；Redis；告警；Web SSH

## 1 前言（系统概述）

### 1.1 研究背景与意义

随着企业信息化与网络规模的不断扩大，数据中心与园区网络中服务器、交换机、路由器、防火墙等设备数量众多，运维人员需要实时掌握设备在线状态、性能指标与告警信息，并在故障发生时快速定位与处置。传统依赖人工登录设备执行命令、查看脚本输出的方式效率低、可追溯性差，且难以在设备数量增长时保持一致性。因此，构建一套集中式的网络监控与运维平台，实现设备发现、状态采集、指标可视化、告警管理与远程运维的自动化，具有重要的实际应用价值。

### 1.2 系统概述

本系统 **NetPulse** 是一套基于 B/S 架构的网络监控与运维平台，面向虚拟机 Linux 与网络设备（路由器、交换机、防火墙等），提供以下核心功能：

- **设备统一管理**：设备增删改查、批量导入、分组筛选、健康汇总、Ping 检测、设备拓扑图展示；
- **健康状态监控**：定时或手动可达性检测，更新设备在线/离线/告警状态，并写入时序库；
- **实时指标展示**：Linux 设备指标来自 InfluxDB/Telegraf 或 SSH 采集，网络设备来自 SNMP/Redis，双数据源统一展示；
- **告警规则与历史**：按设备或指标配置告警规则，记录告警历史，支持邮件通知与可选自动修复；
- **Web SSH 远程终端**：浏览器内 SSH/Telnet 终端，后端 WebSocket 代理至设备；
- **批量命令执行**：多设备 SSH/Telnet 命令下发与结果汇总，支持 AI 推荐与优化；
- **配置备份与恢复**：设备、告警规则、系统配置的 JSON 备份与恢复；
- **用户权限与操作审计**：角色与菜单权限控制、关键操作审计日志与导出；
- **AI 运维助手**：多会话、多轮对话，对接大模型 API 提供智能辅助。

系统采用前后端分离技术栈：后端基于 **Spring Boot 3**，使用 **MySQL** 存储业务数据、**InfluxDB** 存储时序指标、**Redis** 缓存网络设备实时数据；前端基于 **Vue 3** 与 **Vite**，通过 RESTful 接口与 WebSocket 与后端通信。平台可用于中小规模网络环境的集中监控与日常运维，便于二次开发与集成现有监控数据源（如 Telegraf、外部脚本写入 Redis）。

### 1.3 国内外相关现状

在网络监控与可观测性领域，国外已有 **Zabbix**、**Prometheus**、**Grafana**、**Nagios** 等成熟方案；国内不少企业基于上述系统进行二次开发或自研监控平台。在远程运维方面，Web SSH、堡垒机等产品已广泛应用。本系统在设计上借鉴“设备—告警规则—历史记录”的数据模型，并针对 Linux 主机（Telegraf + InfluxDB）与网络设备（SNMP 采集结果写 Redis）双数据源做了统一展示与权限集成，在保证功能完整性的同时便于与现有运维流程结合。

## 2 系统关键技术

### 2.1 后端技术选型

#### 2.1.1 Spring Boot

本系统后端基于 **Spring Boot 3** 构建。选型理由：（1）与项目需求高度匹配——需大量 RESTful 接口、定时任务（设备健康检查、告警评估、SNMP/SSH 采集）、长连接（Web SSH 的 WebSocket 代理）及多数据源访问（MySQL、Redis、InfluxDB），Spring Boot 在上述方面均有成熟方案；（2）内嵌 Tomcat，打包为 jar 后可直接运行，部署简单；（3）依赖注入与分层架构便于 Controller、Service、Repository 分离，可维护性好；（4）文档与社区完善，与 MySQL、Redis、InfluxDB 等集成示例丰富。

#### 2.1.2 数据存储与缓存

- **MySQL**：业务主库（库名 nebula_ops），存储用户（sys_user）、角色（role）、用户角色关联（user_role）、设备（device）、告警规则（alert_rule）、告警历史（alert_history）、配置备份（config_backup）、审计日志（audit_log）、Web SSH 会话（webssh_session）、AI 会话与消息（ai_chat_session、ai_chat_message）、系统配置（system_config）等，满足事务与复杂查询。
- **InfluxDB**：时序数据库，存储 Telegraf 写入的 CPU、内存等时序数据，以及设备状态变化记录（device_status），适合指标随时间变化的场景；系统按 host 与设备 IP 匹配后供实时指标与趋势图使用。
- **Redis**：缓存网络设备按设备 ID 或 IP 的 CPU、内存等实时数据（如 key：netpulse:snmp:device:{设备ID}），支持 Pipeline 批量读取，适合高并发访问；部分业务缓存也可使用 Redis。

#### 2.1.3 远程与采集相关

- **JSch**：实现 SSH 连接，用于 Web 终端、批量命令、告警自动修复及 Linux 设备主机名/厂商信息获取。
- **Apache Commons Net**：实现 Telnet 连接（端口 23），用于仅支持 Telnet 的网络设备。
- **Snmp4j**：网络设备 SNMP（v2c/v3）采集，结果写 Redis 供实时指标展示；支持多厂商 OID（华为、锐捷、Cisco 等）。
- **Spring WebSocket**：Web SSH 的 WebSocket 代理（如 /api/ws/ssh），实现浏览器与设备终端双向转发。
- **Spring Mail**：告警邮件通知（SMTP 配置在系统设置中）。
- **Spring Scheduling**：定时任务，如设备 Ping（MonitorService，默认 60 秒）、告警规则评估、SNMP 采集、Linux 设备厂商信息填充（DeviceVendorFillService）等。

### 2.2 前端技术选型

- **Vue 3 + Vue Router**：组件化开发，路由与菜单按角色（allowedMenus）控制；路由守卫在未登录时跳转登录页并携带 from 参数便于登录后返回。
- **Vite**：构建与开发服务器，热更新快，与 Vue 3 配合良好；生产构建输出 dist 可部署至 Nginx 等静态服务器。
- **Axios**：HTTP 请求，统一 baseURL=/api、超时与请求头；请求拦截器附带 X-User-Name，响应拦截器对 401 清除本地用户并跳转登录。
- **xterm.js**：浏览器内终端，实现 Web SSH 的输入输出展示；配合 FitAddon 适配容器大小。
- **ECharts**：仪表盘设备状态时间线、设备类型饼图、告警分布饼图、实时指标数据来源与 CPU/内存分布饼图、监控趋势折线图等。

### 2.3 外部依赖与部署

- **Telegraf**：部署在 Linux 服务器上，采集 CPU、内存、磁盘、网络等并写入 InfluxDB；设备管理中的设备名称或 IP 需与 Telegraf 上报的 host 一致方可匹配展示。
- **Nginx**（可选）：托管前端静态资源或反向代理后端 /api 与 WebSocket。
- 构建与运行：后端 Maven + JDK 21，默认端口 8082；前端 Node.js / npm，开发端口 5173，通过 Vite 代理将 /api 转发至后端。

## 3 系统分析

### 3.1 业务需求

系统需支撑以下核心业务场景：

（1）**设备统一纳管**：对服务器、路由器、交换机、防火墙等设备进行登记，记录管理 IP、类型、厂商型号、SSH 凭证（含端口，22 为 SSH、23 为 Telnet）、SNMP 配置（v2c 社区名或 v3 安全信息）、分组等；（2）**状态与指标监控**：定期或按需对设备进行可达性检查（Ping 或 TCP 端口探测），更新在线/离线/告警状态（normal、warning、critical、offline），并展示 CPU、内存等实时指标；（3）**告警管理**：支持按设备或设备类型配置告警规则（含指标类与设备上下线类），记录告警历史，支持邮件通知与可选自动修复（SSH 命令或本地脚本）；（4）**远程运维**：通过 Web SSH/Telnet 登录设备执行命令，支持批量命令下发与结果汇总、AI 推荐命令；（5）**配置与审计**：支持配置备份与恢复、系统参数与 API（如大模型）配置、用户与角色权限、操作审计日志；（6）**智能辅助**：集成大模型 API，提供 AI 运维助手多会话、多轮对话。

### 3.2 功能需求

- **认证与权限**：用户名密码登录（POST /api/auth/login），角色（如 ADMIN、运维、只读）控制菜单与接口访问，前后端权限一致；角色权限模板决定各角色可分配菜单，用户可见菜单为所选角色模板的并集。
- **设备管理**：设备增删改查、批量导入（CSV：名称,IP,类型）、分组筛选、健康汇总（GET /api/devices/health）、Ping（GET /api/devices/{id}/ping）、设备拓扑图、通过 SSH 获取主机名与厂商信息（POST /api/devices/{id}/fetch-hostname）。
- **数据采集与实时指标**：定时/手动 Ping 更新状态并写 InfluxDB；Linux 设备指标来自 Telegraf/InfluxDB 或 SSH 回退采集，网络设备来自 Redis（SNMP/SSH 采集）；前端表格与饼图展示，按数据来源（statsSource）区分，支持刷新与趋势图（InfluxDB 查询）。
- **告警**：告警规则配置（设备/类型、指标、条件、严重程度、启用、邮件、自动修复）、告警历史分页与筛选、批量标记已处理、CSV 导出。
- **Web SSH 与批量命令**：浏览器内 SSH/Telnet 终端（WebSocket /api/ws/ssh）；多设备批量执行命令并汇总结果；网络运维控制台支持 AI 生成 CLI 与终端直连。
- **配置备份与系统配置**：全量/设备/告警/系统配置备份与恢复；系统名称、时区、邮件 SMTP、API 密钥等键值配置。
- **审计与导出**：关键操作写审计日志（audit_log）；告警历史、审计日志等 CSV 导出。
- **AI 助手**：多会话、多轮对话，结合设备与监控上下文调用大模型 API（如千问、DeepSeek）。

### 3.3 非功能需求

- **可用性**：核心接口具备异常处理与超时控制；请求拦截器对 401 统一跳转登录；前端对列表接口做空数据兜底与加载态展示，刷新与筛选采用防抖/节流减轻重复请求。
- **可扩展性**：设备类型、监控指标、告警渠道等通过表结构与配置扩展，避免硬编码。
- **数据持久化**：业务数据落 MySQL；时序落 InfluxDB；网络设备实时数据经 Redis 缓存。

## 4 系统设计

### 4.1 系统架构

系统采用 **B/S 架构**与**前后端分离**。用户通过浏览器访问前端（Vue 3 + Vite），前端通过 HTTP（REST）与 WebSocket 与后端（Spring Boot 3）通信；后端对接 MySQL（nebula_ops）、InfluxDB、Redis。数据分工：MySQL 存储业务数据；InfluxDB 存储 Telegraf 时序及设备状态时序；Redis 存储网络设备按 IP/设备 ID 的实时数据。整体架构便于在不同环境中部署与扩展，可配合系统功能架构图（见 docs/系统功能架构图.md）中的总体架构图绘制。

### 4.2 功能模块划分

- **认证与权限模块**：登录、会话、角色与菜单控制（AuthController、UserService、RoleMenuService、InitialAdminRunner）。
- **设备管理模块**：设备 CRUD、批量导入、分组筛选、健康汇总、Ping、设备拓扑、主机名与厂商获取（DeviceController、DeviceService、MonitorService）。
- **数据采集与指标模块**：状态更新（MonitorService）、InfluxDB/Redis 查询（InfluxDbMetricService、DeviceStatsService、SnmpStatsService）、实时指标聚合与来源标识（MetricsController）。
- **告警模块**：告警规则 CRUD、告警评估、告警历史、自动修复、邮件通知（AlertController、AlertService、AlertEmailService）。
- **Web SSH 与批量命令模块**：WebSocket 代理（SshWebSocketHandler）、SSH/Telnet 连接、批量命令执行与结果汇总（BatchCommandService、BatchCommandController）。
- **配置备份与系统配置模块**：备份创建与恢复（ConfigBackupController）、系统配置与 API 配置键值管理（SystemConfigController、ApiSettingsController）。
- **AI 助手模块**：会话与消息持久化、大模型 API 调用（AiChatController、AiChatService）。
- **审计与导出模块**：审计日志写入与查询（AuditService）、CSV 导出（ExportController）。

### 4.3 系统逻辑流程

- **用户登录与权限**：用户输入账号密码 → 后端校验 sys_user、user_role、role → 返回用户信息与 allowedMenus → 前端存入 localStorage，后续请求携带 X-User-Name，路由与菜单按 canShowMenu 控制。
- **设备管理与健康**：设备增删改查/批量导入 → 写入 device 表；定时/手动健康检查（Ping 或 TCP）→ 更新 status、last_poll_time，状态变化触发告警 → 前端展示列表与健康汇总；仪表盘离线设备按 last_poll_time 降序排序。
- **实时指标**：前端请求 /api/metrics/realtime → 后端从 InfluxDB（Linux）、Redis（网络设备）及 SSH 回退获取数据 → 与 device 表匹配后合并返回 stats、statsSource → 前端展示表格与饼图；支持手动刷新与趋势图查询。
- **告警**：配置规则写入 alert_rule；定时或状态变更时根据规则与当前指标判断 → 触发则写 alert_history（status=firing），可选自动修复；恢复后更新 status=resolved、end_time；支持邮件通知。
- **Web SSH**：用户选择设备 → 前端建立 WebSocket /api/ws/ssh?deviceId=xxx → 后端从 device 取 SSH/Telnet 凭证建立连接 → 双向转发；可选记录 webssh_session。
- **配置备份**：创建备份 → 汇总设备/告警规则/系统配置 → 序列化为 JSON 写入 config_backup；恢复时解析 JSON 写回相应表。
- **AI 助手**：用户发消息 → 加载会话历史 → 调用大模型 API（系统设置中配置）→ 写入 ai_chat_message 并返回。

### 4.4 数据库设计

#### 4.4.1 设计原则

与业务实体一致、适度规范化、便于扩展、关键表保留审计与时间戳（create_time、update_time）。

#### 4.4.2 E-R 与实体关系

核心实体：sys_user、role、user_role、device、alert_rule、alert_history、alert_template、config_backup、system_config、audit_log、webssh_session、ai_chat_session、ai_chat_message、role_permission、sys_permission 等。关系：用户与角色多对多（user_role）；角色与权限多对多（role_permission）；设备与告警规则/告警历史/Web SSH 会话为一对多；告警历史与通知记录一对多；AI 会话与消息一对多；配置备份、审计日志与用户多对一。绘图时实体用矩形、联系用菱形或连线并标注 1、n。详见 docs/全局ER图-含属性.md。

#### 4.4.3 主要表结构说明

- **device**：id、name、ip、type（router/switch/server/firewall/other）、vendor、model、snmp_version、snmp_community、snmp_security、snmp_port、ssh_port、ssh_user、ssh_password、remark、group_name、status（normal/warning/critical/offline）、last_poll_time、create_time、update_time、deleted 等。
- **sys_user**：id、username、password、real_name、email、enabled、create_time、update_time。
- **role**、**user_role**、**role_permission**、**sys_permission**：角色及用户-角色、角色-权限关联与权限定义。
- **alert_rule**：id、name、device_id、device_types、metric_key、condition、severity、enabled、notify_email、auto_fix_enabled、auto_fix_type、auto_fix_command、create_time、update_time 等。
- **alert_history**：id、rule_id、device_id、metric_key、trigger_value、start_time、end_time、status（firing/resolved）、severity、message 等。
- **config_backup**：id、name、backup_type、summary、content（LONGTEXT）、user_id、create_time。
- **audit_log**：id、username、action、target_type、target_id、detail、ip、create_time。
- **webssh_session**：id、user_id、device_id、session_id、start_time、end_time、input_log 等。
- **ai_chat_session**、**ai_chat_message**：会话与消息。
- **system_config**：id、config_key（唯一）、config_value、remark 等。

详细字段与索引参见 docs/后端与数据库表结构对照.md 及项目建表脚本。

## 5 系统实现与效果

### 5.1 认证与权限实现

采用表单登录：前端提交用户名与密码至 POST /api/auth/login，后端在 sys_user 中校验并查询 user_role、role，返回用户信息与角色列表及 allowedMenus；前端将用户信息存入 localStorage，后续请求通过拦截器在 Header 中携带 X-User-Name。前端根据 allowedMenus 控制菜单显示（Layout.vue 中 canShowMenu）；路由守卫在访问受保护页时检查 localStorage 与 menuCode，未登录跳转登录页并带 from 参数。角色与权限通过 RBAC 模型（role、sys_permission、role_permission、角色权限模板）实现，用户管理页支持为角色配置可分配菜单模板。

### 5.2 设备管理实现

设备列表增删改查与批量导入（名称,IP,类型）；列表支持按分组筛选、排序（IP、类型、状态等）、分页；健康汇总接口 GET /api/devices/health 按 device 表 status 分组统计；单设备 Ping 由 GET /api/devices/{id}/ping 实现，返回 rttMs 与 up/down。设备拓扑图页面（DeviceTopology.vue）展示设备节点、按分组与状态筛选，支持拖拽布局与本地存储位置，点击设备可跳转 Web SSH。通过 POST /api/devices/{id}/fetch-hostname 可对 Linux 服务器一次性获取主机名与 os-release 并回填名称与厂商；DeviceVendorFillService 定时对在线且厂商为空的 Linux 设备自动填充厂商。

### 5.3 数据采集与实时指标实现

设备状态通过 MonitorService 定时 Ping（默认 60 秒）或 TCP 端口探测更新，结果写 device 表（updateStatusAndLastPollTime）并写入 InfluxDB device_status；状态变化时触发 AlertService 写入设备上下线告警。实时指标：Linux 设备从 InfluxDB 按 host 查询 Telegraf 数据，无数据时由 DeviceStatsService 通过 SSH 回退采集 CPU/内存/磁盘；网络设备从 Redis 按设备 ID 或 IP 读取（SnmpStatsService 或 SSH/Telnet 缓存）；GET /api/metrics/realtime 合并返回 devices、summary、stats、statsSource、snmpEnabled 等。前端每 10 秒刷新，展示 Linux 设备表与网络设备表、数据来源与 CPU/内存饼图；支持手动刷新（节流）与趋势图（getMetricsTrend，InfluxDB 查询 cpu/mem）。

### 5.4 告警实现

告警规则在 alert_rule 表配置，支持指标类（如 CPU>80%）与设备上下线类；评估由定时任务或设备状态变更触发，满足条件则写 alert_history（status=firing），可选执行自动修复（SSH 命令或本地脚本）；恢复时更新 status=resolved、end_time。告警历史支持分页查询、状态与严重程度筛选、人工标记已处理、批量标记与 CSV 导出。邮件通知通过系统设置中配置的 SMTP 发送；AlertEmailService 在规则触发时按配置发送通知。

### 5.5 Web SSH 与批量命令实现

Web SSH：前端建立 WebSocket（/api/ws/ssh?deviceId=xxx），后端 SshWebSocketHandler 从 device 表取 SSH 或 Telnet 凭证（端口 23 为 Telnet）与设备建立连接，终端输入输出在 WebSocket 与 SSH/Telnet 间双向转发；可选记录 webssh_session。批量命令：用户选择多台设备并输入命令，后端 BatchCommandService 逐台 SSH/Telnet 执行并汇总结果返回；支持 AI 推荐与优化命令。网络运维控制台（NetworkAiCommand.vue）支持按厂商与需求生成 CLI、并嵌入终端直连设备执行。

### 5.6 配置备份与系统配置实现

备份时将设备、告警规则、系统配置等序列化为 JSON 写入 config_backup；恢复时解析 JSON 写回相应表。系统配置存 system_config 键值对，供前端系统设置页与后端 AI 服务使用；API 配置（千问、DeepSeek 等）单独存储，供 AI 助手与网络/批量命令中的 AI 功能调用。

### 5.7 AI 助手与审计导出实现

AI 助手：多会话（ai_chat_session）、多轮消息（ai_chat_message）；发送消息时加载历史并调用大模型 API（系统设置中配置），将回复写入库并返回。审计：关键操作由 AuditService 写入 audit_log（用户名、操作类型、对象类型、对象 ID、详情、IP）；导出接口（ExportController）提供告警历史、审计日志等 CSV 导出。

## 7 总结与展望

### 7.1 总结

本文设计并实现了基于 **Spring Boot 3** 与 **Vue 3** 的网络监控与运维平台 **NetPulse**，完成了设备管理、健康状态监控、实时指标展示（Linux + 网络设备双数据源）、告警规则与历史、Web SSH、批量命令、网络 AI 控制台、配置备份、系统配置、用户权限与角色模板、操作审计与 AI 助手等核心功能。系统采用前后端分离架构，基于 MySQL、InfluxDB、Redis 进行数据持久化与缓存，数据库设计规范、实体关系清晰，并提供了系统逻辑流程与 E-R 关系的描述，便于绘制答辩用流程图与 E-R 图。平台可用于中小规模网络环境的集中监控与日常运维，具有较好的实用性与可扩展性。

### 7.2 不足与展望

当前可改进之处：（1）用户密码建议改为 BCrypt 等不可逆摘要并加强会话管理；（2）告警通知渠道（邮件、钉钉、企业微信等）需进一步对接与配置；（3）InfluxDB 与 Redis 的高可用、备份策略需结合生产环境规划；（4）实时指标与告警评估的并发与限流可根据设备规模优化；（5）自动化测试（单元测试、接口测试）可补充以保障回归。后续可在上述方面增强，并扩展更多采集方式、可视化大屏与报表，以更好地支撑企业级运维场景。
