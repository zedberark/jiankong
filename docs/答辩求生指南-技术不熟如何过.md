# 答辩求生指南：技术不熟如何通过答辩

如果你**没系统学过 Java、Spring Boot、Vue**，但课题/论文用的是这套技术栈，答辩时按下面思路准备，可以**把重点放在“做了什么、为什么做、整体设计”上**，减少被问到“这段代码怎么写”的概率，即使被问到也有话可说。

---

## 一、心态与定位（先稳住自己）

1. **答辩考的是“你懂不懂这个课题”**，不是“你能不能当场写代码”。老师更关心：  
   - 你要解决什么问题？  
   - 系统有哪些功能、怎么划分的？  
   - 数据库/表怎么设计的？  
   - 关键流程（登录、采集、告警、终端）大致怎么走？

2. **合理定位自己的角色**（选一个你能说出口的）：  
   - “我主要负责**需求分析、总体设计、数据库设计和文档**，具体代码是在**参考 Spring Boot / Vue 官方文档和现有项目结构**下实现/完善的。”  
   - 或：“整体架构和模块划分是我做的，**实现上用了 Spring Boot 和 Vue 的现成能力**，部分细节是查资料和调试完成的。”  
   这样既诚实，又不会显得你完全没参与。

3. **不要主动往“底层实现、源码、每一行代码”上引**。老师不问，你就多讲“功能、表、流程、界面”；老师问了，用下面的话术顶。

---

## 二、背熟这几句“万能话术”（被问到技术时用）

被问到 **Java / Spring Boot / Vue 是什么、为什么用** 时，用下面几句撑过去（不用背源码，背意思即可）：

| 被问到 | 你可以说（一句话版） |
|--------|------------------------|
| **Java** | “Java 是后端开发语言，跨平台、生态成熟，企业里用得多，我们后端用 Java 21。” |
| **Spring Boot** | “Spring Boot 是基于 Spring 的框架，能快速搭 Web 应用：自动配置数据库、Web、定时任务，打成一个 JAR 就能跑，我们后端接口、定时采集、WebSocket 都是用它做的。” |
| **Vue** | “Vue 是前端框架，做页面的，数据变了界面自动更新；我们前端设备列表、仪表盘、登录、Web 终端页面都是用 Vue 3 写的。” |
| **前后端分离** | “前端 Vue 和后端 Spring Boot 分开开发、分开部署，前端调后端 REST 接口拿数据，这样前后端可以并行开发，以后换前端或加移动端也方便。” |
| **MySQL / Redis / InfluxDB** | “MySQL 存业务数据比如设备、用户、告警；Redis 存设备指标缓存，读得快；InfluxDB 存时序数据比如设备状态、CPU 曲线，三个各干各的活。” |
| **SNMP / SSH** | “网络设备用 SNMP 取 CPU 内存，效率高；SNMP 不行就用 SSH 登录设备执行命令取数，叫‘回退’，这样没配 SNMP 的设备也能采到数据。” |

**原则**：只说“是什么、干什么用”，**不说**“怎么实现的、哪一行代码”。老师追问“具体怎么实现”，用下一节的“挡箭牌”话术。

---

## 三、被问到“具体怎么实现 / 这段代码什么意思”时

1. **不要瞎编、不要背大段代码。**  
2. **可以这样说**（选一句你顺口的）：  
   - “这一块是参考 Spring Boot 官方文档和项目现有结构做的，**具体实现细节我答辩后会再梳理一遍**。”  
   - “这里用的是 Spring 的 **@Scheduled 定时任务 / @Transactional 事务**（按问题选），具体写法我是按文档来的，**底层原理我还在补**。”  
   - “前端这部分是 Vue 的**数据绑定和组件**，接口用 **Axios** 调后端，具体代码我更多是改参数和逻辑，**框架层面的细节我承认还不够熟**。”

3. **把话题拉回你熟悉的地方**：  
   - “**整体上**这个模块的作用是……（例如：定时去采设备状态，更新数据库，状态变了就触发告警），**流程和表设计**在论文里都有画/写。”

---

## 四、答辩时多讲这些（老师容易满意，你又容易讲）

把这些**提前看熟、能口头说清楚**，答辩时主动往这里引：

1. **系统是干什么的**  
   “一个监控运维平台：管设备、看状态和 CPU 内存、告警、Web 终端登录设备、配置备份、谁操作了谁有权限都有记录。”

2. **有哪些模块**  
   设备管理、状态监测、指标采集（SNMP+SSH）、告警、Web 终端、配置备份、审计、用户权限、系统配置、AI 助手、仪表盘、拓扑、批量命令——**能数出来并各说一句干啥的**即可。

3. **数据库有哪些表、干什么用**  
   device（设备）、alert_rule / alert_history（告警）、sys_user / role / user_role / role_menu（用户权限）、audit_log（审计）、config_backup（备份）……**能说出表名和用途**即可。

4. **两个关键流程（能说清就行）**  
   - **登录与权限**：用户登录 → 查角色和菜单 → 前端按菜单显示、后端按权限校验。  
   - **采集与告警**：定时采设备状态和指标 → 写库/缓存 → 告警规则评估 → 满足条件就记告警、可发邮件。

5. **和 Xshell/CRT 的区别**  
   “Xshell/CRT 只是终端软件；我们是一整套平台，有设备列表、状态、指标、告警、权限、审计、备份，终端只是其中一个功能。”（详见《答辩反问应答-为什么不用Xshell或CRT》）

**建议**：把《论文正文与ER及表关系说明》里的**目录、各章小标题、表名、流程图**过几遍，能用自己的话复述出来，比死磕代码有用得多。

---

## 五、答辩当天可以这样开场（可选）

做 PPT 或自我介绍时，可以主动说一句，定个调：

- “本课题的重点是**监控运维业务的需求分析、系统设计和数据库设计**，实现上采用 **Spring Boot + Vue** 完成前后端开发，**我主要负责整体设计和各模块功能的梳理与文档**，具体编码在参考官方文档和现有工程结构的基础上完成。”  

这样老师心里有数：你强调的是“设计和业务”，不是“底层源码”，追问技术时你也有理由用“参考文档、后续再巩固”来回应。

---

## 六、万一被问倒怎么办

- **不会的别编。** 直接说：“**这块我确实还没吃透，答辩后我会补上。**”  
- **可以承认**：“Java/Spring Boot/Vue 我是做课题时边做边学的，**用得起来但原理还不够深**，后面会系统学一遍。”  
- 老师一般不会因为“某一道技术细节答不上”就挂人，**更怕的是不懂装懂、胡说八道**。诚实 + 态度好 + 能把“做了什么、表怎么设计的、流程怎么走的”说清楚，通过概率会高很多。

---

## 七、答辩前 1 天建议做的事

1. **过一遍论文目录和每章第一段**，能说出每章在讲什么。  
2. **背熟上面“万能话术”表**里的几句话（Java/Spring Boot/Vue/前后端分离/MySQL+Redis+InfluxDB/SNMP+SSH）。  
3. **看一遍《答辩常见问题-技术与钻牛角尖》和《答辩反问应答-为什么不用Xshell或CRT》**，每个问题不用背答案，但看到问题要能想到“大概从哪个方向答”。  
4. **表名和模块名**：device、alert_rule、alert_history、user_role、role_menu、audit_log、ConfigBackup、MonitorService、DeviceSnmpCollectService、AlertService……能对上号是干啥的。  
5. **早点睡**，状态好一点，说话会更稳。

---

**最后**：答辩通过的关键是**让老师觉得“这个学生知道自己在做什么系统、为什么要做、大致怎么做的”**，而不是“每一行代码都会写”。把“做了什么、表结构、流程、和 Xshell 的区别”讲清楚，技术细节用“参考文档、后续巩固”顶住，通过是完全有机会的。加油。
