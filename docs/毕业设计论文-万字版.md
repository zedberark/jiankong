# 基于SpringBoot的运维监控平台设计与实现

## 摘要

随着企业信息化与网络规模的不断发展，不论在业务保障还是在运维决策中，设备状态与性能信息的重要性都是毋庸置疑的。虽然这些信息以数据的形式存在于系统之中，但其价值却是可量化的——只有做到对设备状态与指标更好、更快、更准确的采集与展示，才能支撑运维人员做出正确的判断，保障业务连续性。运维信息通过集中监控平台汇聚与发布，让不同岗位的人员打破空间的界限，实时、同步掌握全网设备健康与告警情况，使团队内部获取运维信息的途径差异大大减少，同时也可以随时随地进行监控查看与远程运维操作。

本论文围绕“基于SpringBoot的运维监控平台设计与实现”这一课题，基于当前企业运维管理中对统一纳管、实时监控与智能告警的需求进行业务开发，根据运维痛点与市场主流方案分析，设计了一款以 Spring Boot 为核心后端的运维监控平台 NetPulse，并根据行业流行的技术进行了框架设计和技术选择。基于 B/S 架构的系统开发与设计，结合监控运维平台的具体业务要求，使用 Java 语言与 Vue 技术，将前端与后端分离，便于代码维护和拓展，同时可以更好地处理业务逻辑；使用 Vue 与 ECharts 实现前端页面展示和用户交互，通过 RESTful 接口与 WebSocket 实现前后端业务逻辑处理，并依托 MySQL、Redis、InfluxDB 等存储与缓存技术完成数据持久化与实时指标展示，从而完成系统的核心功能。

本论文完成了基于SpringBoot的运维监控平台的设计与实现，以 NetPulse 为具体载体，采用 Spring Boot 与 Vue 技术栈，提供了设备统一管理、实时指标监控、告警规则与历史、Web SSH 远程终端、配置备份与恢复、操作审计、用户与权限管理以及 AI 运维助手等功能，满足运维人员对集中监控与高效运维的需求，更好地服务于企业网络与信息系统的管理。

**关键词**：网络监控；Spring Boot；运维平台；Vue；前后端分离

## 1 前言（系统概述）

### 1.1 研究背景与意义

随着企业信息化建设的深入推进和网络规模的持续扩大，数据中心与园区网络中服务器、交换机、路由器、防火墙等设备数量急剧增加，运维人员面临着设备状态难以统一掌握、性能指标分散、故障发现与处置滞后等问题。传统运维方式主要依赖人工逐台登录设备执行命令、查看脚本输出或登录各厂商网管界面，不仅效率低下，而且操作难以追溯，在设备数量增长时无法保持一致性，也无法形成统一的监控视图与告警闭环。

在此背景下，构建一套集中式的网络监控与运维平台，实现设备统一纳管、状态自动采集、指标可视化展示、告警规则配置与通知、以及远程运维操作的自动化，具有重要的实际应用价值。一方面，平台能够将分散的设备状态与指标汇聚到统一界面，便于运维人员快速了解全网健康情况；另一方面，通过告警规则与历史记录，可以在故障发生或隐患出现时及时通知相关人员，并结合 Web SSH、批量命令等功能实现快速处置，从而提升运维效率、降低人为差错、保障业务连续性。本课题以“基于SpringBoot的运维监控平台设计与实现”为选题，以实际运维场景为出发点，设计并实现了一款面向 Linux 主机与网络设备（路由器、交换机、防火墙等）的运维监控平台 NetPulse，为中小规模网络环境提供可用的集中监控与日常运维能力，并对接 Telegraf、SNMP 等常见数据源，便于与现有基础设施集成。

### 1.2 系统概述

本系统 NetPulse 是一套基于 Spring Boot 与 B/S 架构的运维监控平台，面向虚拟机 Linux 与网络设备（路由器、交换机、防火墙等），提供以下核心功能。

**设备统一管理**：支持设备的增删改查、批量导入（CSV 格式：名称,IP,类型）、按分组筛选与多字段排序、分页展示；提供设备健康汇总接口，按正常、告警、离线等状态统计数量；支持单设备 Ping 检测与连通性判断；提供设备拓扑图页面，以节点形式展示设备并支持按分组与状态筛选、拖拽布局与位置持久化，点击设备可跳转至 Web SSH 终端。设备信息包括管理 IP、类型（服务器/路由器/交换机/防火墙/其他）、厂商型号、SSH 或 Telnet 凭证（端口 22 为 SSH、23 为 Telnet）、SNMP 配置（v2c 社区名或 v3 安全信息）、分组等，并支持通过 SSH 一次性获取 Linux 主机名与厂商信息（解析 os-release）进行回填。

**健康状态监控**：通过定时任务（默认 60 秒）或手动触发对设备进行可达性检测（Ping 或 TCP 端口探测），更新设备在线/离线/告警状态（normal、warning、critical、offline），并将最后轮询时间与状态写入数据库及 InfluxDB 时序库；设备状态变化时可触发告警规则，生成设备上下线类告警记录。

**实时指标展示**：Linux 设备指标主要来自 Telegraf 写入 InfluxDB 的 CPU、内存、磁盘等时序数据，当 InfluxDB 无数据时可由系统通过 SSH 执行系统命令进行回退采集；网络设备指标来自 SNMP 或 SSH/Telnet 采集后写入 Redis 的缓存数据。前端以表格形式分别展示 Linux 设备与网络设备，并展示数据来源、CPU/内存分布等饼图，支持手动刷新（节流）与监控趋势图（基于 InfluxDB 的时序查询）。

**告警规则与历史**：支持按设备或设备类型配置告警规则，包括指标类（如 CPU 使用率大于某阈值）与设备上下线类；系统定时或根据状态变更评估规则，满足条件时写入告警历史（状态为 firing），并可配置邮件通知与可选自动修复（SSH 命令或本地脚本）；告警恢复后更新历史记录状态与结束时间。告警历史支持分页、按状态与严重程度筛选、人工或批量标记已处理，以及 CSV 导出。

**Web SSH 远程终端**：在浏览器内提供 SSH 或 Telnet 终端，后端通过 WebSocket 代理与设备建立连接，实现输入输出的双向转发；用户选择设备后即可连接，无需单独安装 SSH 客户端。

**批量命令执行与网络运维控制台**：支持对多台 Linux 设备同时下发 SSH/Telnet 命令并汇总执行结果；网络运维控制台支持按厂商与需求通过 AI 生成 CLI 配置命令，并嵌入终端直连设备执行，便于网络设备配置的生成与落地。

**配置备份与恢复**：支持将设备、告警规则、系统配置等序列化为 JSON 进行全量或分类备份，并可在需要时从备份恢复，便于迁移或灾难恢复。

**用户权限与操作审计**：采用角色与菜单权限控制，不同角色可见菜单不同；支持角色权限模板配置，用户可见菜单为所选角色模板的并集。关键操作（如设备增删改、告警规则变更、用户管理等）写入审计日志，支持按操作人、操作类型等条件查询与 CSV 导出。

**AI 运维助手**：集成大模型 API（如阿里千问、DeepSeek 等），提供多会话、多轮对话的智能辅助，可结合设备与监控上下文回答问题，辅助运维决策。

系统采用前后端分离技术栈：后端基于 Spring Boot 3，使用 MySQL 存储业务数据、InfluxDB 存储时序指标、Redis 缓存网络设备实时数据；前端基于 Vue 3 与 Vite，通过 RESTful 接口与 WebSocket 与后端通信。平台适用于中小规模网络环境的集中监控与日常运维，并便于二次开发与集成现有监控数据源（如 Telegraf、外部脚本写入 Redis）。

### 1.3 国内外相关现状

在网络监控与可观测性领域，国外已有 Zabbix、Prometheus、Grafana、Nagios 等成熟方案。Zabbix 支持多种采集方式与丰富的告警渠道，在企业环境中应用广泛；Prometheus 以拉模型与 PromQL 见长，常与 Grafana 配合做指标可视化；Nagios 以插件化与告警配置灵活著称。国内不少企业基于上述系统进行二次开发或自研监控平台，以适配自身网络架构与运维流程。在远程运维方面，Web SSH、堡垒机等产品已广泛应用，实现了在浏览器内登录设备执行命令并记录操作日志。本系统在设计上借鉴“设备—告警规则—历史记录”的数据模型，并针对 Linux 主机（Telegraf + InfluxDB）与网络设备（SNMP 采集结果写 Redis）双数据源做了统一展示与权限集成，在保证功能完整性的同时便于与现有运维流程结合，适合作为毕业设计或中小规模生产环境的监控运维工具。

## 2 系统关键技术

本章介绍本系统所采用的关键技术，包括后端与前端技术选型、数据存储与缓存、远程与采集相关技术及外部依赖与部署，各节按“简介—特性—在本系统中的应用”的结构展开，便于理解技术背景与选型依据。

### 2.1 Spring Boot + Vue

#### 2.1.1 Spring Boot + Vue 简介

本系统采用 Spring Boot + Vue 的前后端分离技术栈：后端基于 Spring Boot，前端基于 Vue，二者通过 RESTful 接口与 WebSocket 协作，共同构成运维监控平台的核心技术基础。

**Spring Boot** 是由 Pivotal 团队（现 VMware 旗下）在 2013 年起基于 Spring 框架推出的快速开发脚手架，其设计目标是简化 Spring 应用的初始搭建与配置，使开发者能够以最少的配置快速构建生产级别的应用。Spring Boot 采用“约定优于配置”的理念，内嵌 Tomcat、Jetty 等 Web 容器，提供自动配置（Auto-configuration）、起步依赖（Starter）等机制，使 Java Web 应用的开发、打包与部署更加便捷，广泛应用于企业级后端服务与微服务架构。后端业务采用 **Java** 语言实现，Java 由美国 Sun 公司于 1995 年推出、现由 Oracle 维护，是一门面向对象的高级编程语言，具有跨平台、安全、稳定、生态丰富等特点；本系统采用 JDK 21 作为运行与编译环境。

**Vue** 是一套用于构建用户界面的渐进式 JavaScript 框架，由尤雨溪创建并开源。Vue 采用响应式数据绑定与组件化开发，核心库只关注视图层，易于与现有项目或其它库集成；配合 Vue Router 可构建单页应用（SPA）。Vue 3 在性能、TypeScript 支持与 Composition API 等方面有显著改进，与 Spring Boot 后端搭配可实现前后端分离、职责清晰的 B/S 架构，是当前企业级 Web 应用中广泛采用的组合之一。

#### 2.1.2 Spring Boot + Vue 特性

##### 2.1.2.1 Spring Boot 特性

**自动配置与起步依赖**。Spring Boot 根据类路径中的依赖自动配置 Bean 与组件，例如引入 spring-boot-starter-data-jpa 即可集成 JPA 与 Hibernate，引入 spring-boot-starter-web 即内嵌 Tomcat 并启用 MVC，大大减少 XML 或 Java 配置代码。起步依赖将常用库及其兼容版本打包为一组依赖，避免版本冲突，提升开发效率。

**内嵌容器与可执行 jar**。应用可打包为包含依赖的 fat jar，通过 java -jar 直接运行，无需单独安装与配置 Tomcat 等 Web 容器，便于在服务器或容器环境中部署与扩展。服务端口、数据源地址等可通过配置文件或环境变量覆盖，适应多环境需求。

**分层架构与依赖注入**。Spring 的 IoC 与 AOP 便于将控制器（Controller）、业务逻辑（Service）、数据访问（Repository）分离，代码结构清晰、可测试性与可维护性好。本系统在此基础上实现 RESTful 接口、定时任务、WebSocket、多数据源访问等能力。

**生态与文档完善**。Spring 生态对 MySQL、Redis、InfluxDB、WebSocket、邮件、定时任务等均有官方或社区集成方案，文档与示例丰富，便于在运维监控平台中快速集成各类中间件与第三方服务。

##### 2.1.2.2 Java 特性

**面向对象与跨平台**。Java 的封装、继承、多态与接口机制适合构建模块化、可扩展的后端服务；一次编译、到处运行（基于 JVM）的特性便于在不同操作系统与部署环境中保持一致行为。

**并发与多线程**。Java 内置多线程、线程池与并发工具包（java.util.concurrent），适合本系统中定时任务、异步采集、WebSocket 长连接等并发场景。

**安全与稳定性**。强类型、异常机制与垃圾回收等特性有助于减少运行时错误；LTS 版本（如 JDK 21）长期支持与向后兼容，保证生产环境的可靠性与可维护性。

##### 2.1.2.3 Vue 特性

**响应式与组件化**。Vue 的数据与视图双向绑定，数据变更自动驱动视图更新；组件可复用、可组合，便于维护与协作开发。Vue 3 的 Composition API 与 script setup 使逻辑组织更清晰，类型推导更友好。

**单页应用与前后端分离**。Vue Router 负责前端路由与 URL 映射，支持路由守卫与懒加载，可与后端 RESTful 接口配合实现前后端分离。前端专注展示与交互，后端专注业务逻辑与数据访问，二者通过 API 通信，便于并行开发与独立部署。

#### 2.1.3 Spring Boot + Vue 在本系统中的应用

本系统采用 **Spring Boot + Vue** 技术栈，后端基于 Spring Boot 3 与 Java 实现，前端基于 Vue 3 实现，前后端通过 RESTful 接口与 WebSocket 协作。选型理由主要有：第一，与项目需求高度匹配——系统需要大量 RESTful 接口供前端调用、定时任务执行设备健康检查与告警评估、WebSocket 支持 Web SSH、同时访问 MySQL、Redis、InfluxDB 等多数据源，Spring Boot 均有成熟组件支持；前端需要组件化、路由与权限控制、图表与终端展示，Vue 与生态库（Vite、ECharts、xterm.js 等）可很好满足。第二，前后端分离使后端与前端可独立开发、测试与部署，内嵌 Tomcat 与可执行 jar 使后端部署简单（默认端口 8082），前端构建为静态资源后可托管于 Nginx。第三，Spring Boot 的依赖注入与分层架构、Vue 的组件化与响应式使代码结构清晰，便于维护与扩展。系统在认证与权限、设备管理、数据采集与指标、告警、Web SSH、配置备份、审计与 AI 助手等模块中均基于 Spring Boot + Vue 实现，满足运维监控平台对稳定性、可扩展性与开发效率的要求。

### 2.2 数据存储与缓存

#### 2.2.1 MySQL + Redis + InfluxDB 简介

**MySQL** 是一种关系型数据库管理系统，由瑞典 MySQL AB 公司开发，现属于 Oracle 旗下产品。MySQL 采用 SQL 作为查询语言，支持事务、索引、外键与多种存储引擎（如 InnoDB），具有开源、跨平台、性能稳定、生态成熟等特点，广泛应用于业务数据的持久化与复杂查询场景。

**Redis** 是一种基于内存的键值存储系统，由 Salvatore Sanfilippo 开发并开源。Redis 支持字符串、哈希、列表、集合、有序集合等数据结构，提供持久化、主从复制、集群等能力，常用于缓存、会话存储与实时数据读写，具有高性能、低延迟、接口简单等特点。

**InfluxDB** 是一种专为时序数据设计的数据库，由 InfluxData 公司开源。时序数据按时间顺序产生（如监控指标、传感器数据），InfluxDB 针对时间戳索引、高写入吞吐与时间范围查询做了优化，支持标签（tag）与字段（field）、保留策略与连续查询，查询语言为 Flux（或兼容的 InfluxQL），适用于监控、物联网等场景。

#### 2.2.2 数据存储与缓存特性

##### 2.2.2.1 MySQL 特性

支持 ACID 事务与复杂 SQL 查询，适合存储用户、设备、告警规则、审计日志等业务实体及其关联关系。与 JPA/Hibernate 配合可实现实体映射与表结构自动更新（如 ddl-auto=update），减少手写 DDL，便于迭代开发。

##### 2.2.2.2 Redis 特性

基于内存的读写延迟低，支持 Pipeline 批量操作与过期键，适合高并发访问与临时数据缓存。本系统将网络设备的 SNMP 或 CLI 采集结果按设备 ID 缓存于 Redis，供实时指标接口快速读取，避免频繁访问设备或重复计算。

##### 2.2.2.3 InfluxDB 特性

针对时间序列的存储与聚合优化，支持按时间范围、标签过滤与降采样聚合。本系统将 Telegraf 上报的 CPU、内存、磁盘等指标及自写的设备状态时序存入 InfluxDB，供实时指标与趋势图查询使用。

#### 2.2.3 在本系统中的应用

**MySQL**：作为业务主库，使用库名 nebula_ops，存储用户与角色权限（sys_user、role、user_role、role_permission、sys_permission）、设备信息与状态（device）、告警规则与历史（alert_rule、alert_history、alert_template）、配置备份（config_backup）、审计日志（audit_log）、Web SSH 会话（webssh_session）、AI 会话与消息（ai_chat_session、ai_chat_message）、系统配置（system_config）等全部业务实体，满足事务性与复杂查询需求。

**InfluxDB**：存储 Telegraf 写入的 CPU、内存、磁盘、网络等 measurement，以及本系统写入的设备状态时序（device_status）。系统按 host 与设备管理 IP 匹配，将 InfluxDB 数据与 device 表关联后供实时指标与趋势图使用；查询采用 Flux，按时间范围与标签过滤，适合监控场景。

**Redis**：主要存储网络设备按设备 ID 的实时指标（如 key：netpulse:snmp:device:{设备ID}），包括 CPU、内存、最后采集时间等；部分厂商 SSH/Telnet 采集结果也写入 Redis。实时指标接口从 Redis 批量读取并与 MySQL 中的设备信息合并返回，保证前端展示的实时性与性能。

### 2.3 前端技术选型（Vite、ECharts 等）

在 2.1 节已介绍 Vue 作为本系统前端核心框架的基础上，本节介绍与 Vue 配套使用的构建工具、图表库与其它前端技术。

#### 2.3.1 Vite + ECharts 等简介

**Vue** 作为本系统前端主框架，其简介与特性见 2.1.1 与 2.1.2.3。Vue 采用响应式数据绑定与组件化开发，核心库只关注视图层，易于与现有项目或其它库集成；配合 Vue Router 可构建单页应用（SPA），配合 Vuex 或 Pinia 可管理全局状态。Vue 3 在性能、TypeScript 支持与 Composition API 等方面有显著改进，适合中大型前端项目。

**Vite** 是新一代前端构建工具，由 Vue 作者主导开发。Vite 在开发阶段基于原生 ES 模块提供极快的冷启动与热更新，在生产构建时使用 Rollup 打包，与 Vue 3、React 等框架配合良好，已逐渐成为 Vue 项目推荐的构建与开发服务器。

**ECharts** 是百度开源的可视化图表库，支持折线图、柱状图、饼图、散点图、地图等多种图表类型，提供丰富的配置项与交互能力，适用于数据大屏、报表与监控看板等场景。

#### 2.3.2 前端技术特性

##### 2.3.2.1 Vue 特性

**响应式与组件化**。数据与视图双向绑定，数据变更自动驱动视图更新；组件可复用、可组合，便于维护与协作开发。Vue 3 的 Composition API 与 script setup 使逻辑组织更清晰，类型推导更友好。

**单页应用与路由**。Vue Router 负责 URL 与组件的映射，支持嵌套路由、路由守卫与懒加载。本系统通过路由守卫在访问受保护页面时检查登录状态与菜单权限（menuCode 与 allowedMenus），实现前后端权限一致。

##### 2.3.2.2 Vite 特性

开发阶段无需打包，按需编译，冷启动与热更新速度快；生产构建输出优化后的静态资源，可部署至 Nginx 等。通过配置 proxy 可将 /api 请求转发至后端，便于前后端联调。

##### 2.3.2.3 ECharts 特性

配置驱动、图表类型丰富、支持主题与自适应。本系统在仪表盘、实时指标页、监控趋势等场景中使用折线图、饼图等展示设备状态时间线、CPU/内存分布、数据来源等，提升监控数据的可读性。

#### 2.3.3 在本系统中的应用

**Vue 3 + Vue Router**：采用 Composition API 与 script setup 进行组件化开发；路由守卫在未登录时跳转登录页并携带 from 参数，已登录时根据 menuCode 与 allowedMenus 判断是否有权访问，无权限则重定向至仪表盘；菜单项根据 canShowMenu 控制显示。

**Vite**：开发时通过 proxy 将 /api 转发至后端 8082 端口；生产构建执行 npm run build 输出 dist，部署至 Nginx 等静态服务器。

**Axios**：封装 baseURL=/api、超时与默认 Content-Type；请求拦截器写入 X-User-Name 供后端审计与权限使用；响应拦截器对 401 清除本地用户并跳转登录页。

**xterm.js**：在浏览器中渲染 Web SSH 终端界面，配合 FitAddon 在容器尺寸变化时自动调整终端行列数。

**ECharts**：仪表盘中的设备状态时间线、设备类型与告警分布饼图，实时指标页的数据来源与 CPU/内存饼图，监控趋势的 CPU/内存折线图等均基于 ECharts 实现。此外，列表与表格请求增加加载态，筛选与刷新采用防抖或节流，减轻重复请求并提升交互体验。

### 2.4 远程与采集相关技术

#### 2.4.1 JSch、SNMP、WebSocket 等简介

**JSch** 是纯 Java 实现的 SSH2 协议库，可在不依赖本地 ssh 命令的情况下建立 SSH 连接、执行命令与 SFTP 传输，广泛应用于 Java 后端的远程执行与自动化运维场景。

**SNMP**（Simple Network Management Protocol）是用于网络设备管理的标准协议，通过 OID 获取或设置设备信息与指标。Snmp4j 是 Java 平台常用的 SNMP 库，支持 v2c 与 v3，便于对路由器、交换机、防火墙等设备进行指标采集。

**WebSocket** 是一种在单个 TCP 连接上进行全双工通信的协议，适合实时推送与长连接场景。Spring 提供 WebSocket 服务端支持，可与前端建立持久连接，实现终端输入输出的双向转发。

**Apache Commons Net** 提供 Telnet 客户端等网络工具，用于仅支持 Telnet 的老旧网络设备接入。

#### 2.4.2 在本系统中的应用

**JSch**：用于 Web 终端与设备建立 SSH 会话、批量命令执行、告警自动修复时执行远程命令，以及通过 SSH 获取 Linux 设备主机名与 os-release 信息以回填设备名称与厂商。系统根据设备 ssh_port 区分：端口 22 使用 SSH，端口 23 使用 Telnet。

**Apache Commons Net**：为仅支持 Telnet 的网络设备提供客户端实现，与 Web SSH 终端、批量命令统一接入。

**Snmp4j**：对已配置 SNMP 的网络设备定时采集（可配置间隔），将 sysName、sysDescr、CPU、内存等写入 Redis；支持多厂商 OID（华为、锐捷、Cisco 等），按设备类型与厂商选择对应 OID 获取指标。

**Spring WebSocket**：Web SSH 端点 /api/ws/ssh 接收前端 WebSocket 连接，根据 deviceId 从 device 表获取凭证后与设备建立 SSH 或 Telnet 连接，将终端输入输出在 WebSocket 与设备连接之间双向转发，实现浏览器内终端。

**Spring Mail**：告警规则触发且勾选邮件通知时，由 AlertEmailService 调用 SMTP 发送告警邮件；SMTP 服务器、端口、发件人、认证信息等在系统设置中配置。

**Spring Scheduling**：通过 @Scheduled 或编程方式配置定时任务，包括 MonitorService 中的设备 Ping（默认 60 秒一轮）、告警规则评估、SNMP 采集、DeviceVendorFillService 对在线且厂商为空的 Linux 设备定时填充厂商信息等。

### 2.5 外部依赖与部署

#### 2.5.1 Telegraf、Nginx 与构建工具简介

**Telegraf** 是 InfluxData 推出的指标采集代理，可采集系统、数据库、网络设备等多种数据源并写入 InfluxDB、Prometheus 等。部署在待监控的 Linux 主机上后，可自动采集 CPU、内存、磁盘、网络等指标，便于与本系统对接形成统一监控视图。

**Nginx** 是高性能的 HTTP 与反向代理服务器，常用于托管静态资源、负载均衡与反向代理。在生产环境中可将前端 dist 与 /api、/api/ws/ssh 反向代理至后端，实现前后端同域访问与统一入口。

**Maven** 是 Java 项目常用的构建与依赖管理工具；**npm** 是 Node.js 的包管理器，用于前端依赖安装与脚本执行（如 npm run build、npm run dev）。

#### 2.5.2 在本系统中的应用

**Telegraf**：需部署在待监控的 Linux 服务器上，采集 CPU、内存、磁盘、网络等指标并写入 InfluxDB。设备管理中的设备名称或管理 IP 需与 Telegraf 上报的 host 一致，系统才能将 InfluxDB 数据与设备匹配并展示；若不一致，可在 Telegraf 配置中设置 agent.hostname 为设备 IP。

**Nginx**（可选）：在生产环境中托管前端静态资源（dist），并配置反向代理将 /api 及 WebSocket 路径 /api/ws/ssh 转发至后端，实现前后端同域访问。

**构建与运行**：后端使用 Maven 构建，JDK 21，运行主类为 NetPulseApplication，默认端口 8082；前端需 Node.js 与 npm，开发时执行 npm run dev（默认端口 5173），生产执行 npm run build 输出 dist。数据库与中间件（MySQL、Redis、InfluxDB）可部署在本机或远程；系统支持通过环境变量（如 MYSQL_HOST、REDIS_HOST、INFLUXDB_URL）覆盖配置文件中的地址，便于多环境部署。本地演示可使用 Spring Profile local，启用 H2 内存库并排除 Redis 自动配置，无需外部 MySQL 与 Redis 即可运行。

## 3 系统分析

### 3.1 业务需求

系统需支撑以下核心业务场景。

**设备统一纳管**：运维人员需要对机房或园区内的服务器、路由器、交换机、防火墙等设备进行统一登记与管理。每条设备记录包括管理 IP、设备类型（服务器/路由器/交换机/防火墙/其他）、厂商与型号、所属分组；对于支持 SSH 或 Telnet 的设备，需记录端口（22 为 SSH、23 为 Telnet）、用户名与密码，用于 Web 终端与批量命令；对于网络设备若需通过 SNMP 采集指标，需配置 SNMP 版本（v2c 或 v3）及社区名或 v3 安全信息。系统应支持单条添加、编辑、删除以及批量导入（如 CSV：名称,IP,类型），并支持按分组筛选与多字段排序、分页展示。同时应提供设备健康汇总（各状态数量）与单设备 Ping 检测，以及设备拓扑图展示与拓扑上的快捷跳转（如进入 Web SSH）。

**状态与指标监控**：系统需定期或按需对设备进行可达性检测（对服务器通常采用 Ping，对网络设备可采用 Ping 或 TCP 端口探测），根据检测结果更新设备状态（正常/告警/离线），并记录最后轮询时间。状态变化应可触发告警规则。此外，系统需展示各设备的实时性能指标：Linux 设备以 CPU、内存、磁盘使用率等为主，数据来源为 Telegraf 写入 InfluxDB 或由本系统通过 SSH 回退采集；网络设备以 CPU、内存等为主，数据来源为 SNMP 或厂商 CLI 采集后写入 Redis。指标需与设备表匹配后统一展示，并区分数据来源，支持刷新与历史趋势查询。

**告警管理**：运维人员需要根据业务重要性配置告警规则，例如某类设备 CPU 使用率超过 80% 则告警、某设备离线则告警等。规则可关联设备或设备类型，并设置严重程度（如严重、警告、一般）、是否启用、是否发送邮件、是否自动修复等。系统在定时评估或设备状态变更时检查规则，满足条件则生成告警历史（状态为 firing），并可执行自动修复动作（如执行 SSH 命令或本地脚本）；告警恢复后更新历史记录。运维人员可查看告警历史、按状态与严重程度筛选、人工或批量标记已处理，并导出 CSV。

**远程运维**：运维人员希望在浏览器内直接登录设备执行命令，无需安装 Xshell、SecureCRT 等客户端。系统需提供 Web SSH（或 Web Telnet）终端，后端根据设备凭证与设备建立连接，并将终端输入输出通过 WebSocket 与前端同步。同时，对于需在多台设备上执行相同命令的场景，系统应支持批量命令：用户选择多台设备并输入命令，后端逐台执行并汇总成功与失败结果返回。网络运维场景下，还可结合 AI 生成配置命令并在终端中执行。

**配置与审计**：为防止误操作或灾难，系统应支持将设备、告警规则、系统配置等备份为 JSON，并支持从备份恢复。系统参数（如系统名称、时区）与第三方 API（如大模型）配置应以键值形式存储并可从前端修改。此外，关键操作（如设备增删改、规则变更、用户管理）应写入审计日志，记录操作人、操作类型、对象、详情、IP 等，支持按条件查询与导出。

**智能辅助**：为提升运维效率，系统可集成大模型 API，提供 AI 运维助手。用户可创建多会话，在会话中进行多轮对话；系统将设备与监控上下文注入提示，调用大模型 API 获取回复并持久化，便于后续查阅与延续对话。

### 3.2 功能需求

**认证与权限**：用户使用用户名和密码登录后，系统会识别其身份并决定能看到哪些菜单、能做什么操作。不同角色（如管理员、普通运维人员）看到的侧栏菜单不一样；进入某个页面时，若当前用户没有权限，会被自动跳转回首页，避免越权操作。管理员可以为每个角色勾选“允许访问的菜单”，用户登录后只能看到自己所属角色被允许的菜单。管理员可以新建、编辑用户和角色，普通用户一般只能修改自己的密码。

**设备管理**：在设备列表里可以新增、修改、删除设备。系统提供**批量导入**功能：用户将多台设备信息整理成表格（CSV 或 Excel 导出为 CSV），按约定列填写后上传，系统解析并一次性创建多条设备记录，适合机房新设备批量录入。批量导入的表格格式要求如下。

| 列名 | 含义 | 是否必填 | 说明与示例 |
|------|------|----------|------------|
| IP | 管理 IP 地址 | 必填 | 如：192.168.1.1、10.0.0.100 |
| 类型 | 设备类型 | 必填 | 服务器 / 路由器 / 交换机 / 防火墙 / 其他（任选其一） |
| 用户 | 登录用户名 | 必填 | SSH/Telnet 登录所用账号 |
| 密码 | 登录密码 | 必填 | SSH/Telnet 登录所用密码 |
| 名称 | 设备显示名称 | 选填 | 如：核心交换机-1、Web 服务器-01 |
| 分组 | 所属分组 | 选填 | 便于按业务或机房分组筛选，如：机房A、核心网络 |
| 备注 | 备注信息 | 选填 | 用于补充说明 |

导入时第一行为表头，从第二行起每行一条设备；IP、类型、用户、密码为必填项，类型需与系统预设一致，IP 需合法，名称、分组、备注等为选填；名称与 IP 不建议与已有设备重复以免冲突。导入完成后可在设备列表中查看、编辑或补充 SSH/SNMP 等详细配置。

此外，列表支持按分组、IP、类型、状态等筛选和排序，并分页显示。系统会统计当前有多少设备正常、多少告警、多少离线，在页面上展示设备健康汇总。对单台设备可以发起“ Ping 检测”，查看当前是否可达、延迟大概多少。设备拓扑图用节点方式展示所有设备，可以按分组和状态筛选、拖拽排布，点击某个设备可直接打开该设备的远程终端。对已配置好 SSH 的 Linux 服务器，系统可以自动读取其主机名和厂商信息并回填到设备信息里，减少手工录入。

**数据采集与实时指标**：系统会按一定时间间隔自动检测设备是否在线（例如 Ping 或探测端口），并更新设备状态；一旦设备从在线变离线或从离线恢复，会触发告警逻辑。实时指标页从各数据源（如 Linux 上的采集程序、网络设备的 SNMP 等）拉取 CPU、内存等数据，在页面上以表格和饼图形式展示，区分“来自 Linux 的设备”和“来自网络设备的设备”，并标明数据来源。用户可以手动刷新数据，也可以按设备和时间范围查看历史趋势图（如某台设备过去一段时间的 CPU、内存曲线）。

**告警**：运维人员可以配置告警规则，例如“某类设备 CPU 超过 80% 就告警”“某台设备离线就告警”等。系统按规则定期检查，条件满足就产生一条告警记录，并可选择发邮件通知或自动执行修复动作；问题恢复后，告警记录会标记为已恢复。告警历史支持按状态、严重程度筛选，可以单条或批量标记为“已处理”，并支持导出成表格文件。

**Web SSH 与批量命令**：在浏览器里打开“远程终端”并选择设备后，系统会帮用户连到该设备的命令行界面，输入输出都在网页里完成，无需再单独安装 SSH 客户端。需要在一批设备上执行相同命令时，可以选择多台设备、输入一条命令，由系统依次在每台设备上执行并把结果汇总展示。网络运维场景下，还可以按厂商和需求让 AI 生成配置命令，并在终端里直接连到设备执行。

**配置备份与系统配置**：可以将当前系统的设备列表、告警规则、系统参数等打包成一份备份（支持“全部备份”或只备份某一类），需要时可以从备份恢复，方便迁移或出问题时回滚。系统名称、时区、邮件服务器、AI 接口等参数在“系统设置”里以配置项形式维护，修改后即生效。

**审计与导出**：对设备增删改、告警规则变更、用户管理等重要操作，系统会记录“谁、在什么时间、对什么做了何种操作”，便于事后追溯。告警历史、操作记录等支持按条件查询，并可以导出为表格文件。

**AI 助手**：系统提供智能问答助手，用户可以新建多个对话，在对话里多轮提问。助手会结合当前设备与监控情况回答问题，回复会保存下来，方便后续查看和继续对话。

### 3.3 非功能需求

**可用性**：核心接口具备异常处理与超时控制，避免未捕获异常导致服务不可用；全局异常处理器对请求体解析错误返回 400、其他未捕获异常返回 500 并附带简短消息。前端请求拦截器对 401 统一清除本地用户并跳转登录；对列表等接口的返回数据做空值兜底（如 r?.data ?? []），避免页面报错。列表与表格在请求过程中展示加载态（遮罩与旋转图标）；筛选、刷新等操作采用防抖或节流，减轻重复请求与按钮连点带来的压力。

**可扩展性**：设备类型、监控指标、告警渠道等通过数据库表结构与配置项扩展，避免在代码中硬编码；新增设备类型或告警渠道时主要扩展配置与表数据即可。

**数据持久化**：业务数据持久化至 MySQL；时序数据写入 InfluxDB；网络设备实时数据经 Redis 缓存，保证读取性能与数据新鲜度。

## 4 系统设计

### 4.1 系统架构

系统采用 B/S 架构与前后端分离。用户通过浏览器访问前端应用（Vue 3 + Vite 构建），前端通过 HTTP（RESTful）请求调用后端 API，通过 WebSocket 建立 Web SSH 等长连接。后端应用（Spring Boot 3）负责业务逻辑、定时任务与数据访问，对接 MySQL（nebula_ops）、InfluxDB、Redis。数据分工明确：MySQL 存储全部业务实体；InfluxDB 存储 Telegraf 时序及设备状态时序；Redis 存储网络设备实时指标缓存。整体架构便于在不同环境中部署与扩展；可通过环境变量或配置文件切换数据源与中间件地址，适合开发、测试与生产环境分离。可结合项目 docs 目录下的系统功能架构图绘制分层图：用户层、表现层（前端）、业务层（后端）、数据层（MySQL/Redis/InfluxDB）及外部系统（设备、邮件、大模型 API、Telegraf）。

### 4.2 功能模块划分

**认证与权限模块**：实现登录、会话维持与角色菜单控制。主要涉及 AuthController（登录接口）、UserService、RoleController、RoleMenuService、InitialAdminRunner（初始化管理员与角色菜单）等。前端 Login.vue、Layout.vue 及路由守卫配合实现登录页与受保护页的访问控制。

**设备管理模块**：实现设备 CRUD、批量导入、分组列表、健康汇总、Ping、设备拓扑所需数据、主机名与厂商获取。主要涉及 DeviceController、DeviceService、MonitorService（Ping 与状态更新）、DeviceVendorFillService（定时填充厂商）等；前端 DeviceList.vue、DeviceTopology.vue 对应设备列表与拓扑页。

**数据采集与指标模块**：实现状态更新（MonitorService）、InfluxDB 与 Redis 查询（InfluxDbMetricService、DeviceStatsService、SnmpStatsService）、实时指标聚合与来源标识（MetricsController）。前端 RealtimeMetrics.vue 展示实时指标表格与图表；Dashboard.vue 使用设备列表与健康汇总、设备状态时间线等接口。

**告警模块**：实现告警规则 CRUD、告警评估、告警历史写入与查询、自动修复与邮件通知。主要涉及 AlertController、AlertService、AlertEmailService 等；前端 AlertNotify.vue 对应告警规则与历史页。

**Web SSH 与批量命令模块**：实现 WebSocket 代理（SshWebSocketHandler）、SSH/Telnet 连接建立与双向转发、批量命令执行与结果汇总（BatchCommandService、BatchCommandController）。前端 WebSsh.vue、BatchCommand.vue、NetworkAiCommand.vue 对应 Web 终端、批量命令与网络 AI 控制台。

**配置备份与系统配置模块**：实现备份创建与恢复（ConfigBackupController、ConfigBackupService）、系统配置与 API 配置的键值管理（SystemConfigController、ApiSettingsController）。前端 ConfigBackup.vue、SystemSettings.vue 对应配置备份与系统设置页。

**AI 助手模块**：实现会话与消息持久化、大模型 API 调用（AiChatController、AiChatService）。前端 AiAssistant.vue 对应 AI 助手页。

**审计与导出模块**：实现审计日志写入（AuditService）与查询、CSV 导出（ExportController）。前端 OperationAudit.vue 对应操作审计页；导出多与告警、审计等页面的按钮联动。

### 4.3 系统逻辑流程

**用户登录与权限**：用户在前端输入用户名与密码并提交 → 前端 POST /api/auth/login → 后端校验 sys_user 表，查询 user_role、role 及角色对应菜单，组装 allowedMenus → 返回用户信息（含 username、allowedMenus 等）→ 前端将用户信息存入 localStorage，跳转至 from 指定页或仪表盘；后续请求在拦截器中读取用户并将 username 写入 X-User-Name。前端侧栏根据 allowedMenus 与菜单项 code 计算 canShowMenu，仅显示有权限的菜单；路由守卫在进入非公开页时检查 localStorage 与目标 meta.menuCode，未登录则跳登录并带 from，无权限则重定向仪表盘。

**设备管理与健康**：设备增删改通过 DeviceController 写入 device 表；批量导入解析 CSV 后调用创建接口。定时任务 MonitorService 按配置间隔遍历未删除设备，对服务器执行 Ping、对网络设备可执行 TCP 探测，根据结果更新 status 与 last_poll_time（通过 DeviceRepository.updateStatusAndLastPollTime，不覆盖 name、vendor 等），并写入 InfluxDB device_status；若状态由在线变为离线或反之，触发 AlertService 写入设备上下线告警。前端设备列表与仪表盘请求设备列表与健康汇总接口，仪表盘离线设备列表按 last_poll_time 降序排序以最近离线的在前。

**实时指标**：前端请求 GET /api/metrics/realtime → 后端从 InfluxDB 查询 Telegraf 数据（按 host 与设备 IP 或名称匹配）、从 Redis 读取网络设备缓存，与 device 表合并后返回 devices、stats、statsSource、summary 等；无 InfluxDB 数据时对 Linux 设备可走 SSH 回退采集。前端展示两个表格（Linux 设备、网络设备）及饼图，支持刷新（节流）与趋势图（选择设备与时间范围后请求趋势接口）。

**告警**：规则配置写入 alert_rule；定时或设备状态变更时 AlertService 根据规则与当前指标/状态判断，满足则写 alert_history（status=firing），可选执行自动修复与发送邮件；恢复后更新 alert_history 的 status 与 end_time。前端告警历史支持分页、筛选、批量标记与导出。

**Web SSH**：用户在前端选择设备并点击连接 → 前端建立 WebSocket 连接 /api/ws/ssh?deviceId=xxx → 后端根据 deviceId 取 device 的 ssh_user、ssh_password、ssh_port，若端口为 23 则建立 Telnet 否则 SSH，将终端输入输出与 WebSocket 消息双向转发。

**配置备份**：用户选择备份类型并创建备份 → 后端汇总对应表数据序列化为 JSON 写入 config_backup；恢复时读取 JSON 解析后写回相应表（需注意冲突与权限）。

**AI 助手**：用户发送消息 → 前端 POST 消息至后端 → 后端加载该会话历史消息，组装 prompt 并调用大模型 API，将回复写入 ai_chat_message 并返回前端展示。

### 4.4 数据库设计

#### 4.4.1 设计原则

数据库设计遵循与业务实体一致、适度规范化、便于扩展的原则。核心实体均有对应表，关系通过外键或关联表表达；关键表保留 create_time、update_time 便于审计与排查；设备、告警等支持逻辑删除（deleted 字段）以保留历史。扩展新设备类型、新告警渠道时以配置与数据为主，避免频繁改表结构。

#### 4.4.2 E-R 与实体关系

核心实体包括：用户（sys_user）、角色（role）、用户角色关联（user_role）、角色权限（role_permission、sys_permission）、设备（device）、告警规则（alert_rule）、告警历史（alert_history）、告警模板（alert_template）、配置备份（config_backup）、系统配置（system_config）、审计日志（audit_log）、Web SSH 会话（webssh_session）、AI 会话（ai_chat_session）、AI 消息（ai_chat_message）等。关系：用户与角色多对多（user_role）；角色与权限多对多（role_permission）；设备与告警规则/告警历史/Web SSH 会话为一对多；告警历史与通知记录一对多；AI 会话与消息一对多；配置备份、审计日志与用户多对一。绘图时实体用矩形、联系用菱形或连线并标注 1、n。详细 E-R 与属性见 docs/全局ER图-含属性.md。

#### 4.4.3 主要表结构说明

**device**：id（主键）、name、ip、type（router/switch/server/firewall/other）、vendor、model、snmp_version、snmp_community、snmp_security、snmp_port、ssh_port、ssh_user、ssh_password、remark、group_name、status（normal/warning/critical/offline）、last_poll_time、create_time、update_time、deleted 等。其中 type 与 status 为枚举，与前端展示一致。

**sys_user**：id、username、password、real_name、email、enabled、create_time、update_time。用户与角色通过 user_role 关联。

**role**、**user_role**、**role_permission**、**sys_permission**：角色及用户-角色、角色-权限关联与权限（菜单）定义。角色权限模板决定各角色可分配菜单，用户可见菜单为所选角色模板的并集。

**alert_rule**：id、name、device_id（可选）、device_types、metric_key、condition、severity、enabled、notify_email、auto_fix_enabled、auto_fix_type、auto_fix_command、create_time、update_time 等。condition 存储如 "> 80" 或 "offline" 等。

**alert_history**：id、rule_id、device_id、metric_key、trigger_value、start_time、end_time、status（firing/resolved）、severity、message 等。支持分页与按状态、严重程度筛选。

**config_backup**：id、name、backup_type、summary、content（LONGTEXT 存 JSON）、user_id、create_time。

**audit_log**：id、username、action、target_type、target_id、detail、ip、create_time。

**webssh_session**：id、user_id、device_id、session_id、start_time、end_time、input_log 等（可选记录）。

**ai_chat_session**、**ai_chat_message**：会话 id、用户、标题等；消息含 role（user/assistant）、content 等。

**system_config**：id、config_key（唯一）、config_value、remark。存储系统名称、时区、邮件 SMTP、API 密钥等。

详细字段与索引参见 docs/后端与数据库表结构对照.md 及项目建表脚本。

## 5 系统实现与效果

### 5.1 认证与权限实现

认证采用表单登录：前端将用户名与密码提交至 POST /api/auth/login，后端在 sys_user 中校验用户名与密码，并查询 user_role、role 及角色对应的菜单权限，组装 allowedMenus 列表返回；前端将返回的用户信息（含 username、allowedMenus、realName 等）存入 localStorage。后续所有请求在 Axios 请求拦截器中从 localStorage 读取用户，将 username 写入请求头 X-User-Name，供后端审计与权限使用。响应拦截器中对 401 状态码清除 localStorage 中的用户并跳转至登录页，URL 带 from 参数以便登录后回到原页。

前端菜单与路由控制：Layout.vue 中根据当前用户 allowedMenus 与菜单项的 code 计算 canShowMenu(item)，仅当 item.code 在 allowedMenus 中时显示该菜单。路由守卫 beforeEach 中，若目标路由需要 menuCode 且用户 allowedMenus 不包含该 menuCode，则重定向至 /dashboard，避免无权限用户通过 URL 直接访问。用户管理页支持为每个角色配置可分配菜单（角色权限模板），保存至 role_permission 或等效结构，用户可见菜单为其所有角色对应模板的并集。

### 5.2 设备管理实现

设备列表页面（DeviceList.vue）提供设备表的增删改查、批量导入、按分组筛选（下拉调用防抖后的 load）、按 IP/类型/状态排序、分页；健康汇总通过 GET /api/devices/health 获取各状态数量并可在页眉或仪表盘展示。单设备 Ping 通过 GET /api/devices/{id}/ping 调用，返回 rttMs 与 status（up/down），前端更新该行状态或提示。设备拓扑图（DeviceTopology.vue）从 GET /api/devices 获取设备列表，按分组与状态筛选后以节点形式渲染，支持拖拽并调用 localStorage 持久化位置；点击设备可跳转至 /ssh/{id} 进入 Web SSH。主机名与厂商获取：POST /api/devices/{id}/fetch-hostname 在后端执行 SSH 命令，一次性获取 hostname 与 os-release 关键行（NAME、ID 等），解析厂商（如 OsReleaseParser.parseVendor）并回填设备 name 与 vendor；仅对类型为服务器、已配置 SSH 且在线（或跳过离线检查）的设备有效。DeviceVendorFillService 定时（如每 10 分钟）对在线、厂商为空、已配 SSH 的 Linux 设备执行类似逻辑自动填充厂商，减少人工维护。

### 5.3 数据采集与实时指标实现

设备状态由 MonitorService 定时执行（默认 60 秒）：对每台未删除设备根据类型执行 Ping 或 TCP 端口探测，根据结果设置 status（如 RTT<0 为 offline，RTT>800 为 warning 等）并调用 DeviceRepository.updateStatusAndLastPollTime 更新 status 与 last_poll_time，不覆盖 name、vendor 等字段；同时将状态写入 InfluxDB device_status。状态由在线变离线或离线变在线时触发 AlertService 写入设备上下线告警。

实时指标：GET /api/metrics/realtime 由 MetricsController 协调，先获取设备列表（可按分组），再从 InfluxDB 按 host 查询 Telegraf 的 cpu、mem 等（DeviceStatsService、InfluxDbMetricService），从 Redis 按设备 ID 或 IP 读取网络设备数据（SnmpStatsService 或 SSH 缓存）；无 InfluxDB 数据时对 Linux 设备可通过 DeviceStatsService 经 SSH 执行 free、/proc/stat、df 等命令回退采集。结果与 device 表匹配后合并为 stats（deviceId -> 指标对象）、statsSource（deviceId -> "influxdb"/"snmp" 等）返回。前端 RealtimeMetrics.vue 每 10 秒请求一次，展示 Linux 设备表与网络设备表、数据来源与 CPU/内存饼图；手动刷新按钮使用节流（如 1500ms）避免重复请求；趋势图选择主机与时间范围后请求 getMetricsTrend，后端从 InfluxDB 查询 cpu、mem 时序返回，前端用 ECharts 绘制折线图。

### 5.4 告警实现

告警规则在 AlertNotify.vue 中配置，提交后写入 alert_rule 表，支持指标类型（如 cpu_usage）、条件（如 "> 80"）、严重程度、是否启用、是否邮件通知、是否自动修复及修复类型与命令。设备上下线类规则条件为 offline 等，由设备状态变化触发。AlertService 在定时任务或设备状态变更时评估规则：获取当前指标与状态，仅使用在有效时间窗口内的数据，满足条件则写入 alert_history（status=firing），并可选执行自动修复（SSH 命令或本地脚本）与发送邮件（AlertEmailService）。告警恢复（如设备上线或指标回落）后更新 alert_history 的 status=resolved、end_time。前端告警历史支持分页、按状态与严重程度筛选、单条或批量标记已处理、CSV 导出。

### 5.5 Web SSH 与批量命令实现

Web SSH：前端 WebSsh.vue 根据所选设备 id 建立 WebSocket 连接至 /api/ws/ssh?deviceId=xxx，后端 SshWebSocketHandler 根据 deviceId 从 device 表获取 ssh_user、ssh_password、ssh_port；若 ssh_port 为 23 则使用 Apache Commons Net 建立 Telnet 连接，否则使用 JSch 建立 SSH 连接。连接建立后，将 WebSocket 收到的文本消息转发至设备终端输入，将设备终端输出转发至 WebSocket 发送给前端；xterm.js 将接收到的内容渲染到终端界面，用户输入通过 WebSocket 发送。批量命令：用户在 BatchCommand.vue 中选择多台 Linux 设备并输入命令，提交后后端 BatchCommandService 逐台建立 SSH（或 Telnet）执行命令，收集标准输出与错误输出及成功与否，汇总后返回；前端展示执行结果列表。网络运维控制台 NetworkAiCommand.vue 支持按厂商与需求调用 AI 生成 CLI 配置，并将生成的命令在嵌入的终端中直连设备执行。

### 5.6 配置备份与系统配置实现

配置备份：用户选择备份类型（全量/仅设备/仅告警/仅系统配置）后点击创建备份，后端 ConfigBackupService 从相应表读取数据序列化为 JSON，写入 config_backup 的 content 字段；恢复时读取 content 解析后按类型写回 device、alert_rule、system_config 等表（需处理主键与冲突）。系统配置与 API 配置：键值对存入 system_config 或专用配置表，系统设置页（SystemSettings.vue）可修改系统名称、时区、邮件 SMTP、大模型 API 密钥等；后端读取后供邮件发送、AI 调用等使用。

### 5.7 AI 助手与审计导出实现

AI 助手：前端 AiAssistant.vue 展示会话列表与当前会话消息列表；用户发送消息时 POST 至后端，AiChatService 加载该会话的历史消息，将设备与监控摘要等上下文与用户消息组装为 prompt，调用配置的大模型 API（如千问、DeepSeek），将助手回复写入 ai_chat_message 并返回前端展示。审计：关键操作（如设备创建/更新/删除、告警规则变更、用户管理）在对应 Service 中调用 AuditService.log，写入 audit_log（username、action、target_type、target_id、detail、ip、create_time）。导出：ExportController 提供告警历史、审计日志等 CSV 导出接口，前端通过链接或按钮触发下载。

### 6.1 测试环境

测试环境为本地开发机或虚拟机。软件环境：MySQL 8.x（库 nebula_ops，已执行建表与初始化脚本），Redis 6.x，InfluxDB 2.x（可选，用于时序与趋势）；后端 JDK 21、Spring Boot 3、Maven，默认端口 8082；前端 Node.js 18+、npm、Vite，开发端口 5173，通过 Vite proxy 将 /api 转发至后端。数据库表结构需包含 device 的 ssh_user、ssh_password、group_name、status、last_poll_time 等列；若使用 JPA ddl-auto=update 可自动补列。为快速演示，可使用 Spring Profile local 启用 H2 内存库并排除 Redis，无需外部 MySQL 与 Redis 即可启动后端；前端仍通过代理访问。

### 6.2 功能测试要点

功能测试覆盖以下方面。认证与权限：登录成功与失败、登出、不同角色登录后菜单显示是否符合 allowedMenus、无权限访问受保护页是否重定向、未登录访问是否跳转登录并带 from。设备管理：设备增删改查、批量导入（合法与非法 CSV）、分组筛选、排序与分页、健康汇总数值是否正确、单设备 Ping 是否更新状态或提示、设备拓扑图数据与筛选、拖拽后刷新是否保留位置、点击设备是否跳转 Web SSH、获取主机名与厂商是否回填。实时指标：Linux 设备与网络设备表是否有数据、数据来源是否与预期一致、刷新是否更新、趋势图选择设备与时间后是否出图。告警：规则创建与列表、告警触发后历史是否生成、分页与筛选、批量标记与导出是否正常。Web SSH：连接是否建立、输入输出是否一致、断线后是否可重连。批量命令：多台设备执行是否全部有结果、成功与失败是否区分。配置备份：创建备份是否成功、恢复后数据是否一致。系统配置与 API 配置：保存后是否生效（如 AI 助手能否调通）。AI 助手：会话创建与消息收发、历史是否持久化。审计：关键操作后审计列表是否有记录、导出 CSV 是否含预期字段。可结合 Postman 调用主要接口验证状态码与响应结构，并在前端逐页操作验证交互与数据一致性。

### 6.3 主要接口与测试结果

重点验证的接口包括：POST /api/auth/login（200 返回用户与 token 信息、401 为账号密码错误）；GET /api/devices（200 返回设备数组，支持 group 参数）；GET /api/devices/health（200 返回各状态数量）；GET /api/devices/groups（200 返回分组名数组）；GET /api/devices/{id}/ping（200 返回 rttMs、status）；POST /api/devices/{id}/fetch-hostname（200 为成功、4xx 为参数或状态不符）；GET /api/metrics/realtime（200 返回 devices、stats、statsSource、summary 等）；GET /api/metrics/device-status-timeline（200 返回时间线数据）；GET /api/alerts/rules、GET /api/alerts/history（200 返回规则列表与历史分页）；POST /api/ws/ssh（WebSocket 握手成功）；GET /api/audit（200 返回审计分页）；GET /api/backup/list（200 返回备份列表）。前端验证：设备列表与实时指标按数据源区分正确、告警历史分页与筛选正常、Web SSH 连接与断线稳定、列表加载态与筛选防抖/刷新节流行为符合预期。可将上述用例整理为测试用例表，列明用例编号、步骤、预期结果、实际结果与通过情况，作为论文测试章节的附表。

### 6.4 部署说明

生产环境部署建议：MySQL、Redis、InfluxDB 单独部署或使用已有集群，保证高可用与备份策略。后端使用 mvn package 打包为 jar，通过 java -jar 或 systemd 等方式运行，通过配置文件或环境变量（如 MYSQL_HOST、REDIS_HOST、INFLUXDB_URL、INFLUXDB_TOKEN）指定数据源与中间件地址；默认端口 8082，可通过 SERVER_PORT 修改。前端在 netpulse-ui 目录执行 npm run build，将生成的 dist 目录部署至 Nginx 或其他静态服务器；Nginx 需配置 /api 与 /api/ws/ 反向代理到后端，以便前端与后端同域访问并支持 WebSocket。文档中应注明默认管理员账号及首次登录后修改密码的建议、端口与防火墙配置、HTTPS 与安全头等注意事项。详细步骤可参考项目 README.md 与 docs 目录下的运维与配置说明。

## 7 总结与展望

### 7.1 总结

本文设计并实现了基于 Spring Boot 3 与 Vue 3 的网络监控与运维平台 NetPulse，完成了设备管理、健康状态监控、实时指标展示（Linux 与网络设备双数据源）、告警规则与历史、Web SSH、批量命令、网络 AI 控制台、配置备份、系统配置、用户权限与角色模板、操作审计与 AI 助手等核心功能。系统采用前后端分离的 B/S 架构，后端基于 Spring Boot 3 与 MySQL、InfluxDB、Redis 实现业务持久化、时序存储与实时缓存，前端基于 Vue 3 与 Vite 实现单页应用与图表展示，通过 RESTful 接口与 WebSocket 与后端通信。数据库设计遵循业务实体与关系建模，主要表结构清晰，并提供了系统逻辑流程与 E-R 关系的描述，便于绘制答辩用流程图与 E-R 图。平台可用于中小规模网络环境的集中监控与日常运维，具有较好的实用性与可扩展性，并为后续扩展更多采集方式、告警渠道与可视化大屏奠定了基础。

### 7.2 不足与展望

当前系统仍存在以下可改进之处。（1）安全方面：用户密码建议改为 BCrypt 等不可逆摘要存储，并加强会话管理（如 Token 过期与刷新），避免明文或弱加密。（2）告警通知：目前支持邮件通知，后续可进一步对接钉钉、企业微信、短信等渠道，并完善通知模板与限频策略。（3）高可用与备份：InfluxDB 与 Redis 的高可用、备份与恢复策略需结合生产环境规划；MySQL 主从或集群可根据规模考虑。（4）性能与限流：实时指标与告警评估在设备数量较大时可能带来并发压力，可引入限流、缓存与异步评估优化。（5）自动化测试：当前缺少成体系的单元测试与接口测试，后续可补充核心模块的测试用例，保障重构与升级时的回归。展望未来，可在上述方面持续增强，并扩展更多采集方式（如 Agent、Prometheus 拉取）、可视化大屏与报表、以及更细粒度的权限与审计，以更好地支撑企业级运维场景。
